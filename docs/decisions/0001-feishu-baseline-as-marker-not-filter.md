# ADR-0001: 飞书基线从过滤逻辑改为标记参考

**状态：** ✅ Accepted（第三类输出语义在 ADR-0007 中微调）
**日期：** 2026-05-10
**决策者：** 用户 + Claude Code (Haiku 4.5)
**相关 Commit：** `b584e94`

> **第三类输出语义在 [ADR-0007](0007-repositioning-to-update-tracking.md) 中微调，基础逻辑（基线作为标记不作过滤）完全保留。第三类含义从"系统建议人工审核"调整为"实体匹配低置信度，系统无法确定与 A 库的对应关系，需运营在中间表中人工判断"。**

---

## 上下文

项目原始设计（`docs/requirements.md` v1）的核心业务流程是：

```
读取飞书基线 → 用基线对照外部数据源 → 过滤掉基线已有的内容 → 推荐新增
```

这个逻辑的隐含假设是：**飞书基线已经包含了"应该有的内容"**，新发现的内容只需要补差集。

**用户在产品讨论中指出问题：**

> "如果只是通过飞书的基线来实现，这就陷入了先有鸡还是先有鸡蛋的问题。热门好评，应该是有更多的来源。"

**问题本质：**
- 飞书基线是"已上架的事实"，不是"全网值得有的"
- 如果基线本身不完整（如缺少近期热门），用基线作过滤等于**永远打不破基线天花板**
- 系统只能发现"基线没有的"，无法发现"基线有但评分低/热度衰退应该补充元数据的"
- 也无法应对"用户飞书基线录入有偏差"的情况

## 决策

**飞书基线的角色从"主过滤逻辑"改为"标记参考"。**

具体：

1. 候选发现完全独立运行（FlixPatrol + TMDb + Trakt + OMDb）
2. 候选生成后，与 `baseline_items` / `canonical_items` 匹配
3. 仅添加字段标记，不过滤：
   - `is_in_baseline: true/false`
   - `baseline_canonical_id: <id or null>`
4. 输出分三类：
   - 🆕 新增推荐（`is_in_baseline=false`）
   - ♻️ 已有可补充（`is_in_baseline=true`，但热度/评分有更新）
   - ⚠️ 待人工确认（低置信度匹配）

## 后果

**正面：**
- 解决"先有鸡还是先有蛋"问题，不依赖基线完整性也能发现新内容
- 已上架内容的元数据可持续更新（评分变化、热度变化）
- 基线录入偏差不再放大为系统盲区
- 推荐列表的内容覆盖度真正反映"全网值得有"
- 与原方案兼容：原"新增"列表仍然存在，只是多了"已有可补充"列表

**负面 / 待解决：**
- 候选规模会变大（之前 ~50/天，现在可能 200+/天）— 需用 hot_score 过滤控制
- 飞书推荐表需要新增字段 `is_in_baseline`、`source_type`
- 重复运行的去重逻辑需要重新设计（不是按"基线没有"，而是按 `content_update_id` 去重）
- 已有内容的"补充推荐"运营是否真的需要——需 V1 上线后验证

## 备选方案

### 备选 A：保留原"基线对比"逻辑，仅补充数据源

**做法：** 加 FlixPatrol，但仍以"基线没有"作为过滤条件。

- 优点：实现简单，与原 requirements.md 一致
- 缺点：未解决核心问题；FlixPatrol 等新数据源的真正价值无法发挥
- **拒绝原因：** 用户的核心诉求是"突破基线限制"，备选 A 实际上没解决这个

### 备选 B：完全去除飞书基线

**做法：** 不读飞书基线，纯做"全网热门"列表。

- 优点：逻辑最简单
- 缺点：失去"已上架去重"能力，运营每天看到大量重复内容；浪费 Phase 0 已完成的 826 条 canonical 数据
- **拒绝原因：** 失去人工审核效率；Phase 0 投入打水漂

### 备选 C：本次决策（基线作为标记）✅

详见"决策"章节。
