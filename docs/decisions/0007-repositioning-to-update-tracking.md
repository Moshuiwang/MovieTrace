# ADR-0007: 系统定位翻转 —— 从"推荐+人工审核"到"更新追踪+中间表"

**状态:** Accepted
**日期:** 2026-05-11
**决策者:** 用户 + Claude Code (Opus 4.7)
**相关 Commit:** 待定(本 ADR 接受后产出)
**取代/影响:** ADR-0001(第三类输出语义微调)、ADR-0004(文字微调)

## 上下文

V1 MVP 已于 2026-05-11 全部交付(7 个任务包,284 测试通过,见
[`reports/phase1_completion_report.md`](../../reports/phase1_completion_report.md))。
交付完成后,用户在产品复盘中重新澄清了对系统职责的理解,与现有文档
存在多处冲突,需要 ADR 级决策。

### 用户核心澄清(2026-05-11)

1. **系统是"全网内容更新追踪系统",不是推荐系统**。系统检测客观事实
   (内容更新事件),不做主观判断(值不值得推荐给最终用户)。
2. **飞书表是"中间表"**(运营业务流的过渡载体),不是审核队列。未来
   可能替换为 Notion/Excel/数据库。
3. **人工审核发生在"中间表→供应商",不在系统侧**。系统职责到"写入
   中间表"为止。
4. **"推荐"一词有歧义,改用"建议"**(避免暗示系统在做主观推荐)。

### 与现有 `docs/requirements.md` 的核心冲突

| 现状 | 用户澄清 |
|------|---------|
| § 1 "**是**'全网值得更新内容的发现+排序+推荐'系统" | 不是推荐系统,是更新追踪系统 |
| § 1 "**不是**纯粹的'新内容更新追踪系统'" | 正是更新追踪系统 |
| § 12.2/12.5 飞书表含 `review_status` 等审核字段 | 审核不在系统侧,字段冗余 |
| § 6.1 输出三类(新增/可补充/待人工确认) | 两类分类 + 第三类作为低置信度收纳 |

### 用户业务现实带来的两层架构约束

用户业务数据库(下称 A 库,当前实现为飞书"线上内容基线表")只有
**两层概念**:节目 + 子节目,扁平结构。例如 `Breaking Bad S01` 与
`Breaking Bad S02` 在 A 库中是两条独立、无父子关联的条目。

而行业网站(TMDb/Trakt/FlixPatrol)采用**三层结构**:剧集 → 季 → 集。

这是 A 库的历史包袱,**短期不能改**。本 ADR 必须在该约束下设计聚合
方案。

## 决策

### 一、系统职责重新定位

**MovieTrace 是"全网内容更新追踪系统"。** 系统职责为:

1. **功能 1:全网热门追踪** —— 每日从外部数据源(FlixPatrol API +
   TMDb + Trakt + OMDb)拉取热门内容,按 `hot_score` 客观阈值过滤,
   与 A 库匹配标记 `is_in_baseline`,写入中间表。
2. **功能 2:基线内容主动追踪** —— 每日按 TMDb 剧集 `status` 字段
   智能轮询 A 库中的 TV 剧集(电影跳过),检测是否有新季发布,写入
   中间表。**功能 2 不受 `hot_score` 阈值约束**。TV 新集
   (episode-level) 追踪于 2026-05-14 明确进入 V2,不属于 V1/P1.12。
3. **功能 3:同步到中间表** —— 上述两个功能的检测结果写入中间表
   (当前为飞书多维表格),供运营从中挑选并提交供应商。

**核心定位的四个"不是":**

- 不是视频资源获取系统(不下载、不入库)
- 不是全网所有新内容追踪系统(仅追踪满足 `hot_score` 阈值的热门内容,
  以及 A 库中已有剧集的更新)
- 不是推荐系统(不负责对最终用户的推荐质量做判断,即不做
  `audience_relevance` 这类主观评估)
- 不是人工审核工具(人工审核发生在"中间表→供应商"环节,在系统职责
  之外)

### 二、A 库 / B 库 / 中间表的架构边界

正式确立三层数据架构概念:

```
┌──────────────────────────────────────────────────────┐
│  A 库:运营业务库("线上内容主数据")                  │
│  • 现阶段:飞书"线上内容基线表"                      │
│  • 未来:可能切换为数据库直连                        │
│  • 维护者:运营(人工)                              │
│  • 系统对 A 库:只读,绝不擅自修改(承袭 ADR-0004)   │
│  • 结构限制:两层(节目+子节目,无父子关联)          │
└──────────────────────────────────────────────────────┘
                  ↑ 单向读取
                  │
┌──────────────────────────────────────────────────────┐
│  B 库:MovieTrace 自维护数据库                       │
│  • 实现:本地 SQLite (`data/movietrace.db`)         │
│  • 内容:                                            │
│    - 从 A 库镜像的"上线状态"子集                    │
│    - 每日检测到的更新事件(功能 1+2 输出)           │
│    - 虚拟剧集聚合层(virtual_series,见决策三)      │
│    - (未来)用户观看数据                            │
│  • 能力:**随时**可吐出"建议更新的数据"              │
│  • 检测与导出**解耦**:检测在后台运行,导出按需触发   │
└──────────────────────────────────────────────────────┘
                  ↓ 系统输出
┌──────────────────────────────────────────────────────┐
│  中间表(=当前飞书"建议更新表")                      │
│  • 系统输出层,运营从中挑选                          │
│  • 未来可替换为 Notion / Excel / 数据库              │
│  • V1.5 不引入抽象 Writer 接口,仍硬编码飞书实现      │
└──────────────────────────────────────────────────────┘
                  ↓ 人工挑选
            供应商(系统职责外)
```

### 三、virtual_series 表:B 库内部解决 A 库两层缺陷

A 库无法改,但功能 2 需要按"剧集"维度轮询(而非按"季"维度,避免
同剧多季重复查 TMDb)。**B 库新增 `virtual_series` 表** 作为聚合层:

- 每条 `virtual_series` 记录代表一个剧集,以 **TMDb `tv_id`** 为聚合
  判据(不靠标题字符串)
- `canonical_items`(season 粒度)新增列 `virtual_series_id`,把同剧
  的 season 聚合
- 复用 schema 中已存在但未启用的 `parent_canonical_item_id` 字段
- A 库不动,聚合完全在 B 库内部完成

**关键字段:**
- `tmdb_tv_id`:聚合主键
- `name`:剧集标题
- `tmdb_status`:`Returning Series` / `Ended` / `Canceled` / ...
- `tmdb_number_of_seasons`:TMDb 当前已知的最高季编号
- `last_polled_at`:上次轮询时间(支撑智能轮询调度)
- `local_max_season`:本系统/A 库内已知最高季(对比检测新季)

### 四、功能 2 的智能轮询策略

按 TMDb `status` 分层轮询,优化 API 配额:

| TMDb status | 含义 | 轮询频率 |
|------|------|---------|
| `Returning Series` | 正在更新中 | 每 1-2 周 1 次(高频) |
| `In Production` | 在制作下一季 | 每 2-3 周 1 次 |
| `Ended` | 已完结 | 每 3-6 月 1 次(低频) |
| `Canceled` | 已取消 | 半年 1 次或跳过 |
| `Pilot` / `Planned` | 早期 | 跳过 |

**电影完全跳过功能 2**(电影无"季"概念,续集发现由功能 1 通过
FlixPatrol/TMDb 热度榜抓)。

### 五、词汇统一:"推荐"→"建议"

| 旧词 | 新词 | 适用范围 |
|------|------|---------|
| 推荐系统 | 更新追踪系统 | 项目定位 |
| (动词)推荐 | (动词)建议 | 系统输出动作 |
| 飞书"推荐更新表" | 飞书"建议更新表" | 飞书表名 |
| ai_reason / 推荐理由 | suggestion_reason / 建议理由(V2) | 字段名 |
| 推荐文案(V2) | 建议文案(V2) | LLM 输出 |

**例外:** 算法术语中的"推荐"(如"实体匹配推荐合并")不动 ——
那是技术术语,无歧义。

### 六、输出分类:两类 + 1 个低置信度归入第三类

输出沿用现有结构,但语义重新对齐:

- **🆕 新增**:`is_in_baseline=false`,A 库中不存在的新内容
- **♻️ 已有可补充**:`is_in_baseline=true`,A 库已有但有热度/新季可补充
- **⚠️ 待人工确认**:实体匹配低置信度(`match_confidence_low=true`),
  系统无法确定与 A 库的对应关系,需运营在中间表中人工判断

**低置信度场景说明:** 当 series ↔ TMDb tv_id 聚合失败(如 A 库标题
拼错且实体匹配低置信度),归入第三类,不参与 `is_in_baseline` 二分类。

**对应 ADR-0001 的微调:** ADR-0001 的"三类输出"语义保留,但第三类
含义从"系统建议人工审核"调整为"系统数据匹配不确定,需运营判断对应
关系"。底层逻辑(基线作为标记不作过滤)完全不变。

### 七、B 库的"随时吐出建议"能力

将现有 `daily-discover` 的"检测+写入"解耦:

- `daily-discover`(检测):**只**做数据采集 + 评分 + 基线匹配 + 写入
  B 库,**不**直接写中间表
- `export-recommendations`(导出):新增命令,从 B 库读取最近的更新
  事件,写入中间表

**定时调度**(每天/每周/每月触发 `daily-discover` 或
`export-recommendations`)**在系统职责之外**,由 cron 或外部调度器
完成。系统只保证两个命令"随时可调"。

### 八、V1.5 范围限制(不做的事)

- **不引入抽象 Writer 接口**(`FeishuWriter` / `NotionWriter` 多态)。
  V1.5 仍硬编码飞书实现,接口抽象等未来真正需要时(V2)再做。
- **不动 V1 已交付的算法层**(hot_score 公式、FlixPatrol API 客户端、
  实体匹配核心逻辑)。
- **不在 B 库引入"用户观看数据"**(V2 范围)。
- **不切换 A 库到数据库直连**(V2 评估)。
- **不做 TV 新集主动追踪**(episode-level `new_episode`,V2 范围)。

## 后果

**正面:**

- 系统职责边界清晰:检测客观事实(更新),不做主观判断(质量)
- 中间表语义与运营实际使用对齐
- A库/B库/中间表三层架构明确,为未来 Notion/数据库切换留语义空间
- virtual_series 表解决 A 库两层缺陷,**且不动 A 库**,符合 ADR-0004
  精神
- TMDb `status` 智能轮询确保 API 配额充足(粗估每日 ~125 次,远低于
  TMDb 限额)
- 6 个已确立 ADR 中 5 个完全不受影响,只 ADR-0001 语义微调、ADR-0004
  文字微调
- 检测与导出解耦,"随时吐出建议"成为 B 库能力

**负面 / 待解决:**

- V1.5 工作量约为原计划"语义重对齐"的 2-3 倍(预计 7 个子任务,
  3-4 个会话完成)
- 一次性回填脚本需要处理 855 条 baseline 中约 500 条 TV 记录
  (一次性,~9 分钟执行)
- A 库未来切换数据库时,B 库镜像逻辑需要重写(V2 风险点)
- "电影续集发现"在 V1.5 不做,部分电影续集可能漏检(由功能 1 通过
  热度榜兜底,但不是 100% 覆盖)
- 历史飞书表已写入的数据在 schema migration 时需要明确处理(详见
  P1.5-B 任务包)

## 备选方案

### 备选 A:维持 V1 现状,定位翻转只改文档

- 优点:零代码改动
- 缺点:文档与代码永久错位;`review_status` 字段成死代码;违反
  CLAUDE.md 第 9 条
- **拒绝原因:** 用户明确要求"如果跑偏没必要继续",此处不能用文档变通

### 备选 B:V1 完全作废,重做 V1.5

- 优点:无历史包袱
- 缺点:浪费 V1 60-70% 可保留代码 + 284 测试 + 架构验证
- **拒绝原因:** 变化本质是语义层,非算法/数据层,翻新比重做合理

### 备选 C:V1.5 只做语义重对齐,功能 2 推到 V2

- 优点:工作量小
- 缺点:用户明确把"基线主动追踪"列为定位核心(功能 2),不做等于翻转
  不完整;文档说有但代码无,产生新文档-代码错位
- **拒绝原因:** 用户选了"路径乙完整实现"

### 备选 D:本次决策(路径乙完整实现)✅

详见"决策"章节。

## 2026-05-12 修正：飞书从系统链路完全移除

**修正日期：** 2026-05-12 +08
**触发：** 生产环境 DB（A 库）CSV 数据接入，`upstream_programs` / `upstream_episodes` 表落地

本 ADR 原文 § 二 的三层架构中，A 库当时实现为飞书"线上内容基线表"，中间表为飞书"建议更新表"。A 库真实数据接入后，两个飞书角色均被替代：

| 原角色 | 原实现 | 新实现 |
|--------|--------|--------|
| A 库（内容目录来源） | 飞书"线上内容基线表" | `upstream_programs` / `upstream_episodes` (SQLite) |
| 中间表（输出渠道） | 飞书"建议更新表" | `reports/recommendations_*.md` + `*.json` 文件导出 |

**修正内容：**
- **P1.5-E 原方案（飞书写入翻新）砍掉**，替换为 A 库全量实体匹配（新 P1.5-E）
- **P1.5-F/G 合并** —— 日报 + CLI + 文件导出，飞书不再参与
- `feishu_import_runs` / `source_records` / `baseline_items` 表保留（历史记录，不删）
- 原文 § 七 的"写入中间表"改为"导出为报告文件"
- 原文 § 八"仍硬编码飞书实现"的约束自然失效

本修正不影响 ADR-0007 的核心定位（系统是更新追踪系统、三层架构概念、virtual_series 聚合、智能轮询策略），仅改变 A 库和中间表的具体实现载体。

## 2026-05-14 修正：TV 新集更新追踪进入 V2

**修正日期：** 2026-05-14 +08
**触发：** P1.12 review 发现当前实现只写 `new_discovery` / `new_season`，没有 `new_episode`；用户确认“新的剧集是否追踪”放入 V2，其他 hotfix 继续。

本 ADR 原文曾把功能 2 描述为检测“新季/新集”。结合 P1.5-D 的实际交付、当前代码行为和用户 2026-05-14 的范围确认，V1/P1.12 的功能 2 边界修正为：

- V1 只主动追踪 **TV 新季**（season-level `new_season`）。
- TV 新集更新（episode-level `new_episode`）进入 V2 backlog。
- P1.12 只修复 review 发现的新季链路、dry-run、日志脱敏、依赖声明等问题，不新增 `new_episode`。

本修正不影响 ADR-0007 的核心定位：MovieTrace 仍是更新追踪系统，不是主观推荐系统；A 库/B 库/中间表架构边界不变。
