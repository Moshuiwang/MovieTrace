# ADR-0002: V1/V2 严格划分原则

**状态：** ✅ Accepted  
**日期：** 2026-05-10  
**决策者：** 用户 + Claude Code (Haiku 4.5)  
**相关 Commit：** `b584e94`

---

## 上下文

在产品方向重新对齐讨论中，Claude 列出了 7+ 个值得做的方向（FlixPatrol、LLM 契合度、多 Agent、RT/Metacritic、Watchmode、IMDb Pro、Reddit/Twitter 等）。

如果不做明确的阶段划分，会出现以下风险：
- **范围蔓延（scope creep）：** 一次性做所有想到的好方向，永远做不完
- **过度投入：** 在没有验证 V1 价值前就引入付费 API 和 LLM 成本
- **优先级混乱：** Agent 接任务时不知道哪些是当前要做的

**用户的明确诉求（2026-05-10）：**

> "在现阶段，我们将相对简单的又价值极高的 flixpatrol 加入工程计划，作为争取实现的内容。将困难的、要花钱的放到产品的下一阶段目标内。"

> "下一阶段是当前产品完整的开发上线一段时间后，积累一些运营经验，完成一个有价值的产品迭代需求的时候。"

## 决策

**V1 与 V2 用清晰的判断标准严格划分，不混合。**

### V1 准入标准（必须全部满足）

1. ✅ **免费或低成本**（本质免费，少量爬虫不算成本）
2. ✅ **低复杂度**（可以用现有技术栈实现，不需要新框架/新库）
3. ✅ **价值已被讨论确认**（不是"以后可能用得上"）
4. ✅ **可在 2-3 周内完成验证 + 实现**

**符合 V1 的方向：**
- 使用 TMDb / Trakt / OMDb（已接入）
- 引入 FlixPatrol（合规公开页面）

### V2 候选条件（满足任一即归 V2）

1. ❌ 需要付费 API（Watchmode、IMDb Pro、JustWatch 等）
2. ❌ 需要 LLM API（Claude/GPT，按 token 付费）
3. ❌ 需要复杂爬虫（多页面、JavaScript 渲染、反爬绕过）
4. ❌ 需要新技术栈（向量数据库、消息队列、scheduler 框架等）
5. ❌ 单项工作量超过 2 周

### V2 启动条件（必须全部满足）

1. V1 已稳定运行至少 1-2 个月
2. 运营反馈了**具体**的需求短板（而非"感觉可以更好"）
3. V2 投入产出比清晰（成本 vs 推荐质量提升的可量化收益）
4. 业务方明确同意承担 V2 额外成本

## 后果

**正面：**
- Agent 在每次接任务时有明确的"做/不做"判断标准
- V1 有清晰上线节点，不会拖到无限期
- 控制成本（V1 阶段无 LLM token 费用、无付费 API 订阅）
- 用真实运营反馈驱动 V2 选型，而不是"工程师猜测"
- 保留所有 V2 候选方向的思考（在 product_roadmap.md），不丢失但也不立即投入

**负面 / 待解决：**
- V1 的产品差异化能力较弱（缺 LLM 契合度判断）
- V1 数据信号维度有限（缺 RT/Metacritic 多源评分验证）
- V1 → V2 切换的时间点判断有主观性，需用户和运营共同判断

## 备选方案

### 备选 A：单阶段做完整版

**做法：** V1 直接包含 LLM、付费 API、多 Agent。

- 优点：上线即差异化，产品价值高
- 缺点：开发周期 2-3 个月起；前期投入风险高（不知道哪些方向真正有价值）
- **拒绝原因：** 用户明确反对"困难的、花钱的放当前阶段"；solo + AI 协作不适合长周期开发

### 备选 B：模糊划分（"以后再说"）

**做法：** 列一个 "future" 清单，没有明确触发条件。

- 优点：灵活
- 缺点：永远开始不了 V2；或者 V1 还没上线就被 V2 想法干扰
- **拒绝原因：** 用户特别强调"避免遗忘，所以希望开始更新文档"——含蓄表达需要明确边界

### 备选 C：本次决策（V1/V2 + 触发条件）✅

详见"决策"章节。
