# MovieTrace 飞书 API 验证沉淀

状态：Phase 0 验证记录  
日期：2026-05-09  
用途：沉淀飞书多维表格 API 联调中发现的问题、根因和后续编码约束，避免正式实现时重复踩坑。

## 1. 本次验证结论

飞书 App 凭证、多维表格读取、测试子表创建、字段创建、记录写入、记录更新和记录回读都已验证可行。

已验证能力：

1. 使用 App ID / App Secret 获取 `tenant_access_token`。
2. 使用多维表格 URL 中的 `app_token` 列出 base 下的数据表。
3. 读取正式基线表 `节目` 的字段和记录样本。
4. 创建独立测试子表 `MovieTrace写入测试`。
5. 在测试子表中创建字段。
6. 写入一条测试记录。
7. 更新该测试记录。
8. 回读更新后的记录并确认字段值。

重要边界：

- 原始 `节目` 表只做读取验证，未写入或修改。
- 写入测试只发生在独立测试子表。
- App Secret 等敏感信息只放在本机临时文件，不进入仓库。

## 2. 已发现并解决的问题

### 2.1 多维表格内授权不等于 API scope 授权

现象：

App 已经被添加到多维表格，并在多维表格内拥有管理权限，但调用列出表接口仍返回 `Access denied`。

飞书返回的缺失权限类似：

```text
bitable:app:readonly
bitable:app
base:table:read
```

根因：

飞书权限分两层：

1. 多维表格内的协作权限。
2. 开放平台 App 的 API scope 权限。

只给 App 多维表格管理权限还不够，App 还必须在飞书开放平台开通多维表格相关 API scope，并发布新版应用配置。

后续编码约束：

- 初始化检查必须同时验证 token 获取和 bitable API 访问。
- 权限失败时，错误提示要明确区分“凭证无效”“API scope 缺失”“App 未加入多维表格”。
- Phase 0 读写验证建议直接要求开通 `bitable:app`，只读验证最低可用 `bitable:app:readonly` 或 `base:table:read`。

### 2.2 App 凭证有效不代表表可访问

现象：

获取 `tenant_access_token` 成功，但读取多维表格失败。

根因：

`tenant_access_token` 只证明 App ID / App Secret 正确，不证明该 App 对目标多维表格有可用 API 权限。

后续编码约束：

- `validate-feishu` 不能只检查 token。
- 至少要执行以下检查：
  1. 获取 `tenant_access_token`。
  2. 使用 `app_token` 列出 tables。
  3. 对目标 `table_id` 读取 fields。
  4. 对目标 `table_id` 读取少量 records。
  5. 对测试表执行写入和更新验证。

### 2.3 不要直接污染正式基线表

现象：

正式 base 中已有 `节目` 表，属于业务基线表。写入验证如果直接在该表执行，会污染真实业务数据。

解决方式：

创建独立测试子表 `MovieTrace写入测试`，只在该测试表执行字段创建、记录写入和记录更新。

后续编码约束：

- 所有写入验证必须使用测试表或显式标记的测试视图。
- 默认禁止向正式 `节目` 表写入测试记录。
- 正式基线表在 Phase 0 第一轮只允许读取。

### 2.4 创建表和创建字段应拆开做

现象：

尝试在创建表请求中同时传入字段定义时，飞书返回：

```text
WrongRequestBody
```

解决方式：

将操作拆成：

1. 创建空测试表。
2. 重新列出 tables，拿到测试表的 `table_id`。
3. 使用 fields 接口逐个创建字段。
4. 再创建 records。

后续编码约束：

- 不要假设创建表接口支持同时创建全部字段。
- 建表、建字段、写记录应拆成可重试的独立步骤。
- 每一步都要检查返回 `code` 和 `msg`。

### 2.5 创建表接口返回结构不能按主观假设取值

现象：

创建表接口返回成功，但脚本从预期路径没有取到 `table_id`，导致后续读取字段时传入 `None`，飞书返回：

```text
WrongTableId
```

解决方式：

创建表成功后，不依赖返回体中的某个固定字段路径，而是重新调用列出 tables 接口，用表名定位刚创建的测试表。

后续编码约束：

- 对飞书 API 返回体不要只凭记忆写字段路径。
- 创建资源后应支持“重新查询确认”。
- 如果按名称定位资源，要处理同名表重复的问题；正式代码中应保存并复用已确认的 `table_id`。

### 2.6 临时密钥文件必须放在仓库外

现象：

验证过程需要反复使用 App ID、App Secret、app token、table ID。

解决方式：

将临时配置保存到：

```text
/tmp/movietrace_phase0_secrets.json
```

并设置权限为：

```text
600
```

后续编码约束：

- 真实密钥不得写入仓库。
- 仓库中只允许提交 `.env.example`、`config.example.yaml` 这类脱敏示例。
- 日志和报告不得输出完整 App Secret、Token 或 API Key。
- 由于 App Secret 曾经出现在对话中，后续进入长期使用前应在飞书后台轮换一次 Secret。

## 3. 后续正式代码建议

### 3.1 `validate-feishu` 应覆盖的检查

正式 CLI 中建议提供：

```text
movietrace validate-feishu
```

最小检查项：

1. 读取本地配置，不打印密钥。
2. 获取 `tenant_access_token`。
3. 列出目标 base 下所有 tables。
4. 确认基线表存在。
5. 读取基线表 fields。
6. 读取基线表前 N 条 records。
7. 确认测试表存在；不存在时按配置决定是否创建。
8. 在测试表写入一条测试记录。
9. 更新该测试记录。
10. 回读并确认更新结果。

### 3.2 错误分类建议

正式实现中建议把飞书错误归类为：

| 类型 | 判断依据 | 给用户的提示 |
| --- | --- | --- |
| 凭证错误 | token 接口失败 | 检查 App ID / App Secret |
| API scope 缺失 | `Access denied` 且提示缺少 scope | 在开放平台开通 `bitable:app` 并发布 |
| base 不可访问 | list tables 失败但 scope 已开 | 确认 App 是否加入多维表格 |
| table ID 错误 | `WrongTableId` | 重新列出 tables 并确认 table_id |
| 请求体错误 | `WrongRequestBody` | 检查接口请求体结构 |
| 字段缺失 | 写入记录提示字段不存在 | 先读取 fields，再创建或映射字段 |

### 3.3 基线表当前风险

当前正式基线表 `节目` 已知字段包括：

- `节目名`
- `上传日期（目录名）`
- `已下载`
- `已上传FTP`
- `已转码`
- `已上架`
- `备注`

风险：

- 缺少内容类型字段。
- 缺少 season / episode 字段。
- 缺少 IMDb / TMDb / Trakt 外部 ID。
- 后续只能先做基于节目名和状态字段的弱基线分析。

后续建议：

- Phase 0 报告中单独统计这些字段缺失带来的匹配风险。
- 不要在缺少外部 ID 和季集号时自动高置信度过滤候选。
- 低置信度匹配必须进入人工确认。

## 4. 后续编码检查清单

实现飞书客户端或 CLI 时，必须检查：

- 是否从配置读取密钥，而不是硬编码。
- 是否隐藏日志中的 Secret 和 Token。
- 是否先验证 token，再验证 table 访问。
- 是否把读正式表和写测试表分开。
- 是否对每个 API 返回检查 `code`。
- 是否对 `429`、网络错误和临时失败预留重试。
- 是否支持分页读取 records。
- 是否支持创建后重新查询资源 ID。
- 是否把 `table_id`、字段名和字段 ID 缓存到配置或本地状态。
- 是否明确报告哪些检查通过、哪些失败、下一步需要用户做什么。
