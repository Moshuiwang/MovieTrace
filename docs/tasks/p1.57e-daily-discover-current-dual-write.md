## 相关上下文
- 这是过渡双写任务，目的是让新表在不破坏现有输出的情况下积累真实数据。
- P1.57i 才会停止旧 discovery 事件行输出。
- 当前 `_write_content_updates`（`discovery.py:800`）在循环结束后调用 `conn.commit()`；`run_discovery` 中已有多处 `conn.commit()`（line 109、229、493）。要让 current/observation/`content_updates` 在同一事务内提交，必须**把 `_write_content_updates` 内部 commit 上移到 run_discovery 写库阶段的统一收尾**，否则同事务约定无法兑现。

## 输入
- P1.57b helper
- P1.57d observation eligibility helper
- 当前 `run_discovery` 候选和 source summary

## 输出
- `daily-discover` commit 模式写 current item + observation
- dry-run 不写 current item / observation
- stats 中新增 current/observation 写入计数

## 具体要求
1. 只有通过 P1.57d eligibility 的候选才写 current / observation。
2. dry-run 不写任何新表。
3. 写 current / observation 与旧 content_updates 写入应在同一事务内，失败时不产生半写。
   - 实施要点：删除 `_write_content_updates` 内部 `conn.commit()`，由 `run_discovery` 在三类写入全部完成后统一 commit；写库阶段中途失败必须 rollback，不能落下半组数据。
   - 其他与写库无关的 commit（line 109、229 等步骤进度提交）保留，避免扩大事务边界引入新锁竞争。
4. stats 字段名称清晰，例如 `current_discovery_created`、`current_discovery_updated`、`observations_written`、`observations_skipped_source_date_fallback`。
5. 旧 CLI 输出结构尽量兼容；新增 stats 不破坏 notify。

## 验收标准
1. ✅ commit 模式有效候选写入 current item 和 observation。
2. ✅ dry-run 不写 current item / observation。
3. ✅ pure source-date fallback 不写 observation。
4. ✅ 旧 `content_updates new_discovery` 仍按现状写入，导出/飞书不受影响。
