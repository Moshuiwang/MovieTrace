# 任务包：修复 HTTP 客户端 gzip 解压缺失

**任务包版本：** v1
**创建时间：** 2026-05-20 +08

## 任务名称
P1.35 fix-gzip-http — HTTP get_json() 未处理 gzip 压缩响应

## 任务类型
fix

## 当前阶段
Phase 1

## 来源任务
生产日志 daily_20260520.log：TMDb trending / IMDb backfill / TMDb detail 共 16 条失败，
错误均为 `'utf-8' codec can't decode byte 0x8b in position 1`（gzip magic byte）。

## 目标
`sources/http.py:get_json()` 在读取 response 后检查 `Content-Encoding`，若为 gzip 先解压再 decode，
消除 TMDb API 返回压缩响应时的 UnicodeDecodeError。

## 非目标
- ❌ 不改 `feishu/_http.py`（飞书 API 不返回 gzip，无需改）
- ❌ 不改任何调用方代码
- ❌ 不改 timeout / retry 逻辑

## 允许修改范围
- `src/movietrace/sources/http.py`（仅 `get_json()` 函数内 + 顶层 import）

## 禁止修改范围
- 🚫 `src/movietrace/feishu/` 所有文件
- 🚫 `src/movietrace/sources/` 其他文件
- 🚫 AGENTS.md、CLAUDE.md

## 相关上下文
- Python `urllib.request.urlopen` 不自动解压 gzip，需手动检查 `Content-Encoding` header
- gzip magic bytes：`\x1f\x8b`，即 byte 0x1f 0x8b；0x8b 在 position 1 是明确特征
- `gzip` 是 stdlib，无需新依赖

## 输入
- `src/movietrace/sources/http.py` 现状

## 输出
- 修改后的 `src/movietrace/sources/http.py`
- 新增测试：`tests/test_http_client.py`（模拟 gzip 响应，验证正确解压）

## 具体要求
1. 在 `get_json()` 的 `urlopen` 块内，`response.read()` 之后：
   - 检查 `response.headers.get("Content-Encoding") == "gzip"`
   - 若是，用 `gzip.decompress(raw)` 解压，再 `.decode("utf-8")`
   - 否则直接 `.decode("utf-8")`（保持现有行为）
2. 在文件顶层加 `import gzip`（stdlib，无新依赖）
3. 改动仅限 3 行以内

## 验收标准
1. ✅ 新增测试：mock 一个返回 `Content-Encoding: gzip` 的响应，`get_json()` 能正确解析 JSON
2. ✅ 新增测试：mock 普通响应（无 Content-Encoding），行为与改前一致
3. ✅ `PYTHONPATH=src python -m pytest tests/ -v` 全量通过，无新失败

## 测试要求

  - gzip 压缩响应能正确解压并返回 JSON
  - 无压缩响应不受影响

## 风险点
- TMDb 并非每次都返回 gzip，只有部分响应带 `Content-Encoding: gzip`；改动对无压缩响应透明，无风险
- 极少数情况服务端发 `Content-Encoding: gzip` 但 body 实际未压缩：`gzip.decompress` 会抛 `BadGzipFile`，
  这属于服务端 bug，可在日志里捕获定位，暂不处理
