## 具体要求

### 1. Policy 接口

```python
@dataclass
class HttpPolicy:
    timeout: float = 15.0
    max_retries: int = 3                       # 5xx / 网络错误重试上限
    backoff_base: float = 1.0                  # 指数退避基数
    respect_retry_after: bool = True           # 429 时按 Retry-After header 等待
    retry_on_status: tuple[int, ...] = (502, 503, 504)
    log_context_required: bool = True          # 强制传 log_context，便于 api_usage_log

def request_with_policy(
    url: str, *, method: str = "GET", headers: dict | None = None,
    params: dict | None = None, data: bytes | None = None,
    policy: HttpPolicy | None = None,
    log_context: dict | None = None,
) -> tuple[int, bytes, dict]:
    """统一 HTTP 请求入口。返回 (status_code, body_bytes, response_headers)。
    不做 JSON 解析（让上层决定），不抛业务异常（除非超过重试上限）。"""
```

### 2. 行为约束

- **超时**：默认 15s，可被 policy 覆盖；两个 HTTP 入口必须一致
- **5xx 重试**：`502/503/504` 按 `backoff_base * 2^attempt` 退避，max_retries 上限
- **429 处理**：读 `Retry-After` header（秒数或 HTTP-date），合规规则允许的最大等待 60s 上限；超过 60s 直接放弃当次
- **4xx 不重试**：404/403/401/400 立即返回给上层（业务侧决定如何处理）
- **网络异常**（`URLError` / socket timeout）：按 5xx 同策略重试
- **日志**：每次最终结果（成功 / 放弃）写 `api_usage_log`（status: ok / error / rate_limited / quota_error），与现有 schema 兼容

### 3. 改造调用方

`sources/http.py:get_json`：内部改用 `request_with_policy`，外部签名 / 返回值不变。
`feishu/_http.py:request_json`：同上。

两个入口各自维护自己的"JSON 解析 + 错误格式化"逻辑（这是上层职责），policy 层只管传输。

### 4. 不引入 connection pooling

urllib stdlib 不擅长 keep-alive。本任务**不做** pooling——如未来要做，单独评估是否引入新依赖（届时走 ADR）。

## 风险点

- **改造范围控制**：本任务很容易诱发"顺手重构"。**严格**限定为提取 policy 层 + 两个入口接入；business 调用方一行不动
- **测试 mock 路径漂移**：现有测试若 patch 在 `get_json` 内部某行，policy 抽离后该行可能不存在。需要重新选 patch 点，但不能改断言强度
- **429 Retry-After 解析**：HTTP-date 格式较复杂——本任务只支持"秒数"格式，遇到 HTTP-date 按 60s 上限保守处理
- **与 P1.45 的边界**：P1.45 飞书 batch_create 重试是业务层（"飞书拒绝了我的 record"），P1.46 是传输层（"网络挂了 / 服务端 5xx"）。两层都有重试不冲突；测试要明确边界
- **api_usage_log 字段语义**：policy 层只写传输层 status；业务层 quota_error 等仍由上层判定
