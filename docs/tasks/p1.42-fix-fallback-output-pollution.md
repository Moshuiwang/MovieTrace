# 任务包：P1.42 fix-fallback-output-pollution

**任务包版本：** v1
**创建时间：** 2026-05-22 +08

## 任务名称
P1.42 修复 fallback 数据被推送为"今日新发现"的语义 bug

## 任务类型
fix

## 当前阶段
Phase 1（V1 运行观察期）

## 来源任务
[`docs/reviews/architecture_audit_2026_05.md § 1.7`](../reviews/architecture_audit_2026_05.md) — 第一派与第二派一致认为是 P0 语义 bug。设计本意（API 限量时不阻断 pipeline 让其他渠道继续跑）正确，但出口处把 fallback 数据当今日新发现推送，造成飞书污染 + API 配额浪费。

## 目标

让纯 fallback 来源的候选**不写入 `content_updates`**、**不推送飞书**；混合来源（含至少一个 fresh 源信号）按原逻辑正常推送。

## 根因

- `discovery.py:390` 写候选时硬塞 `c_dict["snapshot_date"] = snapshot_date`（今日），覆盖了 source 级真实快照日
- `content_updates` 表无 `source_snapshot_date` 字段，下游无从判断"信号来自哪一天"
- 飞书推送看到的是"今日 priority >= 阈值的事件"，纯 fallback 旧数据被当今日新发现推走

## 非目标
- ❌ 不修改 fallback 抓取机制本身（`_resolve_source_dates_with_fallback` / `_find_fallback_snapshot` 保留）
- ❌ 不修改 `content_updates` schema（不加新字段、不引入 migration）
- ❌ 不引入 Ledger 或跨日去重机制
- ❌ 不修改飞书推送层（`sync.py` / `notify.py` 不动）
- ❌ 不改 `MergedCandidate` 数据结构本身

## 允许修改范围

## 具体要求

1. **判定 fresh 信号**：在 `run_discovery` Step 2 merge 后、Step 3 enrichment 前，给每个候选标记：

   ```python
   fresh_sources = {s for s, d in source_dates.items() if d == snapshot_date}
   for c in candidates:
       contributing = c.get("contributing_sources") or set()  # 来自 merge 已知字段
       c["has_fresh_signal"] = bool(contributing & fresh_sources)
   ```

   如果 `MergedCandidate` 当前没有"该候选由哪些源贡献"的字段，从已有 `fp_data` / `tmdb_data` / `trakt_data` 是否非空推断（优先复用，不新建字段）。

2. **过滤**：Step 5 阈值过滤后追加：
   ```python
   passed = [c for c in passed if c.get("has_fresh_signal") or c.get("is_soap")]
   ```
   （Soap bypass 行为保留——Soap 强制通过本来就属于业务豁免）

3. **统计字段**：在 `stats` 中增加 `suppressed_fallback_only=N`，便于运营观察。**不**写入 `content_updates`；只在 stats / 日志输出。

4. **不动 `snapshot_date` 覆盖那一行**：`c_dict["snapshot_date"] = snapshot_date` 保持现状——它是候选的"批次日"语义，不是数据快照日；下游字段语义不变，避免回归。

## 风险点

- **MergedCandidate 是否已有 "由哪些源贡献" 字段**：如果没有，需用 `fp_data` / `tmdb_data` / `trakt_data` 非空推断。若实际结构与预期不符，停下来确认再编码（不新建字段、不改 dataclass 签名）
- **Soap bypass 与 fresh 过滤的交互**：Soap 是业务豁免阈值，本任务保留 Soap 通过路径。若 Soap 来自纯 fallback 源，仍允许通过——后续若发现 Soap fallback 也想抑制，单独提任务
- **fixture 调整边界**：若现有 e2e 测试依赖"fallback 数据进库"才能通过，必须把 fixture 改为 fresh，**不能**为了让测试通过而放宽过滤条件
