### scoring.py 字段需求(已存在,只是当前没数据)

- `fp_items: list[{ranking, days_total, platform}]` — FP 维度
- `ext_data: {tmdb_popularity, tmdb_vote_average, tmdb_vote_count, imdb_rating, imdb_votes, trakt_watchers}` — 其他源

### OMDb 实测

- 按 imdb_id 查 `?i=tt{imdb_id}`,返回 `imdbRating` + `imdbVotes`
- FP 数据 `imdb_id` 97% 填充,但格式是纯数字(`1190634`),**需要补 tt 前缀并补零到 7 位**
- TMDb trending 返回里没有 imdb_id,只能通过 FP 或后续详情接口拿
- OMDb 免费配额 1000 次/天,90 条远低于上限

## 具体要求

### 1. 多源并集(`multi_source_merge.py`)

```python
@dataclass
class MergedCandidate:
    tmdb_id: int | None
    imdb_id: str | None
    title: str
    media_type: str  # 'tv' | 'movie'
    fp_items: list[dict]          # 来自 flixpatrol_top10
    tmdb_data: dict | None        # 来自 tmdb_trending(单条,从 trending/day 优先)
    trakt_data: dict | None       # 来自 trakt_trending
    source_flags: set[str]        # 出现在哪些源 {'fp','tmdb','trakt'}

def merge_three_sources(
    conn: sqlite3.Connection,
    snapshot_date: str,
) -> list[MergedCandidate]:
    """
    1. 读三张表当日数据
    2. 按 tmdb_id 主键合并(imdb_id 兜底)
    3. 没 tmdb_id 也没 imdb_id 的条目用 (title, media_type) 兜底键
    4. 返回去重后的 MergedCandidate 列表
    """
```

#### 合并规则

- **主键优先级**:tmdb_id > imdb_id > (title_norm, media_type)
- **title_norm**:小写、去空格、去标点
- **同一 tmdb_id 在 FP/TMDb/Trakt 都出现**:合并为一条,各源数据分别填入对应字段
- **冲突解决**:title 采用 TMDb 的(更标准),原始 FP title 保留在 fp_items 里

### 2. OMDb 富化(`omdb_enrichment.py`)

```python
def enrich_with_omdb(
    conn: sqlite3.Connection,
    candidates: list[MergedCandidate],
    omdb_api_key: str,
    cache_ttl_hours: int = 24,
) -> None:
    """
    对每个 candidate:
    1. 若已有 imdb_id,补 tt 前缀(数字 → tt0001234 格式)
    2. 查 api_cache 表是否有未过期的缓存
    3. 命中缓存 → 解析 imdbRating/imdbVotes 填入 ext_data
    4. 未命中 → 调 OMDb API,写缓存,填 ext_data
    5. 缺 imdb_id 的跳过(不报错)
    """
```

OMDb 缓存键格式:`omdb:{imdb_id_with_tt}` 存入现有 `api_cache` 表。

### 3. 富化扩展(从 trending 表/TMDb 详情补 release_date 和 language)

- TMDb trending 里的条目天然带 `release_date` 和 `original_language`
- FP 来源但未在 TMDb trending 出现的条目(预计 60% 左右),需要 `TmdbDetailClient.get_tv_details` / `get_movie_details` 按 tmdb_id 单条查
  - **本任务实现**:复用 P1.5-C 已有的 `TmdbDetailClient`
  - 缓存键:`tmdb:detail:{tmdb_id}:{media_type}`,缓存 24h
- 每天最多额外 60 次 TMDb 调用

### 4. 评分集成(改 `discovery.py`)

新的 `run_discovery` 主流程:

```python
def run_discovery(date_from, dry_run, ...):
    # 1. 确保三源数据(P1.7-B/C 的 fetch 函数)
    if not dry_run:
        _ensure_fp_data(conn, date_from)
        # 新增:
        fetch_and_store_tmdb_trending(...)
        fetch_and_store_trakt_trending(...)

    # 2. 多源合并
    candidates = merge_three_sources(conn, date_from)

    # 3. 富化
    enrich_with_omdb(conn, candidates, omdb_key)
    enrich_with_tmdb_details(conn, candidates, tmdb_token)  # 仅未在 trending 出现的

    # 4. 打分
    for c in candidates:
        hot, breakdown = compute_hot_score(_to_scoring_dict(c), cfg)
        c.hot_score = hot
        c.priority = map_priority(hot)
        c.score_breakdown = breakdown

    # 5. 阈值过滤
    threshold = cfg.get("priority_thresholds", {}).get("P2", 50)
    passed = [c for c in candidates if c.hot_score >= threshold]

    # 6. 写入 content_updates(替代写入 candidates 表)
    write_content_updates(conn, passed, snapshot_date=date_from)
```

### 5. content_updates 写入策略

- `content_update_id` 命名:`discovery:{tmdb_id}:{snapshot_date}`(去重粒度)
- `update_type`:
  - 在 A 库且 in_baseline=True → `re_promotion`(已有可补充)
  - 不在 A 库 → `new_discovery`(新增推荐)
- `priority` / `hot_score`:来自打分
- `source_summary_json`:
  ```json
  {
    "fp": {"platform":"netflix","ranking":1,"days_total":32},
    "tmdb": {"popularity":569.2,"vote_average":8.45,"vote_count":12247},
    "trakt": {"watchers":5520,"rating":8.38,"votes":36983},
    "imdb": {"rating":"8.6","votes":"853,757"},
    "release_date": "2019-07-25",
    "language": "en"
  }
  ```
- `match_confidence_low`:从 candidate_matches 表查(若该 canonical_item 有低置信度记录)

## 风险点

- **content_updates 当前没有 hot_score 列或 priority 列默认值约束**:本任务需要在 INSERT 时确保字段完整
- **api_cache 表已有但未被使用**:本任务首次启用,确认 `expires_at` 字段语义(YYYY-MM-DD HH:MM:SS UTC vs local)
- **`source_summary_json` 字段已存在**(P1.5),只需要扩展格式,不需要新增列
- **content_update_id 冲突**:同一 tmdb_id 同一天可能被功能 1 和功能 2 都触发,需要明确分离:
  - 功能 1:`discovery:{tmdb_id}:{snapshot_date}`
  - 功能 2:`new_season:vs_{vs_id}:s{season}`(已有约定)
- **A 库匹配率**:`in_baseline` 判断依赖 P1.5-E 的 canonical_item_id 标记,可能部分新候选未匹配到 → 默认 `in_baseline=False`
- **测试 mock 复杂度**:三源 + OMDb + TMDb 详情,测试需要良好的 fixture 设计,**建议把 mock 工厂函数提到 conftest.py**
