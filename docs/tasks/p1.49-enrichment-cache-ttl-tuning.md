# 任务包：P1.49 enrichment-cache-ttl-tuning

**任务包版本：** v1
**创建时间：** 2026-05-22 22:20 +08

## 任务名称
P1.49 提升 OMDb / TMDb detail 缓存命中率——TTL 从 24h 延长至 72h

## 任务类型
refactor

## 当前阶段
Phase 1（V1 运行观察期）

## 来源任务
用户性能观察：2026-05-22 live run 实测——OMDb enrichment 169 条中 94 次真实 API 调用 / 72 次缓存命中（命中率 43%），耗时约 3 分钟；TMDb detail enrichment 耗时未知但类似。
对比 P1.44 已将 TMDb search TTL 改为 72h 的先例，OMDb 和 TMDb detail 的 24h TTL 过短——影视评分和详情数据几天内几乎不变，延长 TTL 可大幅提升命中率、缩短 pipeline 耗时。

## 目标

将 `enrich_with_omdb()` 和 `enrich_with_tmdb_details()` 的默认 `cache_ttl_hours` 从 **24 → 72**，使 pipeline 在连续两天运行后缓存命中率显著提升，OMDb 阶段耗时从 ~3 分钟降至 ~1 分钟以内（缓存命中不 sleep）。

## 非目标
- ❌ 不改缓存存储结构（`api_cache` 表 schema 不变）
- ❌ 不改 TMDb search TTL（P1.44 已设为 72h，不动）
- ❌ 不改 OMDb 的 1s sleep 合规等待（只减少需要 sleep 的次数）
- ❌ 不改 TMDb detail 的 skip 条件（当前逻辑正确，不过度激进）
- ❌ 不改函数签名（默认值变更，调用方零感知）

## 允许修改范围
- `src/movietrace/pipeline/omdb_enrichment.py`（两处默认值：`enrich_with_omdb` + `enrich_with_tmdb_details`）
- `tests/test_omdb_enrichment.py`（确认新 TTL 默认值的测试）

## 禁止修改范围
- 🚫 `src/movietrace/pipeline/discovery.py`（调用方不传 TTL，沿用默认值即可）
- 🚫 `src/movietrace/sources/tmdb.py`
- 🚫 `src/movietrace/sources/omdb.py`
- 🚫 `src/movietrace/db/`
- 🚫 任何缓存读写逻辑

## 风险点
- **缓存过期边界**：72h TTL 意味着最多 3 天后才强制刷新。若某部剧评分在 3 天内大幅变化（极少见），展示值会略滞后——对运营决策无实质影响
- **已有缓存不受影响**：现有 24h 内写入的缓存条目，72h TTL 下会继续有效（TTL 是读取时的过期判断，不改写已有数据）
- **与 P1.47 / P1.48 无冲突**：修改范围不重叠；建议执行顺序 P1.47 → P1.48 → P1.49（P1.49 独立，可并行）
