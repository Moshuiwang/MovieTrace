# 任务包：P1.5-D 功能 2 基线内容主动追踪模块

**任务包版本：** v1
**创建日期：** 2026-05-11
**依赖：** P1.5-B（schema v5）+ P1.5-C（virtual_series 表已回填）

## 任务名称

P1.5-D 实现 ADR-0007 § 决策一中的"功能 2"：每日按 TMDb 剧集 `status` 字段智能轮询 A 库已有 TV 剧集，检测是否有新季/新集发布，写入 B 库 `content_updates` 表（`is_in_baseline=true`，不受 `hot_score` 阈值约束）。电影完全跳过。

## 任务类型

feat（新增核心功能模块 + 调度逻辑）

## 当前阶段

Phase 1.5（V1 定位翻转，ADR-0007）

## 来源任务

- [ADR-0007](../decisions/0007-repositioning-to-update-tracking.md) § 决策一（功能 2）+ § 决策四（智能轮询）
- [`docs/requirements.md` § R13](../requirements.md) 基线内容主动追踪原子需求

- P1.5-C 已回填的 virtual_series 表

## 目标

让系统具备"主动检测 A 库 TV 剧集新季/新集"的能力。每日运行一次，按 TMDb `status` 分层智能轮询，2 周覆盖完所有 `Returning Series` 类剧集，月度低频覆盖 `Ended` 类，跳过 `Canceled` / `Pilot` / `Planned`。

## 非目标

- ❌ **不追踪电影**（ADR-0007 § 决策四明确：电影完全跳过功能 2）
- ❌ **不修改 hot_score 计算逻辑**（功能 2 不受 hot_score 约束）
- ❌ **不实现飞书写入**（属 P1.5-E，本任务只写入 B 库 `content_updates` 表）
- ❌ **不实现日报输出**（属 P1.5-F）
- ❌ **不实现检测/导出解耦**（属 P1.5-G，本任务嵌入到现有 `daily-discover` 流程中作为新增 step）
- ❌ **不重建 virtual_series 关联**（依赖 P1.5-C 已回填）
- ❌ **不追踪集级别（episode）的更新**（V1.5 只追踪到季级别；episode-level 留 V2）

|---|---|---|

**关键设计决策（待用户确认）：**

| 问题 | 选项 | 影响 |
|------|------|------|
| **Q6：新季检测的写入粒度** | (a)每检测到一个新季写一条 `content_updates`（如 S01 → S05 跳变写 4 条新季）/ (b)只写最新一条（如 S01 → S05 只写一条 "S05"）/ (c)累计差值合并一条 | (a) 信息最全；(b) 噪音少；ADR-0007 R13 § 验收 4 倾向 (a) |
| **Q7：轮询调度的"轮次"判定** | (a)每天选 N 条（N=`Returning Series 数 / 14`）/ (b)按 `last_polled_at` 排序，每天选 X 条最久未轮询的 / (c)固定每天 30 条 | (b) 最稳健，自然实现"2 周覆盖完" |
| **Q8：功能 2 与 daily-discover 的集成方式** | (a)新增 CLI 子命令 `baseline-track`（独立运行）/ (b)在 `daily-discover` 流程中追加 step（一起跑）/ (c)两者都做（保留独立命令 + 集成到 daily） | (c) 灵活性最高，便于排错和独立验证 |

### 1. 智能轮询调度器（`src/movietrace/pipeline/poll_scheduler.py`）

```python
@dataclass
class PollPlan:
    """单次轮询计划"""
    virtual_series_id: int
    tmdb_tv_id: str
    name: str
    poll_priority: str
    last_polled_at: str | None

def build_daily_poll_plan(
    conn: sqlite3.Connection,
    today: date,
    config: dict,
) -> list[PollPlan]:
    """根据 ADR-0007 § 决策四生成今日轮询计划。

    配置（来自 config.yaml）：
    - urgent_coverage_days: 14  # Returning Series 2 周覆盖
    - normal_coverage_days: 21  # In Production 3 周
    - low_coverage_days: 180    # Ended 6 月
    - skip_priorities: ['skip']
    - daily_max_calls: 50       # 每日 TMDb 调用上限（保护配额）
    """

def update_last_polled(
    conn: sqlite3.Connection,
    virtual_series_id: int,
    polled_at: datetime,
) -> None:
    """更新 virtual_series.last_polled_at"""
```

调度算法：

```
1. 按 poll_priority 分组（urgent / normal / low / skip）
2. 跳过 skip
3. 对每组：
   3.1 计算"每日应轮询数" = group_size / coverage_days
   3.2 按 last_polled_at 升序（NULL 最优先）选出该数量
4. 总计 = urgent + normal + low，如超过 daily_max_calls 则按比例缩减
5. 返回 list[PollPlan]
```

### 2. 基线主动追踪模块（`src/movietrace/pipeline/baseline_tracking.py`）

```python
@dataclass
class NewSeasonEvent:
    """检测到的新季事件"""
    virtual_series_id: int
    canonical_item_id: int | None  # 可能为 None（如果 TMDb 有但 A 库没有这一季）
    tmdb_tv_id: str
    name: str
    new_season_number: int
    previous_local_max_season: int
    detected_at: datetime

def detect_new_seasons(
    conn: sqlite3.Connection,
    poll_plan: list[PollPlan],
    tmdb_client: TmdbDetailClient,
) -> list[NewSeasonEvent]:
    """按轮询计划检测新季事件。"""

def write_content_updates_for_new_seasons(
    conn: sqlite3.Connection,
    events: list[NewSeasonEvent],
) -> int:
    """把新季事件写入 content_updates 表，返回新增记录数。

    每条 NewSeasonEvent 写入：
    - canonical_item_id: 关联到 virtual_series 的 series 级 canonical_item（如已有）；
      或新建一条 season 级 canonical_item
    - update_type: '新季发布'
    - hot_score: NULL 或 0（功能 2 不计算 hot_score）
    - priority: NULL 或 'baseline_tracked'
    - source_summary_json: 含 {'source': 'baseline_tracking', 'tmdb_tv_id': ..., 'season': ...}
    - is_in_baseline: TRUE（virtual_series 在 A 库中）
    - match_confidence_low: FALSE（virtual_series 聚合时已确认）
    - 去重 key: (canonical_item_id, update_type) → ADR-0007 R4
    """

def run_baseline_tracking(
    db_path: str,
    config: dict,
) -> dict:
    """模块主入口：build_poll_plan → detect_new_seasons → write_content_updates。

    返回统计 dict：
    - polled_count
    - new_seasons_detected
    - content_updates_written
    - api_errors
    - skipped_reasons
    """
```

## 具体要求

### A. 调度器设计

1. **配置可调**：所有阈值（覆盖天数、daily_max_calls 等）从 `config.yaml` 读取，不硬编码
2. **优雅降级**：如 virtual_series 表为空（P1.5-C 未执行），调度器返回空列表并 log warning，不抛错
3. **可观察**：返回的 `PollPlan` 列表必须可序列化为 JSON，便于 P1.5-F 日报展示

### B. 检测逻辑

1. **检测判据**：`virtual_series.tmdb_number_of_seasons > virtual_series.local_max_season` → 有新季
2. **新季编号集**：从 `local_max_season + 1` 到 `tmdb_number_of_seasons`，按 Q6 决策处理（推荐 (a)，逐季写入）
3. **去重**：`(canonical_item_id, update_type='新季发布', season_number)` 复合去重（依靠 `content_updates` 唯一索引 `ux_content_updates_item_type` —— 注意 P1.5-B 未修改此索引，可能需要补充季号到去重 key；如索引不足，本任务可补 migration 006，但**优先选 (a)**：在写入前先 `select` 验证不存在）

### C. 错误隔离

1. 单条剧集 TMDb API 失败 → 跳过本条，不中断批量
2. 单条 content_updates 写入失败 → 记录错误，继续下一条
3. 全部失败 → 抛出 `BaselineTrackingError`，被 daily-discover 主流程捕获记入 `result['baseline_tracking_error']`，不导致 `daily-discover` 整体失败

### D. 写入 B 库的字段对齐

- `canonical_item_id`：从 `virtual_series_id` 反查 series 级 canonical_item；如不存在（极少数情况），创建一个 season 级 canonical_item 作为该新季的主键载体（这是新增逻辑，需测试）
- `update_type`：固定 `'新季发布'`（与 `requirements.md` § 11.1 一致）
- `priority`：固定 `'baseline_tracked'`（新增 priority 值，明确区别于 hot_score 派生的 P0/P1/P2/P3）—— **注：** P1.5-E 翻新飞书写入时,需要处理这个新的 priority 值
- `hot_score`：NULL 或 0（功能 2 不计算）
- `baseline_match_status`：不再写入（V1.5 已删除该字段，参考 P1.5-A § 12.2 改动）
- `match_confidence_low`：FALSE（功能 2 的源数据来自已聚合的 virtual_series，置信度高）
- `source_summary_json`：包含 `{"source": "baseline_tracking", "tmdb_tv_id": "...", "detected_season": N, "previous_local_max": M, "tmdb_status": "..."}`

### E. 与 daily-discover 集成

1. 功能 2 作为 daily-discover 的最后一个 step（位于 hot_score 评分、基线匹配之后）
2. `config.baseline_tracking.enabled` 控制开关（默认 true）
3. 失败不阻断 daily-discover 整体，单独记录到 result dict

### F. CLI 命令

```
PYTHONPATH=src python -m movietrace.cli baseline-track [--dry-run] [--limit N] [--db path]
```

- `--dry-run`：只走流程不写 DB
- `--limit N`：覆盖 daily_max_calls，只跑前 N 条
- `--db`：DB 路径

## 风险点

1. **风险 1：TMDb API 配额触发风险**。第一次实际运行可能比预估多（如 daily_max_calls=50 但当天 Returning Series 选 36 条 + Ended 选 12 条 = 48 条 < 50，OK；但如果某天 In Production 集中爆发，可能超）。**对策：** `daily_max_calls` 严格作为硬上限，超出的剧集本日跳过、下一日优先。
2. **风险 2：去重索引不足**。现有 `ux_content_updates_item_type` 索引按 `(canonical_item_id, update_type)` 去重，但同一剧不同新季都是 `update_type='新季发布'`，如不加 season_number 会误判为重复。**对策：** "具体要求 § B"中已说明（先 select 验证，避免改索引）；如发现性能问题，再补 migration 006。
3. **风险 3：A 库无对应 canonical_item 的新季处理**。例如 A 库只有 `Breaking Bad S01` 但 TMDb 显示 `Breaking Bad` 已到 S05，那 S02-S05 这 4 季在 B 库的 canonical_items 表中可能不存在。**对策："**具体要求 § D"中已说明（创建 season 级 canonical_item 作为载体）；如该剧的实体匹配是低置信度，跳过。
4. **风险 4：与 P1.5-E 飞书写入的耦合**。功能 2 写入 `content_updates` 的字段集（特别是 priority='baseline_tracked' 这个新值）需要在 P1.5-E 飞书写入时正确处理。**对策：** 在 P1.5-D 完成报告中明确告知 P1.5-E 起草者"`priority` 字段新增 `baseline_tracked` 值，飞书写入时应映射为'基线追踪'或类似中文显示"。
5. **风险 5：首次实际运行可能产生大量新季事件**。如果上次 V1 运行后过去了好几个月，许多剧集已经更新过多季，第一次跑 P1.5-D 可能写入几十甚至上百条 `content_updates`。**对策：** 在生产 DB 上首次运行前明确"首次运行预期写入量大,后续运行才会稳定到每日几条"。
6. **风险 6：Q6/Q7/Q8 决策不在系统侧可控**。开工前必须澄清。
