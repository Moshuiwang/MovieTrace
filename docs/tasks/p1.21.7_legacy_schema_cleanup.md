## 来源任务

## 目标

明确每张 ADR-0007 前的遗留表的现状（真正死 / transient / 仍在用），删除真正死的表+代码，截断 transient 表的存量过时数据，写一份 ADR 说明取舍。**完成后 DB 没有混淆的"老/新"双路径**。

## 非目标

- ❌ 重构仍在用但属于翻转前路径的代码（如 baseline_import.py / entity_matching.py / baseline_matching.py 的整体重构）
- ❌ 删除任何在 ADR-0007 之后还被引用的表
- ❌ 改 content_updates / virtual_series / canonical_items / upstream_* 等核心表
- ❌ 把 transient 表（match_candidates 等）改为永久存储

## 输出

1. **审计报告**（任务执行的 step 1）：每张表用 `rg` / `grep` 检查所有引用，明确分类并写入 ADR-0014。
2. **migration 016**：按分类执行：

3. **代码删除**：把真正不再调用的模块删掉（baseline_import.py、daily_writer.py、可能还有 CLI 子命令）。

## 具体要求

### Step 1：审计（不动代码，仅产报告）

对每个候选表运行：
```bash
rg -n "(?i)\b{table_name}\b" src/ --type py
```

输出形如：
```
- baseline_items
  - src/movietrace/pipeline/baseline_import.py:53 INSERT INTO ...
  - src/movietrace/pipeline/entity_matching.py:129 SELECT FROM ...
  - src/movietrace/db/schema.py:line CREATE TABLE
  调用入口：cli.py 中有 cmd_xxx 吗？grep cli.py 确认
```

判断标准：
- **可 DROP**：表无 INSERT/UPDATE 调用点 OR 所有调用点都在已死的模块里
- **应清空**：表仍有 INSERT 调用，但残留数据无业务意义（如 transient 中间态）
- **保留**：表仍有写入且历史数据有意义

也要 grep `scripts/`、`tests/`、`docs/` 中的引用。如果某模块（如 baseline_import.py）整个不再调用，也是删除候选。

### Step 2：migration 016 SQL

按 step 1 的结论编写。模板：
```sql
-- Migration 016: drop ADR-0007 pre-flip legacy tables

DROP TABLE IF EXISTS source_records;
DROP TABLE IF EXISTS feishu_import_runs;
DROP TABLE IF EXISTS baseline_items;
DROP TABLE IF EXISTS candidates;          -- 如果 step 1 确认 dead
DROP TABLE IF EXISTS candidate_matches;   -- 如果 step 1 确认 dead

DELETE FROM match_candidates;             -- 仍在用但清空残留
DELETE FROM baseline_quality_issues;      -- 仍在用但清空残留（待 step 1 确认是否保留）

INSERT OR IGNORE INTO schema_migrations(version) VALUES (16);
```

### Step 3：代码删除

如果 step 1 确认整个模块死掉，删除：
- `src/movietrace/pipeline/baseline_import.py`
- `src/movietrace/pipeline/baseline_matching.py`（如果只服务 candidate_matches）
- `src/movietrace/reports/daily_writer.py`（如果其消费者已废弃）
- `src/movietrace/db/schema.py` 中对应的 CREATE TABLE 语句（不影响新装；旧装由 migration 016 覆盖）
- CLI subcommands（如 `cmd_import_feishu_baseline`、`cmd_match_baseline` 等若存在）
- 对应测试文件

### Step 4：DB 备份 + 应用 migration

```bash
cp data/movietrace.db data/movietrace_backup_$(date +%Y%m%d_%H%M)_pre_p121.7.db
PYTHONPATH=src .venv/bin/python -c "from movietrace.db.schema import initialize_database; initialize_database('data/movietrace.db')"
```

验证：被 DROP 的表不存在；schema_version = 16。

### Step 5：ADR-0014

`docs/decisions/0014-legacy-schema-cleanup.md`：
- **背景**：ADR-0007 翻转、ADR-0012 事件历史、ADR-0013 状态快照后，pre-flip 流水线被绕过但 schema 没收
- **决策**：哪些表 drop / 哪些清空 / 哪些保留
- **每张表的判断依据**（引 step 1 审计结果）
- **影响**：删了哪些 .py、哪些测试、CLI 是否减少了命令
- **后续**：如果未来需要类似数据，应优先复用 content_updates / virtual_series 不要重建旧表

## 已知风险与不确定点

- **R1**：`candidate_matches` 可能仍被 `daily_writer.py` 引用，而 daily_writer 是否还被任何 CLI/脚本调用需确认。若被调用，不能 drop。审计需仔细。
- **R2**：`baseline_quality_issues` 仍被 entity_matching 写入，但数据历史意义不明。安全做法是清空残留 + 保留表；若证明完全无业务价值，再纳入 drop 候选。
- **R3**：删除 `baseline_import.py` 后若仍有第三方脚本 `import movietrace.pipeline.baseline_import` 会 ImportError。grep 范围要包括 `scripts/`、`docs/` 代码块、CI 配置。
- **R4**：旧 backup（如 `data/movietrace_backup_20260514_2028.db`）保留了旧表数据 — 这是有意，不要动备份文件。
- **R5**：migration 016 是单向的（DROP TABLE 不可逆）。备份必须先做。
