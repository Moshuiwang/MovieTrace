**任务包版本：** v1
**创建日期：** 2026-05-11

## 任务名称

P1.5-C 实现 `virtual_series` 表的一次性回填脚本：遍历 B 库中所有 TV 类型的 baseline_items / canonical_items，通过现有 entity_matching 结果或新调用 TMDb API 获取 tmdb_tv_id，创建对应的 `virtual_series` 记录，并把同剧的 season 级 canonical_items 关联到 virtual_series。**电影完全跳过。**

## 目标

让 B 库的"剧集聚合层"从空表变为可用状态：
- 所有 TV 类型的 baseline_items / canonical_items 都能聚合到一个 virtual_series 记录
- 每条 virtual_series 携带 TMDb 返回的 `name`、`status`、`number_of_seasons`、`poll_priority`
- 后续 P1.5-D 的智能轮询可以直接基于 virtual_series 表运行

## 非目标

- ❌ **不动电影记录**（electric_type='movie'）—— 电影不需要 virtual_series 聚合
- ❌ **不实现功能 2 主动追踪逻辑**（属 P1.5-D）
- ❌ **不修复实体匹配中的低置信度记录**（保留 NULL 关联，运营后续修复）
- ❌ **不重写 entity_matching 模块**（只读取现有匹配结果）
- ❌ **不修改飞书写入逻辑**（属 P1.5-E）
- ❌ **不实现增量回填**（一次性脚本，运行完即归档；后续新数据由 P1.5-D 增量维护 virtual_series）

### 3. 一次性回填脚本（`scripts/p1.5_c_backfill_virtual_series.py`）

脚本主流程：

```
1. 连接 B 库
2. 读取所有 canonical_items WHERE content_type='tv' AND virtual_series_id IS NULL
3. 对每条记录：
   3.1 查 external_ids 表，找 source='tmdb' 的 external_id
   3.2 如有 → 调 TMDb /tv/{id} 取 details
        → upsert virtual_series
        → 更新 canonical_items.virtual_series_id
   3.3 如无 → 按 Q3 决策（重新调 search/multi 或跳过）
   3.4 如低置信度 → 按 Q4 决策（写入 baseline_quality_issues）
4. 限流 1 次/秒，捕获 HTTP 错误，失败记录到日志
5. 跑完输出统计：
   - 总 TV 记录数
   - 成功聚合数
   - 失败原因分布
   - 总 API 调用次数
6. 生成 reports/p1.5_c_backfill_report.md
```

## 具体要求

### A. virtual_series 模块设计

1. **幂等性**：`upsert_virtual_series` 必须支持重复调用（如脚本中断重跑），通过 `tmdb_tv_id` 唯一索引保障
2. **可复用性**：模块必须可被 P1.5-D 直接复用（P1.5-D 在增量场景下也需要 upsert virtual_series）
3. **错误隔离**：单条记录失败不应中断整批，记录到 `baseline_quality_issues` 表（沿用 V1 表）

### B. TMDb tv details 客户端

1. **不破坏现有 V1 调用**：现有 `TmdbSearchClient`（仅 search/multi）保持不动；新增 `TmdbDetailClient`（含 tv details 等详情接口）
2. **错误处理**：HTTP 4xx 记录后跳过；5xx 重试 3 次（指数退避）；超过重试上限后跳过本条
3. **限流**：调用间隔 ≥ 1 秒（保守，远低于 TMDb 50/秒上限）

### C. 一次性回填脚本

1. **CLI 接口**：`python scripts/p1.5_c_backfill_virtual_series.py [--dry-run] [--limit N] [--db path]`
   - `--dry-run`：只走流程不写 DB，用于验证
   - `--limit N`：只处理前 N 条 TV 记录（用于小规模测试）
   - `--db`：数据库路径，默认 `data/movietrace.db`
2. **可中断重启**：中途 Ctrl-C 后，重跑应跳过已处理的 canonical_items（通过 `virtual_series_id IS NOT NULL` 判断）
3. **进度输出**：每处理 10 条输出一次进度（`[i/total] tmdb_tv_id=... name=...`）
4. **失败汇总**：跑完输出失败原因 dict（如 `{'no_tmdb_id': 5, 'api_5xx': 1}`）

### D. Q3/Q4/Q5 决策的代码体现

- **Q3 推荐选 (c)**：先用现有 external_ids，缺失的回退到 search/multi。代码层面分两段 SQL 查询 + 一段 fallback 调用。
- **Q4 推荐选 (c)**：低置信度（mapping_confidence='low' 或 'medium'）→ 不创建 virtual_series，写入 `baseline_quality_issues` 表，issue_type='virtual_series_aggregation_low_confidence'
- **Q5 推荐选 (a)**：通过 `upsert_virtual_series` 的幂等性自然实现 —— 第二条 canonical_items（如 `Breaking Bad S02`）找到同一 TMDb tv_id 时，upsert 返回已存在的 virtual_series.id，仅链接 canonical_items.virtual_series_id

## 风险点

1. **风险 1：TMDb API 配额 / 限流的实际验证**。任务包估算 ~500 次调用 ~9 分钟，但实际可能因为 retry / 5xx 错误更多。**对策：** 在脚本中打印每分钟调用数，发现超过 30/分钟（远低于限额 50/秒）时输出警告。
2. **风险 2：现有 entity_matching 的 external_ids 可能不一致**。V1 可能有 855 条 baseline_items 但只有 500 条匹配到 tmdb_id，其余在 baseline_quality_issues 中。需要在回填前先 grep `external_ids` 表统计实际可用 TMDb tv_id 数量。
3. **风险 3：Q5 多季同剧的合并可能误合并不同剧**。如 `Friends`（1994 NBC）和 `Friends`（2001 韩剧）同名但实际是不同剧，如果实体匹配阶段错误地映射到同一 TMDb tv_id，virtual_series 会错误合并。**对策：** 信任 entity_matching 的现有结果（V1 已通过 P1-A 验证），不在 P1.5-C 重做匹配；如发现错误案例，记入 issues 表由人工修复。
4. **风险 4：脚本中断后的状态恢复**。中途 Ctrl-C 可能让部分 canonical_items 已链接但部分未链接。**对策：** 脚本设计为幂等可重跑（已在"具体要求 § C.2"明确）。
5. **风险 5：生产数据备份**。脚本在生产 DB 上首次实际运行前，必须先 `cp data/movietrace.db data/movietrace.backup.YYYYMMDD.db`，避免不可逆错误。**对策：** 在完成报告中明确"已备份 → 已执行 → 已验证"三步。
