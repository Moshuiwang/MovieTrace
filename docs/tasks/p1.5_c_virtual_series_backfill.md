# 任务包：P1.5-C virtual_series 一次性回填

**版本：** v2（重写，Q5 已定 + A 库接入 + 飞书移除）
**创建日期：** 2026-05-12 +08
**依赖：** P1.5-B（schema v6 migration）+ P1.5-E（实体全量匹配，canonical_items 已填充）

## 任务名称

遍历 B 库中所有 TV 类型的 `canonical_items`，通过 `external_ids` 获取 TMDb tv_id，创建 `virtual_series` 记录，并将同剧的 season 级 canonical_items 关联起来。电影完全跳过。

## 任务类型

feat（新增一次性脚本 + 模块）

## 当前阶段

Phase 1.5（V1 定位翻转，ADR-0007）

## 背景

- A 库（`upstream_programs`）的节目粒度是**季级**，没有 Series 聚合（如 Better Call Saul S01~S06 是 6 条独立记录）
- `canonical_items` 表在 V1 已完成 TMDb 实体匹配，`external_ids` 表存有 TMDb ID 映射（实测 288/289 TV 记录覆盖，99.7%）
- 本次回填是**一次性**的；后续新内容由 P1.5-D 增量维护

### 关键设计决策（已定）

| 问题 | 决策 | 理由 |
|------|------|------|
| **Q3：TMDb tv_id 来源** | 直接用现有 `external_ids` 表 | 实测 99.7% 覆盖；1 条缺失跳过记 issue |
| **Q4：低置信度处理** | 跳过聚合，记入 `baseline_quality_issues` | issue_type='virtual_series_aggregation_low_confidence' |
| **Q5：多季合并 + season_number** | 同名 tv_id 的多季 upsert 到同一条 virtual_series | 幂等设计，season_number 从 name 正则提取 |

## 目标

让 `virtual_series` 表从空表变为可用状态：

- 所有 TV 的 canonical_items 聚合到对应 virtual_series
- 每条 virtual_series 携带 `name` / `tmdb_status` / `number_of_seasons` / `poll_priority`
- P1.5-D 可以直接基于此表运行

## 非目标

- ❌ 不动电影记录（content_type='movie'）
- ❌ 不重做实体匹配（只读现有 external_ids）
- ❌ 不实现增量回填（一次性脚本；P1.5-D 负责增量）
- ❌ 不动 Feishu 相关代码

### 3. 一次性回填脚本（`scripts/p1.5_c_backfill_virtual_series.py`）

```
流程：
1. 查到所有 content_type='tv' AND virtual_series_id IS NULL 的 canonical_items
2. 对每条：
   a. 查 external_ids WHERE source='tmdb'
   b. 有 → 调 TMDb /tv/{id} → upsert virtual_series → link
   c. 无 → 记入 baseline_quality_issues
   d. 低置信度 → 跳过，记入 baseline_quality_issues
3. 限流 1 次/秒
4. 输出统计

CLI：python scripts/p1.5_c_backfill_virtual_series.py [--dry-run] [--limit N] [--db path]
```

## 风险点

1. **TMDb API 限流** — 估 ~200-300 次调用（按 289 TV 去重后 tv_id 数），1 次/秒约 5 分钟。对策：限流 1 次/秒，打印进度。
2. **低置信度未聚合** — 少量 canonical_items 可能无 external_id。对策：记入 baseline_quality_issues，由 P1.5-D 增量处理。
3. **同名不同剧误合并** — 信任 V1 实体匹配结果（已通过 P1-A 验证），不重做匹配。
