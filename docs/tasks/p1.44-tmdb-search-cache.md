## 来源任务

## 目标

为 `TmdbSearchClient.search_tv` / `search_movie` 增加基于 `api_cache` 的本地缓存，TTL 72 小时，命中时跳过网络调用。

## 非目标
- ❌ 不缓存 `/search/multi`（被 search_tv/movie 替代，调用稀少；保持原状）
- ❌ 不修改 `entity_matching` 业务逻辑（candidate 选择算法不动）
- ❌ 不修改 `api_cache` schema
- ❌ 不引入新依赖（用 stdlib `json`、复用现有 cache helper）
- ❌ 不改 `search_tv` / `search_movie` 的入参 / 返回类型

## 允许修改范围
- `src/movietrace/sources/tmdb.py`（在 `search_tv` / `search_movie` 内增加缓存读写）
- 可能新增 `tests/test_tmdb_source.py`（如无则新建）；如有则在原文件追加
- `tests/test_entity_matching.py`（若现有 mock 路径依赖每次调用网络，需适配缓存命中场景）

## 禁止修改范围
- 🚫 `src/movietrace/db/`
- 🚫 `src/movietrace/pipeline/entity_matching.py` 业务逻辑
- 🚫 `sources/tmdb.py` 其他方法（`get_tv_details` / `get_movie_details` / `search_multi`）

## 相关上下文
- 现有 cache helper 可参考：`omdb_enrichment._read_cache` / `_write_cache`（key 形式 `tmdb:detail:{type}:{id}`）
- 合规规则 22 明文：detail endpoint 必须走 24h cache；search 未强制——本任务自定 TTL 72h
- `api_cache` 表结构：`(source, cache_key, response_json, fetched_at, expires_at)`，唯一索引 `(source, cache_key)`
- search 结果会随时间变化（新发布、排序调整），TTL 不可过长

## 输入
- 现有 `api_cache` 表 + TMDb search 调用点

## 输出
- 改造后的 `tmdb.py`
- 新增 search cache 单测

## 具体要求

1. **Cache key 设计**：
   - `tmdb:search:tv:{query_norm}`
   - `tmdb:search:movie:{query_norm}`
   - `query_norm = query.strip().lower()`（不引入 unicodedata 等额外归一化，避免行为漂移）

2. **缓存层位置**：在 `search_tv` / `search_movie` 函数内部、`get_json` 调用前。需要传入 `conn`（或保持 client 持有 conn 引用，参考现有 `TmdbDetailClient` 模式）

3. **TTL**：72 小时（`cache_ttl_hours: int = 72`，可作为参数，默认 72）

4. **读路径**：命中且未过期 → 直接 `parse_tmdb_search_results(cached_dict, default_media_type=...)` 返回，**不**走 `get_json`

5. **写路径**：API 调用成功且 payload 是 dict → 写入 `api_cache`（`insert or replace`）；空结果也缓存（避免反复查无效 query）

6. **不缓存失败**：异常 / 网络错误 / 非 dict payload → 不写 cache，让下次重试

7. **API usage log 仍记录**：缓存命中场景仍写 `api_usage_log`（service=tmdb, endpoint=/search/tv, status=cache_hit），保持可观测性

## 风险点

- **TmdbSearchClient 是否持有 conn**：若 client 当前不持有 `conn`，需要透传或在 client 构造时传入，参考 `TmdbDetailClient(bearer_token, db_path=..., request_date=...)` 模式；不要为缓存而拆类
- **query 归一化**：仅 `.strip().lower()`——不引入更复杂归一化（如全角半角转换），避免行为漂移引发 entity_matching 回归
- **TTL 选型**：72h 是经验值。若 TMDb 上线高峰期 search 排名变化频繁导致 match 质量下降，调短到 24h；首次保守
- **空结果缓存的副作用**：若 query 字面错误导致空 result，72h 内重试都返回空——可接受（人工修正 query 即可）；如果担心可短期跑数据观察
