## 目标

将 `omdb_enrichment.py` 中循环内的 `conn.commit()` 改为批量提交，单次 enrichment 的 fsync 次数从 O(N) 降到 O(1)。

## 根因

`omdb_enrichment.py` 三处循环内 commit：
- L189: `_fetch_tmdb_detail` 写 `api_cache` 后 commit
- L218: `_update_canonical_zh_fields` 更新 canonical_items 后 commit
- L326: `_write_cache` 写 OMDb cache 后 commit

`enrich_with_tmdb_details` 对每个候选会触发 2-3 次 commit；60 候选 = 120-180 次 fsync。

## 非目标
- ❌ 不改 enrichment 业务逻辑（缓存命中、API 调用、字段映射全部保留）
- ❌ 不引入 asyncio / 并发
- ❌ 不改 `api_cache` / `canonical_items` schema
- ❌ 不改函数公开签名（外部调用方零感知）
- ❌ 不顺手重构邻近代码

## 具体要求

1. **去掉 helper 内的 commit**：`_fetch_tmdb_detail`（L189）、`_update_canonical_zh_fields`（L218）、`_write_cache`（L326）三处删掉 `conn.commit()`，**保留 `conn.execute(...)`**

2. **顶层批量提交**：在 `enrich_with_tmdb_details` 与 `enrich_with_omdb` 的循环外（函数返回前）添加单次 `conn.commit()`，或用 `with conn:` 包裹整段循环

3. **异常路径**：
   - 单条候选异常（API 失败、JSON 解析失败等）**继续**处理后续候选，不中断循环（保持现有行为）
   - 仅当函数整体失败时回滚——`with conn:` 已自动处理
   - 不新增 try/except 包装，复用现有 except 块

4. **不改函数签名**：所有 helper 与 public function 的入参 / 返回值不变；调用方零感知

5. **保持日志可读**：现有 logger.info 打印 `api_calls / cache_hits / enriched` 计数不变

## 风险点

- **异常策略选边**：当前 helper 内 try/except 决定"单条失败是否影响整体"。改造前需明确选哪种语义，并保证测试与实现对齐；不可两种都"看起来对"
- **`with conn:` 自动 commit 与显式 commit 的等价性**：SQLite Python 驱动 `with conn:` 在 normal exit 时调用 `commit`，异常时调用 `rollback`——这是 stdlib 行为，不引入新依赖
- **`api_cache` 唯一索引冲突**：批量插入时若同 key 出现两次，`insert or replace` 在事务内行为正常，但需测试覆盖
