## 目标
记录 P1.12 hotfix 的执行顺序、依赖关系和明确不进入本批的范围。

## 非目标
- ❌ 不直接修改运行时代码。
- ❌ 不执行任何 hotfix。
- ❌ 不把新集更新追踪纳入 P1.12。

## 允许修改范围
- `docs/tasks/p1.12_execution_order.md`
- `STATE.md`
- `SCOPE.md`

## 禁止修改范围
- 🚫 `src/`
- 🚫 `tests/`
- 🚫 DB schema 或 migration 文件
- 🚫 AGENTS.md、CLAUDE.md

## 相关上下文
- 新集更新追踪经用户确认放入 V2，不进入本批 hotfix。
- P1.12 仅处理 review 发现的 correctness / dry-run / security / dependency / reporting 缺陷。

## 输入
- 2026-05-14 review 结果
- 用户确认：新集更新追踪放入 V2

## 输出
- 5 个待执行 hotfix 任务包
- `STATE.md` 中的 P1.12 待执行清单
- `SCOPE.md` 中的新集追踪 V2 边界说明

## 具体要求
建议执行顺序：

```
P1.12-A（TMDb namespace 闭环修复）
    ↓
P1.12-B（daily-discover dry-run 不写业务结果）
    ↓
P1.12-C（OMDb key 日志脱敏）
    ↓
P1.12-D（PyYAML 依赖声明）
    ↓
P1.12-E（多新季 content_update 汇总）
```

说明：
- A 和 B 都触及 `discovery.py`，先修数据链接正确性，再修 dry-run 写入语义。
- C、D 相互独立，可并行执行；若单 Agent 执行，按上述顺序。
- E 不改 schema，避免把 CR-005 的唯一键设计扩成新架构。
- 新集更新追踪不建立 P1.12 任务包，记录为 V2 backlog。

## 验收标准
1. ✅ P1.12 任务包文件均存在。
2. ✅ `STATE.md` 能追溯到 P1.12 任务清单。
3. ✅ `SCOPE.md` 明确新集更新追踪为 V2。

## 测试要求

- 新集更新追踪虽已放入 V2，但 `SCOPE.md` 旧文本此前提到“新季/新集”，需同步修正。
