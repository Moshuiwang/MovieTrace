## 任务名称
P1.57l 验证 DB 与飞书投影可自动升级 / 重建

## 来源任务
用户补充要求：原始 DB 文件和飞书表格在本次升级后必须能安全重建，不影响现有数据；未来生产环境没有本次开发环境这么多已知信息，升级流程应由程序自动执行。

验证从旧 DB + secrets + 飞书表配置出发，程序可以自动完成 schema migration、current/observation 回填、导出和飞书字段/记录投影重建，且不破坏现有数据。

- ❌ 不删除历史 `content_updates`
- ❌ 不清空生产或开发飞书表
- ❌ 不依赖手工 record_id / 字段 ID / 标题清单
- ❌ 不创建 PR

## 输入
- 一份升级前 v18 DB 备份或由测试构造的 v18 DB
- `~/.config/movietrace/secrets.json`
- 开发飞书表 app/table 配置
- P1.57a-j 实现

## 输出
- 自动升级 / 重建验证脚本
- 验证日志
- 一份明确的升级步骤记录，可供未来生产使用

## 具体要求
1. 验证脚本必须先复制 DB 到临时路径或备份路径，不能直接破坏原始 DB。
2. 自动执行或明确串联：`migrate` → current discovery 回填 → `export-recommendations` → 飞书字段 ensure → `sync-feishu-table`。
3. 验证脚本不得写死开发环境已知行数、标题、record_id、field_id。
4. 验证旧 `content_updates` 行数升级前后不减少。
5. 验证 current/observation 可从旧 DB 自动重建。
6. 验证空飞书表或缺字段飞书表可通过程序自动创建字段并 upsert 记录；如不能安全清空开发表，使用 mock 或 dry-run 验证缺字段路径。
7. 任何失败必须输出“现象 → 已排除 → 下一步定位”，不能静默继续。

## 验收标准
1. ✅ 从 v18 DB 副本可自动迁移到 v19+ 并回填 current/observation。
2. ✅ 升级前后 `content_updates` 总行数不减少。
3. ✅ 不依赖本次开发环境已知数据，换一份符合 schema 的旧 DB 仍可运行。
4. ✅ 飞书字段 ensure 和记录 upsert 在开发表可自动执行或由 mock 覆盖。
5. ✅ 生成未来生产可复用的升级步骤记录。

## 测试要求
- 至少覆盖一个临时 v18 DB 的自动升级路径。
- 覆盖缺字段飞书表的 ensure 行为；真实 API 不适合清表时用 mock。

## 验证命令
```bash
source .venv/bin/activate
PYTHONPATH=src python -m pytest tests/test_p1_57_upgrade_rebuild.py tests/test_migrate_cli.py -v
./scripts/verify_p1_57_upgrade_rebuild.sh
git diff --check
```

## 风险点
- 自动重建验证如果直接操作真实开发表，可能制造重复投影；应优先使用测试表、mock 或 dry-run，真实开发表验证前必须确认表配置。
- 未来生产信息更少，脚本输出必须可审计，不能把本次人工判断藏在注释里。
