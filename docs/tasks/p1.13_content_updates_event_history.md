# 任务包：P1.13 — content_updates 事件历史化

**任务包版本：** v1
**创建时间：** 2026-05-14 15:45 +08

## 任务名称
P1.13：将 `content_updates` 从全局去重池改为事件历史表

## 任务类型
fix

## 当前阶段
Phase 1.13：content_updates 数据模型修正（排在 P1.12 review hotfix 之后）

## 来源任务
CR-005（`reports/code_review_2026-05-14.md`）+ ADR-0012 + 用户 2026-05-14 产品决策。

## 目标
允许同一内容同一 `update_type` 在不同日期重新写入 `content_updates`，避免重新变热内容被历史去重静默吞掉；同一天/同一业务事件仍保持幂等。

- ❌ 不新增 observation 明细表。

- ❌ 不做历史运营数据人工清洗。

## 相关上下文
- 当前 schema 有唯一索引 `ux_content_updates_item_type`：`(canonical_item_id, update_type)`。
- 当前 discovery 写入的 `content_update_id` 已包含 `snapshot_date`：`discovery:{tmdb_id}:{snapshot_date}`。
- 当前 baseline 新季写入的 `content_update_id` 已包含 season：`new_season:vs_{virtual_series_id}:s{season}`。
- `export-recommendations --days N` 和 `inspect-updates --days N` 已按 `created_at` 时间窗口查询。

## 输入
- ADR-0012 决策。
- 现有 `content_updates` schema 与写入逻辑。
- 现有测试基线：458 passed。

## 输出
- migration 013：移除 `(canonical_item_id, update_type)` 全局唯一语义，改为 `content_update_id` 事件级唯一。
- schema.py 升级到新 schema version。
- discovery / baseline 写入逻辑按 `content_update_id` 幂等。
- 回归测试覆盖跨天重复写入和同天重复去重。
- 文档同步说明 `content_updates` 是事件历史表。

## 具体要求
1. 新增 migration 013：
   - drop 旧唯一索引 `ux_content_updates_item_type`。
   - create unique index `ux_content_updates_update_id` on `content_updates(content_update_id)`。
   - 如现有生产数据存在重复 `content_update_id`，migration 必须先检测并失败报错，不静默删除。
2. 更新 `schema.py`：
   - `SCHEMA_VERSION` +1。
   - 初始 schema 改为创建 `content_update_id` 唯一索引，不再创建 `(canonical_item_id, update_type)` 唯一索引。
3. 更新 discovery 写入测试：
   - 同一 `tmdb_id` + 同一 `snapshot_date` 重跑只写 1 条。
   - 同一 `tmdb_id` + 不同 `snapshot_date` 可写多条。
4. 更新 baseline tracking 测试：
   - 同一 `new_season:vs_{id}:s{season}` 重跑只写 1 条。
   - 同一剧不同 season 可写多条。
5. 修正 `_write_content_updates()` 统计：
   - `written` 只能统计实际插入行数，不能 `insert or ignore` 后无条件 +1。
6. 导出/查询保持按 `created_at` 最近 N 天读取；如需要排序，优先按 `created_at desc, hot_score desc`。
7. 不实现折叠视图；如后续运营觉得重复太多，再新开任务包做导出层折叠。

## 验收标准
1. ✅ `content_updates` 不再有 `(canonical_item_id, update_type)` 唯一索引。
2. ✅ `content_update_id` 具备唯一约束。
3. ✅ 同内容同类型跨天重新命中会新增一条事件。
4. ✅ 同一天/同一事件重复运行保持幂等。
5. ✅ `export-recommendations --days 7` 能看到最近 7 天内重复出现的事件。
6. ✅ `new_episode` 仍不进入本任务。

## 风险点
- 旧生产库若存在重复 `content_update_id`，migration 必须先人工处理，不能静默丢数据。
- 最近 N 天导出可能出现同一内容多天重复，这是本决策接受的产品取舍。
- 如果 `discovery.py` 仍存在 TMDb namespace 问题，应先完成 P1.12-A，避免新历史事件写到错误 canonical item。
