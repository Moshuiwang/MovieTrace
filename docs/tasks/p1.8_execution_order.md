## 建议执行顺序

### 1. P1.8-D API usage logging

先做 API 用量日志。

原因：

- 后续 FP 扩围、TMDb external_ids、OMDb fallback 验证都会增加或改变 API 请求量。
- 没有统一 usage log，会继续无法准确回答“今天用了多少次 API”。
- 这是后续所有外部 API 策略的观测基础。

### 2. P1.8-H FlixPatrol 覆盖范围与 API 预算策略调整

第二步调整 FP 抓取范围。

原因：

- FROM 缺 FP 信号的根因是当前 FP 维度太窄。
- 新策略已控制在约 824-864 次/月，低于 1000 calls/month。
- 需要在 usage log 之后执行，确保扩围后的调用量可统计。

### 3. P1.8-C TMDb 字段结构化与 TV freshness

第三步修正 TV 新鲜度和结构化 TMDb 字段。

原因：

- 老剧新季被误判为老内容，会直接压低 freshness。
- `last_air_date` 等字段应先结构化，后续评分和报告才能解释清楚。

### 4. P1.8-F + P1.8-G 合并执行

第四步合并实现 TMDb external_ids 与评分兜底。

合并范围：

- P1.8-F：external_ids client、缓存/入库、统计、限流、可重跑。
- P1.8-G：评分前补 IMDb ID、OMDb enrichment 接入、TMDb fallback 评分。

原因：

- 两个任务都会修改 `tmdb.py`、`discovery.py`、`omdb_enrichment.py`、`scoring.py` 和相关测试。
- 分开做会重复实现 external_ids client 和缓存逻辑。
- 用户已确认降低 OMDb 依赖，并接受 TMDb fallback。

### 5. P1.8-E 多源字段结构化与 IMDb ID 补全

最后做更广的多源字段结构化。

原因：

- P1.8-E 范围最大，包含 FP / Trakt / OMDb / TMDb 多源字段。
- 应在关键评分链路稳定后推进，避免一次任务跨越过多目标。
- P1.8-F/G 完成后，P1.8-E 可复用 external_ids 基础设施。
