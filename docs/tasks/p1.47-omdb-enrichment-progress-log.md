## 任务类型

## 当前阶段

## 来源任务
用户观察：`daily-discover` 在 [6/8] 丰富化阶段**两个循环均静默**：OMDb enrichment 约 3 分钟（169 条 × 1s sleep）、TMDb detail enrichment 也无中间日志，合计可能超过 10 分钟无任何输出，难以判断进程是否存活。

## 目标

在两个高耗时循环里，**每处理 20 条候选**各打一行进度日志：
```
OMDb enrichment: 20/169 (api=5 cache=15 enriched=20)
TMDb detail enrichment: 20/169 (api=12 cache=8 enriched=20)
```
让运维和开发在终端里能看到实时进度，不再依赖 ps 判断进程是否存活。

## 具体要求

### 改法（两处，模式相同）

**`enrich_with_omdb()` 循环末尾**（在 `time.sleep(1.0)` 之后）：
```python
for idx, c in enumerate(candidates):
    ...（现有逻辑不动）...
    time.sleep(1.0)
    completed = idx + 1
    if completed % 20 == 0 or completed == len(candidates):
        logger.info(
            "OMDb enrichment: %d/%d (api=%d cache=%d enriched=%d)",
            completed, len(candidates), api_calls, cache_hits, enriched,
        )
```

**`enrich_with_tmdb_details()` 循环末尾**（在 `continue`/主体结束处，循环最后）：
```python
for idx, c in enumerate(candidates):
    ...（现有逻辑不动）...
    completed = idx + 1
    if completed % 20 == 0 or completed == len(candidates):
        logger.info(
            "TMDb detail enrichment: %d/%d (api=%d cache=%d enriched=%d)",
            completed, len(candidates), api_calls, cache_hits, enriched,
        )
```

**注意**：`completed == len(candidates)` 确保最后不足 20 条的尾批也能打出。两处汇总行（函数末尾的 `logger.info`）内容不同（含更多字段），不冲突。

### 不改的东西
- 任何 `continue` / `break` 逻辑
- `time.sleep` 位置
- 函数签名和返回值 dict

## 风险点
- `continue` 语句（no-imdb-id 跳过）会让 `idx` 递增但计数器不变——进度行仍每 20 次 `idx` 打一次，语义是"遍历了多少候选"而非"成功处理了多少"，可接受
- TMDb detail 循环有多处 `continue`（skip 条件），进度行在循环末尾，`continue` 会跳过——最后一批若全部 skip 可能漏打尾行；属于 edge case，不阻塞
