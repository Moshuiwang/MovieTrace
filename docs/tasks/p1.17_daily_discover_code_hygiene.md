2026-05-14 代码层面复盘：不建议做架构级重构，但可以在真实运行后围绕 `daily-discover` 做小范围、行为不变的可读性整理。

## 目标
在不改变业务行为的前提下，改善 `daily-discover` 主流程的可读性、阶段边界和回归可验证性。

## 非目标
- ❌ 不做架构级重构。
- ❌ 不新增 Writer 抽象、Repository 层、ORM 或新依赖。
- ❌ 不改变数据库 schema 或 migration。
- ❌ 不改变评分公式、阈值、fallback 规则、API 请求策略。
- ❌ 不处理 V2 功能，包括 episode-level 新集追踪、LLM、付费数据源。
- ❌ 不为了“优雅”改动无关模块。

## 允许修改范围
- `src/movietrace/pipeline/discovery.py`
- `tests/test_discovery.py`
- `tests/reports/test_export_writer.py`（仅当 discovery 输出结构整理导致必要回归调整）
- `tests/test_inspect_updates_cli.py`（仅当输出读取回归需要）
- `STATE.md`

- 🚫 `src/movietrace/db/schema.py`
- 🚫 `src/movietrace/db/migrations/`
- 🚫 `src/movietrace/sources/` 外部 API client 行为
- 🚫 `config.yaml`、`config/config.example.yaml`

- 🚫 AGENTS.md、CLAUDE.md

- `src/movietrace/pipeline/discovery.py`
- 真实运行 3-7 天后的问题记录或维护痛点

## 输出
- 行为不变、阶段函数更清晰的 discovery 主流程
- 必要的回归测试补强
- `STATE.md` 更新本次代码卫生结果

## 具体要求
1. 开工前先读 `discovery.py` 和相关测试，列出当前主流程阶段。
2. 只允许围绕现有阶段做拆分、命名和小函数提取，例如：
   - fetch 结果归一。
   - source fallback 日期解析。
   - merge 输入构造。
   - enrichment/scoring 调用边界。
   - write result 统计。
3. 不得改变公开 CLI 参数、返回字段语义、报告字段语义。
4. 不得改变真实 API 请求次数上限。
5. 不得扩大测试 fixture 的外部 API 依赖；自动化测试仍不得触发真实 API。
6. 至少补强或确认以下回归：
   - dry-run 不写业务结果。
   - `written` 只统计实际插入。
   - source fallback 状态仍进入 `source_summary_json`。
7. 每一步改动都应能用现有测试解释其必要性。

## 验收标准
1. ✅ `daily-discover` 行为保持不变。
2. ✅ `discovery.py` 主流程阶段边界更清晰。
3. ✅ dry-run、written count、source fallback 报告回归通过。
4. ✅ 全量测试通过。
5. ✅ 未引入新依赖、schema 变更或 V2 功能。
