## 来源任务
OMDb 免费 key 当日触发 `Request limit reached!`，用户需要统计程序实际使用各类 API key 的请求次数，避免再次无法解释额度消耗。

- ❌ 不更换或提交任何 API key。
- ❌ 不记录完整 URL 查询串中的密钥。
- ❌ 不记录完整 Authorization header。
- ❌ 不改动 FlixPatrol / TMDb / Trakt / OMDb 的业务解析规则，除非是为了接入日志。

## 具体要求

### 1. 新增表

新增 `api_usage_log` 表，建议字段：

- `id integer primary key autoincrement`
- `service text not null`：`tmdb` / `trakt` / `omdb` / `flixpatrol`
- `endpoint text not null`：例如 `/movie/{id}`、`/tv/popular`、`/top10s`、`/?i`
- `operation text`：例如 `tmdb_trending.fetch_tv_popular`、`omdb.detail_by_imdb_id`
- `request_date text not null`：本地日期，`YYYY-MM-DD`
- `started_at text not null default current_timestamp`
- `finished_at text`
- `status text not null`：`success` / `http_error` / `network_error` / `parse_error` / `skipped`
- `http_status integer`
- `cache_status text`：`hit` / `miss` / `bypass` / `not_applicable`
- `quota_error integer not null default 0`
- `rate_limited integer not null default 0`
- `duration_ms integer`
- `item_count integer`
- `error_code text`
- `error_message text`
- `key_fingerprint text`
- `metadata_json text`

### 2. 密钥安全

- `key_fingerprint` 只能保存不可逆摘要或脱敏标识，例如 SHA-256 前 12 位。
- 禁止保存完整 key、token、Authorization header、`apikey` 参数值。
- `metadata_json` 里禁止出现密钥、完整 URL query string 或请求 header。
- 测试必须覆盖“不泄露 key”。

### 3. 记录粒度

- 每次实际发起外部 HTTP 请求记录 1 条。
- 读取本地 `api_cache` 命中时不算外部请求，但可以记录 `cache_status='hit'` 的 `skipped` 日志；执行阶段可选择是否开启，默认建议记录，以便统计“省了多少请求”。
- 同一 API 方法内部重试时，每次 HTTP 请求都应记录，`metadata_json` 中可记录 `attempt`。
- 批量 endpoint 一次 HTTP 请求算 1 次，`item_count` 记录返回条目数。

### 4. 服务覆盖

必须覆盖：

- OMDb search/detail
- TMDb search/detail/trending/popular
- Trakt search/trending
- FlixPatrol top10

### 5. 错误识别

- OMDb 返回 `Request limit reached!` 时，必须记录：
  - `service='omdb'`
  - `status='http_error'` 或可判断的错误状态
  - `http_status=401`
  - `quota_error=1`
  - `error_message` 包含脱敏后的响应错误摘要，不含 key
- Trakt / TMDb / FlixPatrol 的 429 应记录 `rate_limited=1`。

### 6. 查询 CLI

新增 CLI 命令 `inspect-api-usage`，至少支持：

```bash
PYTHONPATH=src python -m movietrace.cli inspect-api-usage --date 2026-05-13
PYTHONPATH=src python -m movietrace.cli inspect-api-usage --days 7
PYTHONPATH=src python -m movietrace.cli inspect-api-usage --service omdb
PYTHONPATH=src python -m movietrace.cli inspect-api-usage --format table
PYTHONPATH=src python -m movietrace.cli inspect-api-usage --format json
```

输出至少包含：

- 总请求数
- 按服务统计请求数
- 成功 / 失败 / quota_error / rate_limited 数
- 按 endpoint 统计请求数
- cache hit / miss / bypass 统计

## 风险点
- API 用量日志本身会写入数据库，需保证不会因日志失败阻断主流程；主流程最多 warning。
- 如果在底层 `get_json()` 统一记录，需要能传入 service / endpoint / operation，否则日志语义会太粗。
- 若在各客户端分别记录，重复代码会增多；可通过小 helper 控制重复。
- 日志不能泄露密钥，这是最高优先级验收点。
- 过去已经发生的 API 调用无法完整回补，只能从功能上线后开始精确统计。
