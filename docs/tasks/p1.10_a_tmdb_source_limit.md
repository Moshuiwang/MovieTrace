## 当前阶段

## 来源任务
来自 2026-05-14 用户讨论：TMDb / Trakt 作为补充热源，不应大口径拉取长尾数据。按二八原则，将每个抓取接口限定为 20 条，降低 API 消耗、候选噪音和后续富化成本。

## 目标
将 TMDb 每日热源采集限制为每个 endpoint 只抓 1 页，即每个接口约 20 条。

用户已确认：每接口默认 20 条，但保留 config / CLI 覆盖能力。

## 非目标
- ❌ 不修改 Trakt 抓取 limit。
- ❌ 不调整 hot_score 权重或 P0/P1/P2/P3 阈值。
- ❌ 不改变 TMDb popularity / trending 字段语义。
- ❌ 不补拉历史 TMDb 数据。
- ❌ 不处理抓取失败兜底；兜底由 P1.10-C/D/E 处理。

## 相关上下文
当前 TMDb 热源采集：

- `/trending/all/day`
- `/tv/popular`
- `/movie/popular`
- 默认 `pages_per_endpoint=3`

每页约 20 条，因此当前约 `3 endpoints × 3 pages × 20 = 180` 条/天。P1.10-A 后应降为约 `3 endpoints × 1 page × 20 = 60` 条/天。

## 输入
- TMDb Bearer Token（运行时读取，禁止输出完整 token）
- 当前运行日期
- `config.yaml` 中的 source limit 配置

## 输出
- TMDb 每日热源抓取默认降为每 endpoint 20 条。
- `daily-discover` 使用配置中的 TMDb limit。
- 独立 CLI `fetch-tmdb-trending` 仍可通过参数覆盖页数，便于人工验证。

## 具体要求

### 1. 配置项

建议新增配置：

```yaml
source_fetch_limits:
  tmdb:
    pages_per_endpoint: 1
```

要求：

- `config.yaml` 和 `config/config.example.yaml` 同步。
- 默认值为 1。
- 配置缺失时也必须回落到 1，而不是旧的 3。

### 2. daily-discover 行为

`daily-discover` 调用 `fetch_and_store_tmdb_trending()` 时必须传入配置值。

要求：

- 默认只抓 1 页。
- 控制台输出或 stats 中能看出 TMDb 抓取页数。
- 不改变后续 merge / scoring 逻辑。

### 3. 独立 CLI 行为

`fetch-tmdb-trending` 继续支持 `--pages`。

要求：

- 不传 `--pages` 时默认读取配置。
- 传 `--pages` 时以 CLI 参数为准。
- `--pages` 必须为正整数。

### 4. API usage 预期

执行后 TMDb trending/popular 请求数应从默认 9 次/天降到 3 次/天。

## 风险点
- 如果 TMDb 前 20 条不足以覆盖部分中腰部内容，可能降低召回。
- 由于 TMDb 是补充热源，建议先接受召回下降，用 FP/Trakt/兜底机制弥补。
- 默认 20 条是产品决策，不应在实现中悄悄回退到旧的 3 页；如需临时扩大，只能通过显式 config / CLI 覆盖。
