## 任务名称
P1.11-A API 致命错误熔断机制

## 任务类型
feat

## 当前阶段
Phase 1.11：API 调用韧性增强

## 来源任务
来自 2026-05-14 P1.10 dry-run 验收发现：FP API 402 Payment Required 时仍然发出全部 24 次请求（4 国 × 6 平台），每次 ~0.8s，纯浪费。OMDb 401 Unauthorized 时对 584 个 candidate 逐个请求，耗时 ~10 分钟。

用户决策：**第一次遇到确定会重复的错误（401/402/403）就立即熔断该 source 的所有剩余请求。**

## 目标
在 `http.py` 和/或 API client 层实现致命错误熔断：当某个 source 的第一次请求返回 401/402/403 时，立即停止该 source 的所有后续请求，直接进入 fallback。

## 非目标
- ❌ 不回退重试（retry with backoff）——那是另一个任务包。
- ❌ 不修改 merge/scoring/report 逻辑。
- ❌ 不引入外部熔断库（circuitbreaker、tenacity 等）。
- ❌ 不实现跨日熔断（今天熔断不影响明天尝试）。
- ❌ 不改变 `run_discovery()` 或 `multi_source_merge.py`。

## 输入
- `http.py::get_json()` — 所有外部 API 调用的统一入口
- `FlixPatrolClient.fetch_all_platforms()` — 循环遍历国家 + 平台
- `omdb_enrichment.py::enrich_with_omdb()` — 循环遍历 candidates

## 输出
- 致命错误异常类（如 `FatalApiError`）
- `get_json()` 在 401/402/403 时抛出该异常
- FP client 捕获后停止遍历
- OMDb enrichment 捕获后停止遍历
- 熔断信息记录到 `api_usage_log`

## 具体要求

### 1. 致命错误分类

| HTTP 状态 | 是否熔断 | 理由 |
|-----------|---------|------|
| 401 Unauthorized | ✅ 立即熔断 | key 无效/过期 |
| 402 Payment Required | ✅ 立即熔断 | 订阅到期/配额耗尽 |
| 403 Forbidden | ✅ 立即熔断 | 权限不足 |
| 429 Too Many Requests | ❌ 不熔断 | 临时限流，可退避 |
| 5xx Server Error | ❌ 不熔断 | 可能临时故障 |

### 2. http.py 改动

`get_json()` 收到 401/402/403 时，抛出自定义异常 `FatalApiError(status_code, message)`。异常信息包含 `service`、`endpoint`、`status_code`。

### 3. FlixPatrol 改动

`FlixPatrolClient.fetch_all_platforms()` 的循环中捕获 `FatalApiError`：
- 记录到 `api_usage_log`
- 广播到所有剩余 plan item（标记为 skipped）
- 返回已有结果（0 条或部分成功的结果）

这样 `_ensure_fp_data()` 收到 0 条结果 → P1.10-D fallback 自动接管。

### 4. OMDb 改动

`enrich_with_omdb()` 的循环中捕获 `FatalApiError`：
- 停止后续 lookup
- 返回已完成的 stats

要求：第一次 OMDb 401 就立即停止，不再对剩余 583 个 candidate 发请求。

### 5. 日志记录

熔断发生时：
- `api_usage_log` 记录一条 `status='circuit_breaker'` 的条目（或使用 `error_message` 字段）
- logger.warning 输出清晰提示（含 source、HTTP 状态、原因）

## 验收标准
1. ✅ FP 第一次 402 后，不再对其他 23 个 country/platform 组合发请求。
2. ✅ OMDb 第一次 401 后，不再对剩余 candidates 发请求。
3. ✅ TMDb / Trakt 不受影响（401/402/403 也会熔断，但不改变本次行为）。
4. ✅ `daily-discover` 熔断后继续 merge/scoring（不因熔断崩溃）。
5. ✅ 相关测试通过。

## 测试要求
- `get_json()` 在 401/402/403 时抛出 `FatalApiError`。
- `get_json()` 在 429/500 时**不**抛出 `FatalApiError`（走原有异常路径）。
- FP mock：第一次 402 后剩余 countries 被跳过。
- OMDb mock：第一次 401 后剩余 candidates 被跳过。
- `daily-discover` mock：FP 熔断 + TMDb fresh + Trakt fresh → 仍能 merge 并返回结果。

## 风险点
- 熔断后当天该 source 的数据量为 0，完全依赖 P1.10-D fallback。如果连续 30 天 API 不可用，fallback 也会失败。
- 401 可能是临时 token 过期（如刷新后恢复），但当前 OMDb key 是持续失效状态，熔断是正确的。
- 如果 API 返回 401 但 header 中有 `Retry-After`，本任务包不做特殊处理——视为致命错误熔断。
