## 来源任务

用户已确认：fallback 数据允许进入供应商主列表，但报告和导出必须显著标记。

- ❌ 不新增供应商业务字段，除非是为了表达 source freshness。

## 相关上下文
P1.10-D 会在 stats / candidate dict / source_summary_json 中提供 source 状态。P1.10-E 只负责让人类看见这些状态。

建议展示文案：

- `TMDb: fresh(2026-05-14)`
- `Trakt: fallback(2026-05-13)`
- `FlixPatrol: failed_no_fallback`

## 输入
- P1.10-D 输出的 source status / effective dates。
- `content_updates.source_summary_json`。
- `source_fetch_runs` 表。

## 输出
- CLI 控制台展示 source 状态。
- `inspect-updates` table / detail / json 展示 source 状态。
- `export-recommendations` Markdown 和 JSON 展示 source 状态。
- 如 daily report 仍使用，也同步展示。

## 具体要求

### 1. daily-discover 控制台

运行结束时输出数据源状态摘要，例如：

```text
Source data:
  FlixPatrol: fallback from 2026-05-13
  TMDb: fresh 2026-05-14
  Trakt: fresh 2026-05-14
```

有 fallback 或 failed 时必须明显提示。

### 2. content update source summary

`source_summary_json` 建议包含：

```json
{
  "source_data_status": {
    "tmdb": {"status": "fresh", "snapshot_date": "2026-05-14"},
    "trakt": {"status": "fallback", "snapshot_date": "2026-05-13"},
    "flixpatrol": {"status": "failed_no_fallback", "snapshot_date": null}
  }
}
```

要求：

- 不覆盖已有 `fp` / `tmdb` / `trakt` 热度证据。
- JSON 输出必须保留结构化状态。

### 3. Markdown 报告

`export-recommendations` Markdown 至少在报告头部展示总体 source 状态。

如果单条内容使用 fallback source，也应在详情或表格中体现简短标记，例如：

```text
数据源：TMDb fresh / Trakt fallback(2026-05-13)
```

### 4. inspect CLI

`inspect-updates --format json` 必须输出 source status。

`inspect-updates --id <id>` detail 必须展示 fallback 状态。

## 验收标准
1. ✅ 有 fallback 时，CLI 和报告均能看见。
2. ✅ JSON 导出包含结构化 `source_data_status`。
3. ✅ Markdown 导出包含人类可读的 source 状态摘要。
4. ✅ fresh 数据不会被错误标记为 fallback。
5. ✅ 旧 content_updates 缺少状态字段时报告不崩溃。
6. ✅ 相关测试通过。

## 风险点
- 报告过度展示技术细节会干扰供应商阅读，建议在供应商主表中用简短标记，详细状态放 JSON。
- fallback 数据允许进入供应商主列表；如果标记不明显，会让人误以为所有来源都是当天 fresh 数据。
- 旧数据没有 source status，必须兼容。
- 如果 P1.10-D 未完成，本任务不应单独执行。
