### 1. 单行时长公式(已用户决策:缺失集累加)

```python
def compute_row_duration_hours(candidate, conn) -> float:
    """单行时长(h) = 缺失集累加 × 1h;Movie 固定 2h。"""
    if candidate.media_type == "movie":
        return 2.0
    if not candidate.tmdb_id:
        return 0.0  # 无法定位 → 留空(写 0,飞书 UI 展示自然为空)

    tmdb_data = candidate.tmdb_data or {}
    last_aired_season = (tmdb_data.get("last_episode_to_air") or {}).get("season_number")
    if not last_aired_season:
        return 0.0

    a_lib_max = _query_a_lib_max_season(conn, candidate.tmdb_id)  # 0 = A 库无该剧
    if a_lib_max == 0:
        # 全季缺失 → TMDb 整剧已播总集数
        return float(_sum_aired_episodes(tmdb_data, last_aired_season))

    total = 0
    # 1) a_lib_max+1 ... last_aired_season 的整季集数累加
    for s in range(a_lib_max + 1, last_aired_season + 1):
        season_detail = tmdb_client.get_tv_season_details(candidate.tmdb_id, s)  # 走 24h cache
        total += int(season_detail.get("episodes", [])  # episodes 数组长度
                     and len(season_detail["episodes"])
                     or season_detail.get("episode_count", 0))
    # 2) a_lib_max 季内可能的缺集(A 库未补齐):
    a_lib_eps_in_max = _query_a_lib_episode_count(conn, candidate.tmdb_id, a_lib_max)
    tmdb_eps_in_max = int(tmdb_client.get_tv_season_details(candidate.tmdb_id, a_lib_max)
                          .get("episode_count", 0))
    if a_lib_eps_in_max < tmdb_eps_in_max:
        total += (tmdb_eps_in_max - a_lib_eps_in_max)

    return float(total)
```

**实现注意**:
- `get_tv_season_details` 必须走 24h `api_cache` (cache_key=`tmdb:detail:{tv_id}:season:{n}`,沿用 `tmdb_detail_cache.py` 的缓存模式)
- TMDb season detail 返回里 `episode_count` 字段权威可信;`episodes` 数组也可数
- S0(specials)**不计入**:`range(a_lib_max+1, last_aired+1)` 自然只覆盖 ≥1 的正剧季
- 失败兜底:任一 API 调用失败,row_duration 写 0,飞书展示空,日志 warning(不重试不静默)

## 风险点

- **R1 飞书字段创建 scope 不足**:首次跑可能 99991661/99991663(无 `bitable:app:fields`)。处置:`schema_setup` 检测到该错误码直接 raise + 提示在飞书后台开权限,不静默重试(`.claude/rules/23-feishu-integration.md`)
- **R2 Soap genres 来源缺失**:`tmdb_data` 可能不含 `genres` 字段(detail 没拿到 / 缓存 miss)。处置:`tmdb_detail_cache` 已确保 detail 必读;genres 缺失视为非 Soap,日志 warning
- **R3 "季号" rename 的历史值不一致**:旧值 = number_of_seasons(总,含未播),新值 = last_episode_to_air.season_number(已播)。**用户决策(2026-05-17):主动跑一次回填脚本** → 新增子任务 p1.24-G,扫 content_updates 历史行 + 飞书表所有"在播最新季"列,从 `api_cache:tmdb:detail:%:tv` 读 `last_episode_to_air.season_number`,SQL 内回填 source_summary,sync 重写飞书。不调用 TMDb API(零外部成本)
- **R4 TMDb season detail 额外配额**:每天 ~10 条 new_season + ~10 条 new_discovery TV,平均跨 1-3 季 → ~30-60 次/日新增调用。TMDb 配额宽松,可接受;24h cache 起效后日均 < 10 次。需在 `api_usage_log` 监控
- **R5 IMDb 链接空率**:`imdb_backfill` 不保证 100% 成功;TV 部分剧 IMDb ID 缺。处置:PRD 已写"无 IMDb ID 留空",不强制阻塞
- **R6 Soap 强制通过阈值带来的飞书表体积**:预估 +5-10%(Soap 占比小)。可接受,无需做特殊分页
- **R7 `compute_row_duration_hours` 在 sync 时计算还是 discovery 时计算**:决策为**discovery 时计算并写入 `source_summary`**,sync 只是读取。优点:不重复调 TMDb;缺点:历史 content_updates 行没数据,sync 显示空(可接受,只前向生效)
- **R8 `setup-feishu-fields` 子命令在 daily_run.sh 调用频率**:建议 `daily_run.sh` 每次 sync 前 ensure 一次(幂等成本低 — 1 次 GET fields);否则首次部署后人工跑一次,后续不需要
