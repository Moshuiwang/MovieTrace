## 任务名称
P1.32 新增 GitHub Actions `workflow_dispatch` 入口，允许在不 SSH 生产的前提下，手动触发指定阶段（discover / export / sync）重跑

约束：[[project_ci_cd_pipeline]] 禁止手工 SSH 生产。

## 目标
提供一个 GitHub Actions workflow_dispatch 入口，从 UI 选阶段 + 参数即可在生产 VPS 上重跑 pipeline，结果回传飞书。

## 非目标
- ❌ 不动 pipeline 代码本身（discover / export / sync 逻辑）
- ❌ 不重写 CLI
- ❌ 不替代 cron 日常调度（cron 继续每日 08:00 自动跑）
- ❌ 不暴露给非 owner（依赖 GitHub 仓库权限即可，不另设鉴权）
- ❌ 不把 manual workflow 改造成"任意 shell"（只暴露白名单参数，安全边界守住）

## 允许修改范围
- `.github/workflows/manual-pipeline.yml`（新建）
- `STATE.md`（同步 P1.32 状态）
- 本任务包文件

## 禁止修改范围
- 🚫 `.github/workflows/ci.yml`（P1.31 才动它）
- 🚫 `src/**`、`scripts/**`
- 🚫 `data/**`
- 🚫 AGENTS.md、CLAUDE.md
- 🚫 secrets 相关

## 相关上下文
- P1.31 任务包 `docs/tasks/p1.31-db-migrate-on-deploy.md`（前置）

- 现有 notify 模式：`.github/workflows/ci.yml:147-202`（飞书 IM 推送）

## 输入
workflow_dispatch UI inputs：
- `stage`: choice [`export`, `discover`, `sync`, `all`]（default `export`）
- `days`: string（default `1`，仅 `export` / `discover` 使用，对应 `--days N`）
- `dry_run`: boolean（default `false`）

## 输出
1. `.github/workflows/manual-pipeline.yml` 文件
2. workflow_dispatch UI 可见，参数符合上面定义
3. 跑完后飞书 IM 收到一条带"手动重跑 / 阶段 / 退出码 / WF run URL"的通知

## 具体要求

### A. Workflow 结构
```yaml
name: Manual Pipeline Rerun
on:
  workflow_dispatch:
    inputs:
      stage:
        type: choice
        options: [export, discover, sync, all]
        default: export
      days:
        type: string
        default: "1"
      dry_run:
        type: boolean
        default: false

jobs:
  rerun:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - 复用 ci.yml 的 Setup SSH 步骤（写 deploy_key + ssh-keyscan）
      - SSH 到 VPS，按 stage 跑对应 CLI 子命令
      - 失败时步骤标红
  notify:
    needs: rerun
    if: always()
    steps:
      - 复用 ci.yml notify 的飞书推送代码片段；
      - 消息：✅/❌ MovieTrace 手动重跑（{stage}, days={days}, dry_run={dry_run}）/ WF URL
```

### B. CLI 调用映射（在 SSH heredoc 内）
| stage | 命令 |
|---|---|
| `export` | `PYTHONPATH=src python -m movietrace.cli export-recommendations --days $DAYS` |
| `discover` | `PYTHONPATH=src python -m movietrace.cli daily-discover --days $DAYS` + `--dry-run` 若 `dry_run=true` |
| `sync` | `PYTHONPATH=src python -m movietrace.cli sync`（具体子命令名需先 `grep` 确认；不确定就 fallback 跑 cron 同款主脚本）|
| `all` | 上面三个串联 `set -e` |

⚠️ **写代码前必须验证 CLI 子命令名是否真实存在**：`PYTHONPATH=src python -m movietrace.cli --help`。如果某个 stage 找不到对应命令，要么删掉该选项、要么改用现有脚本入口；**不要猜命令名**（参考 [[feedback_verify_api_response_before_coding]]）。

### C. 鉴权
- 不引第三方鉴权
- 依赖仓库的 "Actions permissions" 默认配置（仓库 collaborator 才能触发 workflow_dispatch）
- secrets 全部复用 `VPS_SSH_KEY` / `VPS_HOST` / `VPS_USER` / `FEISHU_*`

### D. 安全边界
- `days` input 必须做正则校验（只允许 1-3 位数字），防 SSH command injection
- 拒绝把 input 直接拼进 SSH 命令；用 `env:` 块传入 + 在远端 shell `${DAYS:?}` 引用
- `stage` 走 GitHub choice 类型，天然受白名单约束

## 验收标准
1. ✅ 在 GitHub Actions UI "Manual Pipeline Rerun" workflow 看到 `Run workflow` 按钮、4 个 input
2. ✅ 选 `stage=export, days=1, dry_run=false` 触发后，能 SSH 到 VPS 并执行成功（生产库 export 文件刷新；或日志显示成功）
3. ✅ 触发完成后飞书收到 IM 通知，含 stage / 结果 / WF run URL
4. ✅ 故意输入 `days=abc` 时 workflow 步骤校验失败（不会传到 SSH）
5. ✅ STATE.md 已同步 P1.32 状态 + 今天事故已闭环说明
6. ✅ 本次事故的 2026-05-19 export 已通过该 workflow 补跑成功（产物可见）

## 测试要求
- workflow 文件本身无单测；走 smoke test：dispatch 一次 dry_run=true 的 discover 验证 SSH 路径通
- 然后 dispatch 一次 stage=export days=1 dry_run=false，对应今天的数据补救
- 失败场景：临时把 SSH key 改错一位字符做 dispatch（在 throwaway 分支上测试，**绝对不要碰生产 secrets**），观察 notify 步骤是否标红并推送

## 风险点
- **必须先完成 P1.31 再合本任务**：否则补跑 export 仍会撞 `no such column: title_zh`
- **SSH heredoc 安全**：用户输入拼接到远端 shell 是常见命令注入面；本任务严格用 `env:` + 远端 `${VAR:?}` 引用，input 走 GitHub `choice`/regex 校验
- **写代码前 `git fetch && git diff origin/main`**：确保基于最新 ci.yml 结构（[[project_ci_cd_pipeline]]）
- **复用 ci.yml notify 代码片段**：避免重复维护，可以抽取成 reusable workflow（YAGNI 边界：本任务不抽，下一次再有 workflow 需要它再说）
- **不要顺手扩范围加 "log tail" / "数据库快照" 等 stage**：保持目标单一（CLAUDE.md 第 4 条铁律）
