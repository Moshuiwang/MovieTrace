## 相关上下文
- 生产 `scripts/daily_run.sh` 当前也是 08:00 +08；本任务接受同时间窗口运行，因为生产和开发 DB / 飞书表隔离。
- 开发 shadow cron 使用真实 API，因此 TMDb / Trakt / OMDb / FlixPatrol 用量约为生产 + 开发双倍。
- FlixPatrol API 仍可能 402，系统应走 fallback。
- 同 08:00 跑生产 + 开发会让所有外部 API 的瞬时和日累计调用翻倍；安装 cron 前必须预检配额，避免 shadow 把生产日内额度吃光导致生产挂。

## 具体要求
0. **配额预检（安装 cron 之前必做）**：
   - 跑 `inspect-api-usage --days 7` 取生产侧 OMDb / TMDb / Trakt / FlixPatrol 日累计调用量，估算 ×2 是否超过各家配额（OMDb 每日 1000 / 私钥；TMDb 40 req/10s；Trakt 1000 calls/5min；FlixPatrol 已 402 走 fallback，不计）。
   - 任一项 ×2 后接近或超过限额，必须先调整 shadow 节流（增 sleep / 改时间窗）或拆分 source 跑，再装 cron；不许"先装上看效果"。
   - 预检结果写入任务汇报。
1. shadow 脚本流程对齐生产：`daily-discover` → `export-recommendations` → `sync-feishu-table` → `sync-feishu-doc` → `notify-feishu`。
2. 运行时间固定为每天 08:00 +08；如预检发现配额风险，可改为 09:30 错峰或限流，需在汇报中说明并更新本任务包。
3. 日志使用开发专属文件名，例如 `reports/logs/dev_shadow_daily_YYYYMMDD.log`，避免与生产日志混淆。
4. stats 使用开发专属文件名，例如 `dev_shadow_discover_stats_YYYYMMDD.json` / `dev_shadow_sync_stats_YYYYMMDD.json`。
5. 脚本启动时输出当前 git branch 和 commit hash，方便一周后追溯。
6. 每天运行后检查：退出码、current item 数、observation 数、Feishu created/updated/errors、API 用量、配额剩余。
7. 任一天失败不得静默重试；先记录现象、已排除、下一步定位。
8. shadow 脚本必须能用一个总开关停掉（例如检测 `~/.config/movietrace/shadow_disabled.flag`），便于发现 API 配额异常时立即止血。

## 验收标准
1. ✅ crontab 中存在开发环境 08:00 +08 shadow cron，且不会调用生产目录。
2. ✅ 连续 7 天至少 6 天成功；若失败，失败原因有明确记录。
3. ✅ 每天 observation 数随有效 fresh 信号增长，pure source-date fallback 不增长。
4. ✅ 开发飞书表中同一 discovery 稳定键跨天更新同一行。
5. ✅ 一周后输出观察结论：是否建议 PR、剩余风险、是否需要补任务。

## 风险点
- 08:00 同时跑生产和开发会带来 API 用量翻倍；用户已明确接受，但**必须先做配额预检**才能装 cron（详见具体要求 §0）。
- 开发和生产 DB / 飞书表必须保持隔离；如发现 secrets 指向同一飞书表，立即停止 shadow cron。
- 一周观察期内本分支不创建 PR；任何 PR 行为都需要用户重新明确确认。
- shadow 中途如发现 API 配额逼近上限或生产侧报 429/402，立即用总开关停 shadow，不要在 shadow 端继续加退避猜测。
