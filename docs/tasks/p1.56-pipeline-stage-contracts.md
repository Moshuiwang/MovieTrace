## 相关上下文
当前已知问题不是单点 bug，而是阶段契约不清：
- `fallback` 一词同时表达 source 日期回退、merge ID fallback、rating fallback。
- source 抓取结果、effective date、status、stats 分散在多个 dict 中。
- `MergedCandidate` 只携带 `source_flags`，下游再用 `source_dates` 推断 freshness。
- TMDb detail skip 同时受 canonical_items、api_cache、候选内存字段影响，判断口径容易漂移。
- `content_updates` 写入前需要硬 invariant，防止历史 source snapshot 被误当作今日新事件。

现状地图：
- source 日期回退：`discovery.py::_resolve_source_dates_with_fallback`
- merge ID fallback：`multi_source_merge.py::merge_three_sources`
- rating fallback：`scoring.py`
- fresh signal / pure fallback suppress：`discovery.py::run_discovery`
- TMDb detail skip：`omdb_enrichment.py::enrich_with_tmdb_details`

## 具体要求
1. 拆分 fallback 命名：至少区分 `source_date_fallback`、`id_match_fallback`、`rating_fallback` 三类语义；不要求一次性改完所有日志文案，但核心代码路径不能继续用裸 `fallback` 表达多义状态。
2. 引入 `SourceSnapshotDecision` 或等价结构，统一表达 `source`、`target_date`、`effective_date`、`status`、`rows_fetched_today`、`rows_used`、`is_fresh`、`error` 等事实。
3. 下游 merge / stats / notify 所需事实应从 source decision 派生，避免 `source_dates` 与 `source_status` 各维护一半事实。
4. `MergedCandidate.source_flags` 可以保留，但候选必须能明确表达每个来源的 contribution 和 freshness；`has_fresh_signal` 不应再由分散 dict 临时交叉推断。
5. 集中 TMDb detail skip 判断，例如 `has_required_tmdb_detail(candidate, purpose=...)`；canonical_items 只能预填中文、类型、平台等字段，不能单独触发 detail API skip。
6. `canonical_hits` 口径必须保持清楚：只统计 canonical 命中且真的跳过 TMDb detail API 的次数。
7. 写 `content_updates` 前增加最终 invariant：非豁免候选必须有 fresh signal；`source_date_fallback` 只能作为上下文或评分补充，不能单独制造 `new_discovery`。
8. 如果保留 Soap 豁免，必须显式表达是否允许 pure fallback Soap 写入，不能隐含在 fallback 逻辑里。
9. 保持现有 CLI 行为和输出结构尽量兼容；如 stats 字段口径变化，必须在测试和任务汇报中说明。

## 验收标准
1. ✅ source decision 能区分 fresh / error / 0 rows / source_date_fallback / failed_no_fallback，且 `effective_date`、`rows_used`、`is_fresh` 一致。
2. ✅ merge 后候选能追溯每个 source 的 contribution 和 freshness。
3. ✅ pure source_date_fallback 候选不会写入 `content_updates`。
4. ✅ fresh + source_date_fallback 组合候选可以通过 write gate，历史 source 只作为补充上下文。
5. ✅ canonical 有 genres 但缺必要 detail 字段时仍调用 TMDb detail 路径。
6. ✅ canonical 仅预填但仍调用 detail API 时，不计入 `canonical_hits`。
7. ✅ `content_updates` 写入前 invariant 有测试覆盖。
8. ✅ 不新增 migration，不引入新依赖。
9. ✅ 全量测试和 `daily-discover --dry-run` 通过。

## 测试要求
新增或补齐矩阵测试：
- source decision：fresh / error / 0 rows / source_date_fallback / failed_no_fallback 的 status、effective_date、rows_used、is_fresh。
- merge contribution：不同 source effective date 下，candidate contribution 和 freshness 正确。
- write gate：pure source_date_fallback 被 suppress；fresh + source_date_fallback 正常通过。
- TMDb detail skip：canonical 有 genres 但缺 release_date / original_language / TV last_air_date 时仍调用 detail API。
- stats / notify 口径：source_date_fallback 的 cached_count / rows_used 展示与实际读取一致。

## 风险点
- 本任务容易扩大成 orchestrator 重构；必须限制在阶段契约、集中判断、轻量数据结构和测试固化。
- source stats / notify 口径若调整，可能影响运营侧理解；需要保持字段兼容或在汇报中明确变化。
- Soap 豁免规则当前有业务语义，是否允许 pure fallback Soap 写入需要显式决策；默认按更窄解释处理。
- 任务跨 `discovery.py`、`multi_source_merge.py`、`omdb_enrichment.py`，验证范围必须覆盖全量测试和 dry-run。
