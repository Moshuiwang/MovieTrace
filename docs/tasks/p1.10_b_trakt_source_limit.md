## 来源任务
来自 2026-05-14 Trakt 热源语义分析：Trakt `trending` 是社区近 24 小时观看热度，不是平台榜单。当前 `shows/trending limit=500` 造成大量 Trakt-only P3 长尾候选，需要按二八原则收窄为每接口 20 条。

## 目标
将 Trakt 每日热源采集限制为 `shows/trending` 20 条、`movies/trending` 20 条。

用户已确认：每接口默认 20 条，但保留 config / CLI 覆盖能力。

## 非目标
- ❌ 不修改 TMDb 抓取页数；TMDb 由 P1.10-A 处理。
- ❌ 不修改 Trakt `watchers` 评分公式或权重。
- ❌ 不改变 Trakt 数据字段语义。
- ❌ 不改为 `/popular`、`/watched`、`/hot` 等其他 Trakt endpoint。
- ❌ 不处理抓取失败兜底；兜底由 P1.10-C/D/E 处理。

## 相关上下文
当前 Trakt 热源采集：

- `/shows/trending?limit=500&extended=full`
- `/movies/trending?limit=500&extended=full`

本地数据中 Trakt 每日约 625 条，其中大量 P3 来自 Trakt-only TV shows。P1.10-B 后理论条数应为约 40 条/天。

## 输入
- Trakt client_id（运行时读取，禁止输出完整 key）
- 当前运行日期
- `config.yaml` 中的 source limit 配置

## 输出
- Trakt show/movie trending 默认各 20 条。
- `daily-discover` 使用配置中的 Trakt limit。
- 独立 CLI `fetch-trakt-trending` 可通过参数覆盖 limit，便于人工验证。

## 具体要求

### 1. 配置项

建议新增配置：

```yaml
source_fetch_limits:
  trakt:
    shows_limit: 20
    movies_limit: 20
```

要求：

- `config.yaml` 和 `config/config.example.yaml` 同步。
- 配置缺失时默认值必须为 20。
- limit 必须校验为正整数。

### 2. client / pipeline 参数化

`TraktTrendingClient.fetch_shows_trending()` 和 `fetch_movies_trending()` 必须使用传入 limit。

`fetch_and_store_trakt_trending()` 新增参数：

```python
shows_limit: int = 20
movies_limit: int = 20
```

### 3. daily-discover 行为

`daily-discover` 调用 Trakt pipeline 时读取配置。

要求：

- 默认 show/movie 各 20。
- 控制台输出或 stats 中能看出 Trakt limit。
- 不改变 merge / scoring 逻辑。

### 4. 独立 CLI 行为

`fetch-trakt-trending` 建议新增：

```bash
--shows-limit 20
--movies-limit 20
```

要求：

- 不传参数时读取配置。
- 传参数时以 CLI 为准。

## 风险点

- 如果未来需要观察池，可另设任务把 20 以外内容写入非主候选池；本任务不做。
