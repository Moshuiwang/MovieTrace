## 当前阶段

## 来源任务
来自 P1.8 讨论：TMDb trending/popular/detail 默认不返回 IMDb ID；需要在每日发现主线完成后，额外请求 TMDb `/external_ids`，把外部 ID 映射结构化入库，并用于后续 IMDb ID / OMDb 评分补全。

2026-05-13 追加决策：P1.8-F 与 P1.8-G 合并实现。external_ids 不应只作为评分后的后置补拉，而应同时支持评分前即时补 IMDb ID；本任务保留 external_ids 入库、缓存、统计和可重跑要求，评分链路要求以 P1.8-G 为准。

## 目标
为 `daily-discover` 增加可统计、可限流、可重跑的 TMDb external_ids 入库能力，并与 P1.8-G 合并执行：评分前对缺 IMDb ID 的候选即时补齐，评分后可继续补齐/刷新 external_ids 缓存。

## 非目标
- ❌ 不调整评分权重、P0/P1/P2 阈值或推荐排序规则。
- ❌ 不解决 OMDb key 配额、授权或 401 问题。
- ❌ 不重构全部 TMDb 字段结构化；该范围属于 P1.8-C / P1.8-E。
- ❌ 不补拉历史全量 external_ids，除非通过显式 CLI 参数触发。
- ❌ 不引入新依赖。
- ❌ 不提交、替换、打印或记录完整 API key / token。
- ❌ 不单独作为“评分后才补拉”的实现交付；评分前补齐由 P1.8-G 约束，二者合并实现。

## 具体要求

### 1. 入库结构

优先新增表 `tmdb_external_ids`，至少包含：

- `tmdb_id`
- `media_type`
- `imdb_id`
- `tvdb_id`
- `wikidata_id`
- `facebook_id`
- `instagram_id`
- `twitter_id`
- `raw_payload_json`
- `fetched_at`
- `source_date`

唯一约束建议为：

- `(tmdb_id, media_type)`

如项目已有统一 `external_ids` 表且 P1.8-E 已完成，可同时同步到统一表，但不能丢失 TMDb 原始响应。

### 2. TMDb client

在 `src/movietrace/sources/tmdb.py` 增加 external_ids 请求能力：

- `fetch_tv_external_ids(tmdb_id: int)`
- `fetch_movie_external_ids(tmdb_id: int)`
- 或一个统一方法 `fetch_external_ids(tmdb_id: int, media_type: str)`

请求路径必须按 `media_type` 区分：

- `tv` 使用 `/tv/{series_id}/external_ids`
- `movie` 使用 `/movie/{movie_id}/external_ids`

不支持或未知的 `media_type` 不应发请求，应记录为 skip。

### 3. 每日主线补拉步骤

在 `daily-discover` 主线中增加 external_ids 补拉步骤。执行顺序必须兼容 P1.8-G：

- 评分前：对缺 IMDb ID、已有 TMDb ID 的候选即时查询 external_ids，补齐后进入 OMDb enrichment。
- 评分后：可继续对本次候选池做 external_ids 缓存补齐/刷新，用于后续重跑和审计。

补拉对象：

- 默认只处理本次 daily run 候选池中的条目。
- 必须有 `tmdb_id`。
- `media_type` 必须可识别为 `tv` 或 `movie`。
- 已存在 `tmdb_external_ids` 且未过期时跳过。
- 候选已有 IMDb ID 时，仍可入库 external_ids，但不得覆盖更高优先级来源的 IMDb ID。

推荐刷新策略：

- 默认缓存有效期 30 天。
- 支持显式 `--refresh-external-ids` 强制刷新。
- 支持 `--external-ids-limit N` 限制单次请求量，便于控制 TMDb 请求数量。

### 4. IMDb ID 回填规则

当候选无 IMDb ID 且 TMDb external_ids 返回 `imdb_id` 时：

- 将 `imdb_id` 纳入后续 OMDb enrichment 输入。
- 若本次 run 已经错过 OMDb enrichment，报告中必须明确：
  - 本次补到多少 IMDb ID。
  - 有多少条需要下一次 run 才能获得 OMDb 分数。
  - 或实现二阶段补充：external_ids 后立即对新补 IMDb ID 的候选执行 OMDb enrichment。

冲突处理：

- 如果候选已有 IMDb ID 且与 TMDb external_ids 不一致，不得静默覆盖。
- 必须记录冲突数量和样例到报告。
- 优先级沿用 P1.8-E：Trakt > FlixPatrol > TMDb external_ids > OMDb。

### 5. API 用量与限流

- external_ids 请求必须可计数。
- 若 P1.8-D 已完成，写入 `api_usage_log`：
  - service=`tmdb`
  - endpoint=`external_ids`
  - operation=`daily_external_ids_backfill`
  - http_status
  - cache_status
  - item_count
  - duration_ms
  - rate_limited
- 若遇到 429，必须停止或退避，不得无限重试。
- 不允许日志中出现完整 Bearer token。

### 6. CLI 与报告

`daily-discover` dry-run 与真实运行都要展示 external_ids 汇总：

- scanned：候选扫描数
- eligible：有 tmdb_id 且 media_type 可识别数
- cache_hit：已有未过期 external_ids 数
- fetched：实际请求数
- imdb_filled：新补 IMDb ID 数
- conflicts：IMDb ID 冲突数
- skipped：跳过数及原因

验证报告必须包含：

- 当天运行命令
- external_ids 请求量
- 入库行数
- 新补 IMDb ID 数
- 是否触发 OMDb 二阶段补分
- 已知失败或剩余风险

## 风险点
- TMDb external_ids 是额外请求；如果对全候选无节制补拉，会显著增加 TMDb 用量。
- 如果 P1.8-D 未完成，短期内只能通过本任务报告统计 external_ids 请求量，无法形成统一 API 用量总账。
- 如果 OMDb key 当日已达配额，新补 IMDb ID 也无法立刻转换为 IMDb 分数。
- daily-discover 当前 dry-run 行为可能会写入部分源缓存；本任务需要延续现有约定，并在报告中写清楚。
- IMDb ID 冲突需要审阅，不能只保留最后写入值。
