## 任务名称
P1.34 CI 部署链路韧性补强：拆分 concurrency group + deploy 末尾轻冒烟 + auto-merge dispatch 失败告警

## 来源任务
2026-05-19 流程审视会话 —— 用户询问当前开发/push/部署流程是否有问题。审视中识别出三个真实风险：

1. `cancel-in-progress: true` 会腰斩正在跑的 deploy（migrate 中途被 SIGKILL → DB 状态不一致）
2. deploy 成功 ≠ 程序能跑（缺 import 链路冒烟，部署后 cron 静默挂）
3. auto-merge dispatch CI 失败静默（`gh workflow run` 失败 → 代码合了但生产没部署）

关联决策：[[project_v1_no_pr_review]] V1 阶段不引入 review gate；本任务只补"链路韧性"，不补"代码质量"。

## 目标
让"每天 4-5 次部署"的 CI 链路在三类失败模式下不静默：deploy 不被中途腰斩、import 链路断在 deploy 阶段就发现、auto-merge dispatch 失败立即飞书告警。

## 非目标
- ❌ 不引入 PR review gate（V1 阶段不做，V2 再加）
- ❌ 不替换 smoke 为重端到端测试（不调外部 API，不烧 quota）
- ❌ 不实现自动回滚（DB migrate 不可逆，仅代码回滚反而更乱）
- ❌ 不抽 reusable workflow（当前 notify 代码只两处，YAGNI）
- ❌ 不动 manual-pipeline.yml 的并发策略（本次先解决 ci.yml 内部；跨 workflow mutex 另列任务）
- ❌ 不动 secrets f-string 内嵌（V1 阶段未触发，留作后续）
- ❌ 不写 Python 单测（workflow 文件无业务逻辑，靠 yaml 静态验证 + 真实 CI run 验证）
- ❌ 不做"smoke 故意挂"端到端负向测试（避免污染生产 deploy，等真实事故时自然验证）

## 具体要求

### A. ci.yml concurrency 拆分

**当前**（第 13-15 行）：
```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```
作用于所有 jobs（test / deploy / notify）—— 新 push 会中止正在跑的 deploy。

**改成**：删除顶层 `concurrency`；改为 job-level：

```yaml
# 删除顶层 concurrency 块

jobs:
  test:
    runs-on: ubuntu-latest
    concurrency:
      group: ci-test-${{ github.ref }}
      cancel-in-progress: true
    timeout-minutes: 10
    ...

  deploy:
    needs: test
    if: (github.event_name == 'push' || github.event_name == 'workflow_dispatch') && github.ref == 'refs/heads/main'
    concurrency:
      group: ci-deploy-${{ github.ref }}
      cancel-in-progress: false   # 关键：deploy 不被取消
    runs-on: ubuntu-latest
    timeout-minutes: 10
    ...

  # notify 保持不变（无 concurrency）
```

说明：test job 仍然可被新 push 取消（节省 runner 分钟），deploy 一旦开始就跑完。timeout-minutes:10 已经兜底，hang 不会无限占住。

### B. ci.yml deploy 末尾加 smoke test

在 deploy job 现有 "Deploy to production (pull + backup + migrate)" 步骤之后、"Sync secrets to VPS" 之前，新增一个 step：

```yaml
- name: Smoke test (import check)
  run: |
    ssh -i ~/.ssh/deploy_key ${{ secrets.VPS_USER }}@${{ secrets.VPS_HOST }} bash <<'REMOTE'
    set -e
    cd ~/MovieTrace
    source .venv/bin/activate
    PYTHONPATH=src python -c "from movietrace.cli import main; print('smoke: cli import ok')"
    REMOTE
```

为什么放在 secrets 同步之前：smoke 只验 import 链路、不依赖 secrets 更新；放前面失败更快、更省 SSH 来回。逻辑上前后都对，按这个顺序写即可。

### C. auto-merge.yml dispatch 后 poll + 告警

**当前**（第 73-77 行）：
```bash
sleep 3
echo "🚀 触发 main 分支 CI deploy..."
gh workflow run CI --ref main
echo "✅ CI dispatch 已发送"
```

dispatch 之后没有任何验证 —— 失败静默。

**改成**：dispatch 后 poll `gh run list` 确认有新 CI run 进入 `queued` / `in_progress` 状态；超时未见就发飞书告警 + 非零退出。

在现有 step（"Auto-merge with Squash"）的末尾追加（**所有逻辑放在同一个 step 内**，避免跨 step 变量传递）：

```bash
# 合并后触发 main 分支 CI 以执行 deploy（auto-merge 不触发 push 事件）
sleep 3
echo "🚀 触发 main 分支 CI deploy..."

DISPATCH_OK=false
if gh workflow run CI --ref main; then
  sleep 5  # 给 GitHub 几秒钟创建 run
  for i in $(seq 1 10); do
    LATEST_STATUS=$(gh run list --workflow CI --branch main --limit 1 --json status --jq '.[0].status' 2>/dev/null || echo "")
    if [ "$LATEST_STATUS" = "queued" ] || [ "$LATEST_STATUS" = "in_progress" ]; then
      DISPATCH_OK=true
      echo "✅ CI run 已创建并进入 $LATEST_STATUS 状态"
      break
    fi
    sleep 3
  done
fi

if [ "$DISPATCH_OK" = "false" ]; then
  echo "❌ Dispatch CI 失败或 run 未在 30s 内创建，发送飞书告警"
  python3 - <<'PYEOF'
  import os, json, urllib.request
  app_id = os.environ["FEISHU_APP_ID"]
  app_secret = os.environ["FEISHU_APP_SECRET"]
  chat_id = os.environ["FEISHU_NOTIFY_CHAT_ID"]
  pr_number = os.environ.get("PR_NUMBER", "?")
  repo = os.environ.get("REPO", "?")
  run_id = os.environ.get("RUN_ID", "?")

  token_resp = urllib.request.urlopen(urllib.request.Request(
      "https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal",
      data=json.dumps({"app_id": app_id, "app_secret": app_secret}).encode(),
      headers={"Content-Type": "application/json"},
      method="POST"
  ))
  token = json.loads(token_resp.read())["tenant_access_token"]

  text = (
      f"❌ MovieTrace auto-merge dispatch 失败\n"
      f"PR #{pr_number} 已合并但 main CI 未触发\n"
      f"需手动跑：gh workflow run CI --ref main\n"
      f"https://github.com/{repo}/actions/runs/{run_id}"
  )

  req = urllib.request.Request(
      "https://open.feishu.cn/open-apis/im/v1/messages?receive_id_type=chat_id",
      data=json.dumps({"receive_id": chat_id, "msg_type": "text", "content": json.dumps({"text": text})}).encode(),
      headers={"Content-Type": "application/json", "Authorization": f"Bearer {token}"},
      method="POST"
  )
  result = json.loads(urllib.request.urlopen(req).read())
  print(f"Feishu notify: code={result.get('code')} msg={result.get('msg')}")
  PYEOF
  exit 1
fi

echo "✅ CI dispatch 已发送并确认 run 已创建"
```

`PR_NUMBER` 变量是该 step 上方已经 `PR_NUMBER=$(gh api ...)` 设过的本地 bash 变量，同 step 内可直接 `$PR_NUMBER` 引用（无需 `$GITHUB_ENV` 传递）。在 python heredoc 内通过 `os.environ` 取，需要在 `python3 - <<'PYEOF'` 之前 export，或改成在 step 顶部 `env:` 块声明：

```yaml
env:
  GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  WORKFLOW_RUN_ID: ${{ github.event.workflow_run.id }}
  FEISHU_APP_ID: ${{ secrets.FEISHU_APP_ID }}
  FEISHU_APP_SECRET: ${{ secrets.FEISHU_APP_SECRET }}
  FEISHU_NOTIFY_CHAT_ID: ${{ secrets.FEISHU_NOTIFY_CHAT_ID }}
  REPO: ${{ github.repository }}
  RUN_ID: ${{ github.run_id }}
```

并在 bash 段内 `export PR_NUMBER` 让 python heredoc 看到。

## 风险点

- **concurrency 改动的副作用**：拆 group 后 test 仍可被新 push 取消，但 deploy 一旦开始就跑到死。`timeout-minutes: 10` 兜底，hang 最多占 10 分钟。
- **smoke 覆盖盲区**：只检 `from movietrace.cli import main`，抓不到子模块 lazy import（如某个 cmd 才 import 的外部依赖缺失）。如发现盲区，后续扩展为 `cli --help` 或 `cli daily-discover --dry-run`（后者烧 API quota，权衡）。
- **dispatch poll 误报**：30s poll 窗口经验值。若 GitHub API 真挂超过 30s 但实际 dispatch 成功 → 误报。当前接受："宁可多一条告警不要少一条"。
- **飞书 secrets 在 auto-merge 上下文可用性**：GitHub Secrets 在 workflow_run 触发的 workflow 里默认可访问，但需要 step 显式声明 `env:` 引用。实施时确认 secrets 引用语法正确（`${{ secrets.FEISHU_APP_ID }}`）。
- **本任务不验证负向链路**：smoke 故意失败 → ❌ 飞书 / dispatch 故意失败 → ❌ 飞书，这两条只做 yaml review，不端到端测（避免污染生产 deploy）。等真实事故时自然验证（项目对外部展示链路的 auto-repair 容忍度）。
- **不要顺手扩范围**：不动 manual-pipeline.yml / secrets f-string / ssh-keyscan pinning（CLAUDE.md 第 4 条）
