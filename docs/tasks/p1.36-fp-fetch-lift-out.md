# 任务包：FP fetch 提出到 CLI 层

**任务包版本：** v1
**创建时间：** 2026-05-20 +08

## 任务名称
P1.36 fp-fetch-lift-out — 将 FlixPatrol 抓取从 run_discovery() 提出到 CLI 层

## 任务类型
refactor

## 当前阶段
Phase 1

## 来源任务
本次会话设计讨论：FP 当前嵌套在 `run_discovery()` 内部，与 TMDb/Trakt 在 CLI 层独立抓取的模式不一致，
导致无法在进度输出中为 FP 显示独立步骤，也无法统一传递 fp_fetch_result 给后续通知。

## 目标
将 `_ensure_fp_data()` 调用从 `run_discovery()` 内部移至 `cmd_daily_discover()` CLI 层，
与 TMDb / Trakt 并排，结果作为参数（`fp_fetch_result`）传入 `run_discovery()`，
使三个数据源在架构上对称。

## 非目标
- ❌ 不改 `_ensure_fp_data()` 函数本身的逻辑
- ❌ 不改抓取频率、缓存、合规约束
- ❌ 不改进度输出格式（那是 P1.37 的工作）
- ❌ 不引入新依赖

## 允许修改范围
- `src/movietrace/cli.py`（`cmd_daily_discover()` 函数）
- `src/movietrace/pipeline/discovery.py`（`run_discovery()` 签名 + 内部 FP 调用移除）

## 禁止修改范围
- 🚫 `src/movietrace/pipeline/discovery.py` 中 `_ensure_fp_data()` 函数体
- 🚫 `src/movietrace/sources/flixpatrol_api.py`
- 🚫 `src/movietrace/feishu/` 所有文件
- 🚫 AGENTS.md、CLAUDE.md

## 相关上下文
**现状**：
- `cmd_daily_discover()` 先调 `fetch_and_store_tmdb_trending()`、`fetch_and_store_trakt_trending()`，
  再调 `run_discovery(tmdb_fetch_result=..., trakt_fetch_result=...)`
- `run_discovery()` 内部第一步就调 `_ensure_fp_data()`，FP 结果存在局部变量

**目标状态**：
- `cmd_daily_discover()` 先调 `_ensure_fp_data()`，拿到 `fp_fetch_result`
- `run_discovery()` 接收 `fp_fetch_result` 参数，跳过内部的 `_ensure_fp_data()` 调用
- `_resolve_source_dates_with_fallback()` 的 `flixpatrol_error` 参数从 `fp_fetch_result` 读取（已有机制）

## 输入
- `src/movietrace/cli.py` 现状
- `src/movietrace/pipeline/discovery.py` 现状

## 输出
- 修改后的 `src/movietrace/cli.py`
- 修改后的 `src/movietrace/pipeline/discovery.py`

## 具体要求
1. `run_discovery()` 新增参数 `fp_fetch_result: dict | None = None`
2. `run_discovery()` 内部：若 `fp_fetch_result is not None`，跳过 `_ensure_fp_data()` 调用，
   直接用传入的 `fp_fetch_result`；否则保持原有行为（向后兼容，测试可直接调用）
3. `cmd_daily_discover()` 在调 `run_discovery()` 之前调用 `_ensure_fp_data()`，
   结果传入 `fp_fetch_result=fp_result`
4. `_ensure_fp_data()` 函数签名和逻辑**不变**，仍在 `discovery.py` 中定义，但需从 cli.py import
