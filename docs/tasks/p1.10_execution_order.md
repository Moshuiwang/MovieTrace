## 建议执行顺序

### 0. 前置确认：P1.9 hotfix 状态

执行 P1.10 前先确认：

- P1.9 hotfix 是否已经完成或是否需要先合并。
- `PYTHONPATH=src .venv/bin/python -m pytest tests/ -v` 是否能作为可用基线。
- FP / OMDb 阻塞是否影响本轮真实验证。

### 1. P1.10-A TMDb 热源抓取精简

先收窄 TMDb。

原因：

- 影响面小。
- TMDb 已有 `pages_per_endpoint` 参数，只需配置化和接入 daily-discover。
- 可独立验证从 9 请求/天降到 3 请求/天。

### 2. P1.10-B Trakt 热源抓取精简

第二步收窄 Trakt。

原因：

- Trakt 是 P3 膨胀主因。
- 将 `shows_limit=20`、`movies_limit=20` 后，理论从约 625 条/天降到 40 条/天。
- 与 TMDb 精简独立，可单独提交。

### 3. P1.10-C 抓取状态表与状态记录基础

第三步建兜底基础设施。

原因：

- 先有状态表，才能可审计地记录 fresh / fallback / failed。
- 不直接改运行链路，便于小步验证 migration 和 helper。

### 4. P1.10-D 每日抓取失败兜底运行机制

第四步实现 source-level fallback。

原因：

- 依赖 P1.10-C。
- 需要改 `discovery.py` 和 `multi_source_merge.py`，影响主链路，必须在 schema/helper 稳定后执行。

### 5. P1.10-E 兜底来源在报告与导出中可感知

最后改报告。

原因：

- 依赖 P1.10-D 输出的 source status。
- 报告层只负责展示，不应先于运行机制实现。

## 用户已确认决策

1. `source_fallback.max_staleness_days` 默认值为 30 天。
2. 兜底只覆盖 FP / TMDb / Trakt 三个热源，不覆盖 OMDb / TMDb detail / external_ids 富化。
3. 使用 fallback 数据时允许进入供应商主列表，但报告和导出必须显著标记。
4. TMDb / Trakt 每接口默认 20 条，保留 config / CLI 覆盖能力。
