## 任务名称
P1.54 API usage 日志写库遇锁快速放弃，避免阻塞生产 pipeline

## 任务类型
fix

## 当前阶段
Phase 1（V1 运行观察期）

## 来源任务
用户反馈：2026-05-23 生产每日任务耗时约 1 小时，明显超出预期。

生产证据：
- `reports/logs/logs/daily_20260523.log`：`daily-discover` 08:00:01 开始，09:01:34 结束，discover 耗时 3693s；后续 export/sync/doc/notify 约 10s。
- `discover_stats_20260523.json`：OMDb `api_calls=164 cache_hits=0`；TMDb detail `api_calls=141 cache_hits=0`。
- 生产 `api_usage_log`：2026-05-23 出现 transport `ok` + business `success` 双写；OMDb 阶段主连接持有 `api_cache` 写事务后，`log_api_call()` 使用独立 sqlite 连接写日志，默认 busy timeout 可等待约 5s。

根因判断：
`log_api_call()` 是 best-effort 日志，不应阻塞主流程；但当前 `sqlite3.connect(db_path)` 默认 timeout 约 5s。pipeline enrichment 批量写 `api_cache` 时，主连接持有写锁，API 日志的独立连接会反复等待锁，等待失败后又被吞掉。大量 API 调用时，5s 等待被放大成数十分钟。

## 目标
让 `api_usage_log` 写入在 SQLite 被锁时快速放弃，不再让 best-effort 日志拖慢 `daily-discover`。

## 非目标
- ❌ 不回滚 P1.43 的 enrichment 批量提交
- ❌ 不修改 `api_usage_log` schema
- ❌ 不重构 API logging 架构
- ❌ 不修改 OMDb/TMDb/Trakt/FlixPatrol 业务调用逻辑
- ❌ 不补写 2026-05-23 已丢失的 usage log

## 具体要求
1. 给 `log_api_call()` 的 sqlite 连接设置极短 timeout（建议 `timeout=0.1` 秒）。
2. 保持所有异常吞掉的 best-effort 语义。
3. 不改变 `log_api_call()` 函数签名，避免扩大调用面。
4. 新增测试模拟另一个连接持有写锁时，`log_api_call()` 快速返回且不抛异常。
5. 测试不要依赖真实生产库。

## 验收标准
1. ✅ 被锁数据库上调用 `log_api_call()` 不抛异常。
2. ✅ 被锁数据库上调用耗时 < 1s。
3. ✅ 正常未锁数据库仍能写入 `api_usage_log`。
4. ✅ `tests/test_api_usage_logging.py` 通过。
5. ✅ 全量测试通过。

## 风险点
- 短 timeout 会在锁竞争时丢弃 usage log；这符合 best-effort 日志语义，优先保证 pipeline 不被日志阻塞。
- 如果未来要求 usage log 绝不丢，需要单独任务设计同连接写入或异步队列，不在本任务范围。
