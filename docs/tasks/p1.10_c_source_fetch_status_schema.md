## 相关上下文
当前 source tables 只保存抓取结果，没有保存“今天是否使用了兜底数据”。如果直接复制旧行到当前日期，会污染真实 source snapshot。更稳妥的方式是新增状态表，记录目标日期使用了哪个 source snapshot。

建议新增表：`source_fetch_runs`。

## 具体要求

### 1. 表结构

建议字段：

```sql
create table source_fetch_runs (
    id integer primary key autoincrement,
    target_date text not null,
    source text not null,
    status text not null,
    source_snapshot_date text,
    started_at text not null default current_timestamp,
    finished_at text,
    rows_fetched integer,
    rows_inserted integer,
    rows_used integer,
    error_message text,
    config_json text,
    created_at text not null default current_timestamp,
    updated_at text not null default current_timestamp
);
```

唯一键：

```sql
unique(target_date, source)
```

索引：

- `(target_date, source)`
- `(source, status)`
- `(source_snapshot_date)`

### 2. 状态枚举

至少支持：

- `fresh`：当天成功抓取并使用当天数据。
- `fallback`：当天抓取失败或无可用数据，使用历史 source snapshot。
- `failed_no_fallback`：当天抓取失败且找不到可用历史数据。
- `skipped`：因配置或调度跳过，例如 FP movie 非周抓取日。

### 3. helper 能力

新增模块建议：`src/movietrace/pipeline/source_fetch_status.py`

至少提供：

- `record_source_fetch_run(...)`
- `get_source_fetch_runs(conn, target_date)`
- `find_latest_source_snapshot(conn, source, before_date, max_staleness_days)`
- `build_effective_source_dates(conn, target_date)`

### 4. 不复制旧数据

本任务只记录状态，不把上一日数据复制成当前日期数据。实际读取旧 snapshot 由 P1.10-D 完成。

## 验收标准
1. ✅ migration 幂等。
2. ✅ `source_fetch_runs` 表和索引存在。
3. ✅ 同一 `target_date + source` 可 upsert 状态。
4. ✅ 可查询某日所有 source 状态。
5. ✅ 可找到某 source 的最近可用 snapshot。
6. ✅ 不修改既有 source data 表。

## 风险点
- migration 编号需执行前核对，避免和 P1.9 hotfix schema 任务冲突。
- 用户已确认 `max_staleness_days` 默认值为 30 天；报告必须显著标记 fallback，避免人工误以为是当天 fresh 数据。
- 如果 source 表没有任何历史数据，兜底机制只能记录 `failed_no_fallback`。
