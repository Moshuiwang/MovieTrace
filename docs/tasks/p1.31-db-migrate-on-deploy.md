## 具体要求

### A. `movietrace migrate` CLI 子命令
- 复用现有 `initialize_database()`（已幂等）：直接调用即可，不要写新逻辑
- 输出格式：
  ```
  DB path: data/movietrace.db
  before version: 16
  after  version: 17
  applied: [17]
  ```
- 退出码：成功 0；任何异常退 1 并把异常信息写 stderr
- 不接 `--dry-run`（YAGNI，不在本任务范围）

### B. `SCHEMA_VERSION` 修正
- `SCHEMA_VERSION = 17`（const-only bump，符合 [.claude/rules/21-db-migrations.md](../../.claude/rules/21-db-migrations.md) §"SCHEMA_VERSION 同步"；grep 确认 src/tests/scripts 零引用，纯文档常量，无运行时副作用）
- **不修改 `SCHEMA_SQL`**：现状 `SCHEMA_SQL` 是 v1 baseline（末尾 `insert or ignore into schema_migrations(version) values (1)`），migrations 在其上叠加。参考 migration 016 的 DROP 同步模式——SCHEMA_SQL 仅在 DROP 时同步删除 CREATE；ADD COLUMN 类 migration 不动 SCHEMA_SQL。若把 zh 4 列塞进 `canonical_items` CREATE，新建库流程会变成：SCHEMA_SQL 先建带 zh 列的表 → migration 017 的 `ALTER TABLE ... ADD COLUMN title_zh` 报 `duplicate column name`。
- 不动 migration 017 文件本身
- 等价性验证：fresh DB 跑 `initialize_database()`（SCHEMA_SQL + 002..017）与 existing v16 DB 跑 `initialize_database()`（仅 017 应用）应得到结构一致的 `canonical_items`——通过测试 D2 覆盖

### C. `ci.yml` deploy 步骤改造
原来：
```yaml
ssh ... 'cd ~/MovieTrace && git pull origin main'
```
改为单次 SSH heredoc：
```yaml
ssh ... <<'EOF'
  set -e
  cd ~/MovieTrace
  git pull origin main
  cp data/movietrace.db "data/movietrace_backup_$(date +%Y%m%d_%H%M)_pre_migrate.db"
  PYTHONPATH=src python -m movietrace.cli migrate
EOF
```
要点：
- `set -e`：任一步失败即终止，deploy job 失败
- 备份命名遵循 [.claude/rules/21-db-migrations.md](../../.claude/rules/21-db-migrations.md)：`data/movietrace_backup_<YYYYMMDD>_<HHMM>_pre_migrate.db`
- 第一次跑会自动应用 017，事故顺带修复
- notify 步骤无需改动（已根据 `deploy_result` 上报失败）

### D. 测试
新增 `tests/test_migrate_cli.py`（或加入 `test_database_schema.py`）：
1. 用 tempfile 建一个只到 v15 的库（手工 `_apply_migration` 跑 2-15），跑 `cli.main(["migrate"])`，断言 `max(schema_migrations.version) == 17`
2. 已经是 v17 的库再跑 `migrate`，断言 version 不变、无异常、退出码 0
3. 把 `_apply_migration` monkeypatch 成抛异常，断言 CLI 退出码 ≠ 0、stderr 含异常信息

## 风险点
- **生产首次跑 migrate 即面对真实数据，无回退余地**：靠 ① 单测覆盖 ② deploy step 内 `cp` 备份 ③ `_apply_migration` 已幂等 三重保护。失败时备份文件可手工恢复（恢复本身需要走 P1.32 manual workflow，或临时一次性 workflow）。
- ~~`SCHEMA_SQL` 改 `canonical_items` 的 CREATE 不能影响已有库~~ **作废**：代码层核验后已确认不改 SCHEMA_SQL（见 §B 修订原因）。
- **`ssh ... <<'EOF'` heredoc 在 ci.yml 中需注意引号转义**：用 `<<'EOF'`（带单引号）禁用本地变量展开，让 `$(date +%Y%m%d_%H%M)` 在远端执行。CI 上跑前用 `gh workflow view` / push 到一个 throwaway 分支预演一次。
- **本任务不解决"重跑当天 pipeline"**：今天事故的数据补救（重跑 export）依赖 P1.32 的 manual workflow；P1.31 合并 + auto-deploy 完成后，只能等明早 cron 跑出今天遗漏的产出，或者立刻接 P1.32。
- **写代码前必须 `git fetch && git diff origin/main`**：origin/main 可能继续向前，避免基于陈旧 schema.py / ci.yml 改动（[[project_ci_cd_pipeline]]）。
