## 相关上下文
- `content_updates` 已是事件历史表，以 `content_update_id` 唯一。
- `tmdb_trending` 已有 TV freshness 字段，但当前 trending payload 不填充 `tmdbtv_status` / `number_of_seasons` 等 detail 字段。
- `api_cache` 已缓存 `tmdb:detail:{id}:tv` / `tmdb:detail:{id}:movie` 响应，可用于减少重复 TMDb detail 请求。
- 当前 `daily-discover` 会在最后一步调用 `baseline_tracking.run_baseline_tracking()`，需要拆除。

## 输入
- 当前 SQLite B 库 `data/movietrace.db`
- `config.yaml` 中 baseline tracking 配置
- 现有 TMDb detail cache
- 现有 scoring 配置 `config/scoring_weights.yaml`

## 输出
- `daily-discover` 只写热点发现事件。
- `baseline-track` 支持 `--mode catch-up|routine`。
- `export-baseline-updates` 生成独立 baseline MD/JSON 报告。
- `scripts/baseline_run.sh` 可由上层 crontab 每周调用。
- 文档同步新的产品需求和运行方式。

## 具体要求
1. `daily-discover` 不再调用 baseline tracking，CLI 输出不得显示 baseline 写入结果。
2. `baseline-track --mode catch-up` 覆盖所有非 `skip` 的 `virtual_series`。
3. `baseline-track --mode routine` 只追踪仍可能更新的剧集：`Returning Series`、`In Production`、以及状态为空/未知的兜底条目。
4. baseline detail 获取优先读取 24 小时内 `api_cache`；缓存缺失或缺关键字段时再调用 TMDb `/tv/{series_id}`。
5. baseline 写入 `new_season` 时必须写入非空 `hot_score`，复用现有 `compute_hot_score()` / `map_priority()`。
6. baseline `source_summary_json` 需包含 `tmdb_status`、`in_production`、`last_episode_to_air`、`next_episode_to_air`、`hot_score_breakdown`。
7. `export-recommendations` 默认只导出非 `new_season` 热点事件，并继续写 `recommendations_*.md/json` 与 `latest.md/json`。
8. 新增 `export-baseline-updates`，只导出 `new_season`，写 `baseline_updates_*.md/json` 与 `baseline_latest.md/json`。
9. 报告标题写明报告类型和导出时间点，不写 weekly；运行频率由上层脚本决定。
10. `scripts/baseline_run.sh` 执行 `baseline-track --mode routine`，成功后执行 `export-baseline-updates --days 7`，并写入独立日志。

## 风险点
- baseline catch-up 会消耗 TMDb detail API 并增加运行时间；当前按用户决策接受一次追平。
- `hot_score` 对 baseline 新季可能缺少 FP/Trakt/OMDb 信号，应允许只基于 TMDb detail 得出较低但非空的分数。
- 当前真实库已有 2026-05-15 手动 commit 写入的 `new_season` 事件；本任务不回填历史事件 hot_score。
