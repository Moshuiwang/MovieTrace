## 任务名称

## 任务类型
refactor

Phase 1（V1 运行观察期）

## 来源任务
用户性能观察：OMDb enrichment 每次请求后硬编码 `time.sleep(1.0)`，注释仅写 `# polite`——无实测依据。OMDb 实际限制是**每日配额**（1000 req/key/day），不是 per-second rate limit。今日实测用了 9.4% 配额，1s sleep 白白浪费 94 秒，无任何保护作用。

## 前置知识（执行前必须理解）

### OMDb 的错误检测路径已完整，能知道 API 出错

| 错误类型 | OMDb 行为 | 代码检测路径 |
|---|---|---|
| 日配额耗尽 | HTTP 200 + body `{"Error":"Request limit reached!"}` | `omdb.py:177` 检测 `"limit reached"` → `_log_omdb_quota_error()` 写 `api_usage_log` → 抛异常 → `FatalApiError` 捕获 → key 标记 dead |
| HTTP 4xx/5xx | 非 200 状态码 | `FatalApiError` → key 标记 dead + `logger.warning` |
| 网络错误 / 超时 | 连接失败 | `except Exception` → `logger.warning` + 跳过该条 |
| 所有 key 耗尽 | — | circuit breaker `logger.warning` + 提前 break |

**结论：减少 sleep 不会"悄悄出错"。** 任何异常都有日志；配额耗尽会把 key 标记 dead 并在汇总行反映。

### 当前缺失的可观测性

汇报汇总行没有 `quota_errors` 计数，导致无法从终端直接判断"这次 run 有没有触发配额限制"。本任务一并补上。

## 目标

1. 将 `time.sleep(1.0)` 提取为函数参数 `inter_request_sleep: float = 0.2`（降到 0.2s）
2. 在返回 dict 和汇总日志里增加 `quota_errors` 计数字段
3. enrichment 开始时打一行含 sleep 值的日志，便于将来调参时确认生效

## 非目标
- ❌ 不删除 sleep（保留 0.2s 作为默认值，明确可配置）
- ❌ 不修改 OMDb 的错误检测逻辑（已完整）
- ❌ 不修改 `discovery.py` 调用方（沿用默认值）
- ❌ 不修改 TMDb detail 循环（它没有 sleep，不需要此改动）
- ❌ 不引入新依赖

## 具体要求

### 1. 函数签名变更

```python
def enrich_with_omdb(
    conn: sqlite3.Connection,
    candidates: list[MergedCandidate],
    api_keys: list[str],
    cache_ttl_hours: int = 72,          # P1.49 已改（或本任务一起改）
    inter_request_sleep: float = 0.2,   # 新增，原硬编码 1.0
    *,

```

### 2. 循环改动

```python
# 原
time.sleep(1.0)  # polite between OMDb calls

# 改为
time.sleep(inter_request_sleep)
```

### 3. quota_errors 计数

在循环开始前：
```python
quota_errors = 0
```

在 `FatalApiError` 捕获块里，检测是否为配额错误：
```python
except FatalApiError as exc:
    api_calls += 1
    dead_keys.add(key)
    if exc.status_code in (401, 403) or "quota" in str(exc).lower():
        quota_errors += 1
    logger.warning(...)
```

> **注意**：OMDb 配额耗尽实际走的是 `omdb.py` 内部的 `_log_omdb_quota_error()` + 抛异常路径，异常类型需要确认（读 `omdb.py` 后决定检测方式，可能是直接判断 `api_usage_log` 里的 `quota_error=True`，也可能是捕获特定异常子类）。执行前必须读 `src/movietrace/sources/omdb.py` 确认。

### 4. 汇总日志 + 返回值

```python
logger.info(
    "OMDb enrichment: api_calls=%d cache_hits=%d enriched=%d of %d "
    "keys_used=%d keys_exhausted=%d quota_errors=%d sleep=%.1fs",
    api_calls, cache_hits, enriched, len(candidates),
    len(tried_keys), len(dead_keys), quota_errors, inter_request_sleep,
)
return {
    "api_calls": api_calls,
    "cache_hits": cache_hits,
    "enriched": enriched,
    "keys_used": len(tried_keys),
    "keys_exhausted": len(dead_keys),
    "quota_errors": quota_errors,          # 新增
}
```

### 5. enrichment 开始时打日志

在循环开始前：

## 风险点
- **OMDb quota error 异常类型**：执行前必须读 `omdb.py` 确认配额耗尽时抛出的是什么异常（可能不是 `FatalApiError`，而是自定义子类），再决定 `quota_errors += 1` 的触发条件
- **0.2s 是否足够"礼貌"**：OMDb 无公开 per-second limit，0.2s 是保守值。若次日 live run 出现 `quota_errors > 0` 或 key 被标记 dead，可通过 `inter_request_sleep` 参数调大，无需改代码
- **与 P1.49 的关系**：两任务均改 `enrich_with_omdb()` 签名，建议合并到同一分支或 P1.49 先合并再做 P1.50
