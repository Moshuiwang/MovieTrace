## 相关上下文
- 当前 sync 使用 `run_date|content_update_id` 写入 `content_update_id` 字段，并按 `同步批次=run_date` 查询已有记录（`sync.py:107-151` `_list_records_for_date`）。
- ADR-0016 要求 B 库持有业务状态，飞书只做投影。
- 改稳定键 upsert 后，**`同步批次=run_date` 这把搜索过滤器无法找到上个/更早日期创建的同 key 旧行**，必须改造为按 `content_update_id` 内容判定：
  - discovery 行：按"该 `content_update_id` 以 `discovery:` 开头"或单 key 精确查（推荐两阶段——批量拉今日同步过的 + 对未命中的 discovery key 逐条 lookup，避免一次拉全表）。
  - `new_season` 行：仍按 `同步批次=run_date` 兼容查询（事件键含日期，天然不会跨日重复）。
  - 实施时需把现有 `_list_records_for_date` 拆为 `_list_discovery_records_by_key` 与 `_list_new_season_records_for_date`，或在调用方按 `update_type` 分流。

## 输出
- discovery 行稳定键 upsert
- `new_season` 行兼容旧事件键
- sync 单测

## 具体要求
1. discovery 行写入飞书 `content_update_id` 字段时使用稳定键 `discovery:{type}:{tmdb_id}`。
2. discovery 查重必须按内容键（以 `discovery:` 开头），而非 `同步批次=run_date`；应能找到任何历史日期创建的同 key 记录，且批量查询避免一次拉全表。
3. `同步批次` 继续写当前运行日期，表示本次系统同步批次，不作为 discovery 业务唯一键。
4. 更新 discovery 旧行时只覆盖系统字段（最近发现日期、发现次数、最新分数、最新优先级、数据源状态、同步时间），不覆盖运营状态、供应商状态、运营备注等人工字段。
5. `new_season` 仍走 `run_date|content_update_id` 形式的事件键查询路径，不被稳定 discovery key 逻辑误处理；两条路径在 sync 调用方按 `update_type` 显式分流。
6. 空飞书表或只含部分旧行的飞书表都必须可由 sync 自动重建投影；不得要求人工预先复制旧记录或手动查 record_id。
7. 历史飞书表中存在的 `run_date|discovery:t:id:date` 形式旧重复行**本任务不清理**；新逻辑只保证新 sync 不再创造重复，旧行清理由运营或独立任务承担。

## 验收标准
1. ✅ 同一 discovery key 跨两天 sync 后只更新同一飞书记录。
2. ✅ 最近发现日期、发现次数、最新分数、最新优先级会更新。
3. ✅ 人工字段在 update 时不被覆盖。
4. ✅ `同步批次` 写为最新 run_date。
5. ✅ `new_season` 不被稳定 discovery key 逻辑误处理。

## 风险点
- 历史飞书表已经存在 `run_date|content_update_id` 形式的旧行；本任务不清理历史重复行。

- 飞书是投影，不是业务状态源；如飞书表为空，应从 B 库 current/export 自动重建。
