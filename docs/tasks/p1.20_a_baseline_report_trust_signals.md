## 相关上下文
- 当前 `_baseline_local_max` 在 source_summary_json 缺 `baseline_local_max_season` 时 fallback 到 `season_min - 1` 或 `season - 1`，是估算值而非精确值
- 当前 `build_daily_poll_plan` routine 模式过滤 `tmdb_status in ('Returning Series', 'In Production')`，存量 VS 的 `tmdb_status` 为 null 时会返回空 plan，但调用方无提示

## 输入
- `src/movietrace/reports/export_writer.py`（当前 `_baseline_local_max` 实现）
- `src/movietrace/pipeline/poll_scheduler.py`（routine 模式过滤逻辑）

## 输出
- **用户决策：** fallback 值显示为 `~{n}`（前缀波浪号），JSON 增加 `baseline_local_max_season_is_fallback` 布尔字段
- 修改后的 `export_writer.py`：fallback 值显示为 `~{n}` 表示估算
- 修改后的 `baseline_tracking.py`：plan 空时根据 VS 总数和 tmdb_status 为 null 的 VS 数输出诊断日志
- 修改后的 `poll_scheduler.py`：可选加 debug/warning 日志
- 更新对应测试

## 具体要求
1. `_baseline_local_max` 改为返回 `(int | None, is_fallback: bool)` 元组，或直接在 `format_markdown` / `format_json` 中通过 truthy/None 判断标记
   - 精确值：`1`（有 `baseline_local_max_season` 字段）
   - 估算值：`~1` 或 `1?`（从 season_min/season/stored_local_max_season 推算）
2. `run_baseline_tracking` 在 `not plan and mode == "routine"` 时，查询 `select count(*), sum(case when tmdb_status is null then 1 else 0 end) from virtual_series where poll_priority != 'skip'` 输出区别：
   - 全为 null → "Plan empty: all N trackable series have null tmdb_status. Run catch-up first."
   - 有 status 但无需轮询 → "Plan empty: 0 of N trackable series match routine status filter."
   - 没有任何 trackable → "Plan empty: no trackable series found (all skipped?)."
3. JSON 导出也需标记 fallback：加 `baseline_local_max_season_is_fallback: true/false` 字段

## 验收标准
1. ✅ 精确值在报告中显示为纯数字，fallback 值带 `~` 前缀或 `?` 后缀
2. ✅ JSON 导出中 `baseline_local_max_season_is_fallback` 正确反映精确 vs 估算
3. ✅ routine 模式 plan 空时日志明确区分三种原因
4. ✅ 现有 499 测试全部通过

- 空 plan 诊断需要一次额外 SQL 查询，非关键路径，性能无影响
