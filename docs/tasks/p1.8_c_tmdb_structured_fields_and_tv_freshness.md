P1.8-C TMDb 字段结构化与 TV 新鲜度修正

## 任务类型
fix

## 当前阶段
Phase 1.8：条件性调优

## 来源任务
来自 P2+ 评分复盘讨论：当前 TV 内容使用 TMDb `first_air_date` 作为 `release_date` 参与 freshness，导致老剧更新新季时被误判为非近期内容。例如 `FROM`（TMDb `124364`）在 TMDb detail 中 `last_air_date=2026-05-10`，但当前 `tmdb_trending.release_date=2022-02-20`，freshness 被压为 0。

## 目标
完整保留并结构化 TMDb trending / popular / detail 已返回的业务相关字段，并将 TV freshness 改为优先使用 `last_air_date`，避免老剧新季被误判为老内容。

## 非目标
- ❌ 不处理 OMDb key / IMDb 评分缺失问题。
- ❌ 不排查 `FROM` 为什么不在 FP 榜单中。
- ❌ 不调整 hot_score 权重或 P0/P1/P2 阈值。
- ❌ 不引入新依赖。
- ❌ 不删除 `raw_payload_json`。
- ❌ 不做 TMDb 全库历史重抓，除非执行阶段另有明确授权。

- 当前 `tmdb_trending` 已保存 `raw_payload_json`，但结构化字段较少。
- 当前 `normalize_tmdb_trending_row()` 对 TV 使用 `first_air_date` 写入通用 `release_date`。

## 具体要求
1. `tmdb_trending` 必须继续保留 `raw_payload_json`。
2. 新增 migration 008，至少结构化以下 trending/popular 字段：
   - `adult` integer / boolean 语义
   - `softcore` integer / boolean 语义
   - `backdrop_path`
   - `poster_path`
   - `overview`
   - `genre_ids_json`
   - `origin_country_json`
   - `first_air_date`
   - `movie_release_date`
   - `original_name`
3. 新增 migration 008，至少结构化以下 TV detail 字段：
   - `last_air_date`
   - `next_air_date`
   - `last_episode_air_date`
   - `last_episode_season_number`
   - `last_episode_number`
   - `number_of_seasons`
   - `number_of_episodes`
   - `status`
   - `in_production`
   - `genres_json`
   - `seasons_json`
   - `last_episode_to_air_json`
   - `next_episode_to_air_json`
   - `networks_json`
   - `production_companies_json`
   - `spoken_languages_json`
   - `created_by_json`
4. 字段命名约定：
   - TV 最新播出日期字段统一命名为 `last_air_date`。
   - 电影上映日期保留为 `movie_release_date`，并可继续兼容旧字段 `release_date`。
   - 旧字段 `release_date` 不删除；后续可作为兼容字段，但评分层不得再把 TV 的 `first_air_date` 当作最终 freshness 日期。
5. scoring freshness 输入规则：
   - movie：使用 `movie_release_date`，缺失时 fallback 到旧 `release_date`。
   - tv：优先使用 `last_air_date`。
   - tv fallback 顺序：`last_air_date` → `last_episode_air_date` → `seasons_json` 中最大有效 `air_date` → `first_air_date` → 旧 `release_date`。
   - `next_air_date` 不参与“已上线 freshness”，仅结构化保存。
6. `TMDbDetailClient` / `enrich_with_tmdb_details()` 应在获取 TV detail 时填充上述 detail 字段。
7. `merge_three_sources()` / discovery 转评分输入时，应携带 TV 的 freshness 日期，避免 `FROM` 这类内容被按首播日期扣分。
8. 不允许因为某个字段当前无用就丢弃；对于列表或嵌套对象，使用 JSON 字段完整保存。
9. 所有 SQL 写入必须继续使用 prepared statements。

## 风险点
- migration 008 会扩展表字段，需保证生产 DB 可重复执行且不破坏既有数据。
- SQLite 中 JSON 字段仅作为 text 保存，后续查询需 `json_extract` 或 Python 解析。
- TV detail 字段不是 trending/popular payload 自带，只有执行 detail enrichment 后才完整。
- 对旧数据如果不重跑 detail enrichment，`last_air_date` 可能为空；评分层必须保留 fallback。
- 真实 TMDb API 调用受 token、网络和限流影响。
