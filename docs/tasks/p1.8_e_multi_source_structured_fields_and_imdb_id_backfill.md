## 任务名称
P1.8-E 多源字段结构化与 IMDb ID 补全

## 任务类型
feat

## 当前阶段
Phase 1.8：条件性调优前置数据治理

## 来源任务
来自 P2+ 评分复盘和 IMDb 缺失排查：当前 IMDb ID 主要依靠 FlixPatrol / Trakt，TMDb `/external_ids` 尚未接入；同时 FlixPatrol、Trakt、OMDb 与 TMDb 类似，也存在大量有价值字段仅保存在 raw JSON 中、没有结构化的问题。

## 目标
补齐多源原始响应中的关键结构化字段，并新增 TMDb external_ids 补 IMDb ID 路径，降低 IMDb ID 缺失导致的 OMDb 评分缺失。

## 非目标
- ❌ 不处理 OMDb key 配额或授权问题。
- ❌ 不调整 hot_score 权重或优先级阈值。
- ❌ 不修改 P1.8-C 中 TMDb freshness 的核心规则。
- ❌ 不删除任何 `raw_payload_json` 或 `api_cache.response_json`。
- ❌ 不引入新依赖。
- ❌ 不提交、替换或打印真实 API key。

## 相关上下文
- `flixpatrol_top10` 当前已结构化 `tmdb_id` / `imdb_id`，但未结构化 `updated_at`、company/country/title 原始 ID 等字段。
- `trakt_trending` 当前已结构化 `trakt_id` / `tmdb_id` / `imdb_id` / watchers / rating / votes，但 raw payload 中还有 `genres`、`status`、`country`、`network`、`runtime`、`overview`、`first_aired`、`aired_episodes`、`certification`、`updated_at`、`images`、`social_ids` 等字段。
- OMDb 当前只通过 `api_cache` 保存完整响应，并只应用 `imdbRating` / `imdbVotes`；`Rated`、`Genre`、`Country`、`Language`、`Type`、`totalSeasons`、`Ratings` 等未结构化。
- TMDb trending/popular 不返回 IMDb ID；TMDb 提供 `/movie/{movie_id}/external_ids` 和 `/tv/{series_id}/external_ids` 可获取 `imdb_id` 等外部 ID。
- 当前 P2+ 中已有案例：`20/20` 有 TMDb ID 但 FP 返回 `imdb_id=null`，当前无法继续 OMDb 富化。

## 具体要求

### 1. TMDb external_ids 补 IMDb ID

新增 TMDb external_ids 能力：

- TV：`GET /tv/{series_id}/external_ids`
- Movie：`GET /movie/{movie_id}/external_ids`

至少结构化保存：

- `tmdb_id`
- `media_type`
- `imdb_id`
- `tvdb_id`
- `wikidata_id`
- `facebook_id`
- `instagram_id`
- `twitter_id`
- `raw_payload_json`
- `fetched_at`

保存位置可二选一：

- 新表 `tmdb_external_ids`
- 或复用 / 扩展 `external_ids`，但必须保留 raw payload。  

推荐新建 `tmdb_external_ids`，再按需同步 `external_ids`。

补全规则：

- 候选已有 IMDb ID 时，不重复请求 external_ids。
- 候选无 IMDb ID 但有 TMDb ID 时，优先用 TMDb external_ids 补。
- 补到 IMDb ID 后，应参与后续 OMDb rating/votes 富化。
- external_ids 请求必须经过 API 使用量日志（若 P1.8-D 已完成）。

### 2. FlixPatrol 字段结构化

在保留 `raw_payload_json` 的前提下，补充结构化字段，至少包括：

- `fp_updated_at`
- `company_id`
- `country_id`
- `title_id`
- `date_from`
- `date_to`
- `type_raw`

如果当前 normalized item 没携带这些字段，应从 raw item 解析并写入。

### 3. Trakt 字段结构化

在保留 `raw_payload_json` 的前提下，补充结构化字段，至少包括：

- `status`
- `country`
- `network`
- `runtime`
- `language`
- `overview`
- `first_aired`
- `aired_episodes`
- `certification`
- `updated_at`
- `genres_json`
- `subgenres_json`
- `images_json`
- `social_ids_json`
- `available_translations_json`

### 4. OMDb 字段结构化

保留 `api_cache.response_json`，并新增结构化保存方式。可选方案：

- 新表 `omdb_details`
- 或在 enrichment 时将结构化字段写入候选相关缓存表。

推荐新建 `omdb_details`，至少结构化：

- `imdb_id`
- `title`
- `year`
- `rated`
- `released`
- `runtime`
- `genre`
- `director`
- `writer`
- `actors`
- `plot`
- `language`
- `country`
- `awards`
- `poster`
- `ratings_json`
- `metascore`
- `imdb_rating`
- `imdb_votes`
- `type`
- `total_seasons`
- `response`
- `error`
- `raw_payload_json`
- `fetched_at`

### 5. 合并逻辑

- `MergedCandidate.imdb_id` 的来源优先级应明确：
  1. Trakt `ids.imdb`
  2. FlixPatrol `imdbId`
  3. TMDb external_ids `imdb_id`
  4. OMDb search/detail 返回的 `imdbID`（仅作为低优先级补充）
- 当不同来源 IMDb ID 冲突时：
  - 不自动覆盖高优先级来源。
  - 记录 conflict 到报告或质量问题表。
  - 保留所有来源原始值。

### 6. 安全与限流

- 不输出完整 API key。
- external_ids 请求量应受控，优先只对缺 IMDb ID 且已进入 P2+ 或候选池的条目补。
- 所有请求应可被 P1.8-D 的 API usage logging 统计；若 P1.8-D 未完成，本任务报告需说明该限制。

## 风险点
- 本任务涉及 schema 扩展和多源解析，范围较 P1.8-C 更宽，执行时应优先 TDD 分步推进。
- external_ids 会增加 TMDb API 调用，应尽量只对缺 IMDb ID 的候选请求，并配合 P1.8-D 记录用量。
- OMDb 当前 key 可能额度耗尽；OMDb 结构化应优先基于已有缓存和 mock 测试，不强依赖真实请求。
- 不同来源 IMDb ID 可能冲突，不能静默覆盖。
