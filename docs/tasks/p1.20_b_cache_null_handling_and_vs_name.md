## 相关上下文
- `get_tmdb_detail_with_cache` 在 API 异常或返回 None 时执行 `return data or {}, False`，导致空 dict 被下游当作"有数据"消费，同时空 dict 不会被写入 cache（`if data: write_tmdb_detail_cache(...)`），下次调用再次 hit API
- `_update_virtual_series_from_details` 对 name 使用 `coalesce(?, name)`，即在 API 返回 null name 时保留库内值。考虑 TMDb 是权威数据源，null name 应触发日志而非静默保留旧值

- 修改后的 `baseline_tracking.py`：name 回写逻辑

## 具体要求
1. `get_tmdb_detail_with_cache` 中 `return data or {}, False` 改为：
   - API 返回 None/空 → `return None, False`（cache miss + no data）
   - API 返回有效 dict → `return data, False`（cache miss + fresh data）
   - 下游调用方 `detect_new_seasons` 的 `if not details or not details.get("name")` 已正确处理 None 情况
2. `write_tmdb_detail_cache` 仅当 `data` 为非空 dict 时写入
3. `_update_virtual_series_from_details` 中 name 改为直接覆盖：`set name = ?,`（移除 coalesce），因为 TMDb 不会返回空 name；如果返回 null，让数据库约束报错而非静默吞掉
   - 如果确实要保留防御性，改用 `set name = coalesce(nullif(?, ''), name)`（只防御空字符串），并在 API 返回 null name 时打 warning

## 验收标准

3. ✅ VS name 回写逻辑直接覆盖或用 `nullif` 防御空字符串

## 风险点
- `_update_virtual_series_from_details` 如果 TMDb 真的返回 null name（极少见），直接覆盖会把 name 写为 null。**用户决策：直接覆盖，但 null 时打 warning 日志。**
  实现：移除 `coalesce(?, name)`，直接用 `?` 覆盖；null/空 name 时 `logger.warning(...)`。
