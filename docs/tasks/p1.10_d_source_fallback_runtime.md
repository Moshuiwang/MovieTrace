## 来源任务
来自 2026-05-14 用户讨论：如果当天抓取失败，则使用上次抓取结果，并让人类在报告中感知。P1.10-C 提供状态表后，本任务实现 pipeline 运行时兜底。

## 目标
当 FP / TMDb / Trakt 当日抓取失败或无可用数据时，按 source 独立回退到最近一次可用 snapshot，并继续完成 merge / scoring。

## 非目标
- ❌ 不修改报告展示；P1.10-E 处理。
- ❌ 不改变 hot_score 权重。
- ❌ 不复制历史 source rows 到当前日期。
- ❌ 不兜底 OMDb / TMDb detail / external_ids 富化失败；本任务只处理一层热源 source snapshot。
- ❌ 不新增外部 API 请求。

## 相关上下文
当前 `merge_three_sources(conn, snapshot_date)` 只能读取同一个日期的三个 source 表。兜底机制需要支持“目标日期是今天，但某个 source 使用昨天或更早的 snapshot”。

建议采用 source-level effective date map：

```python
source_dates = {
    "flixpatrol": "2026-05-13",
    "tmdb": "2026-05-14",
    "trakt": "2026-05-14",
}
```

## 输入
- P1.10-C 的 `source_fetch_runs` 表和 helper。
- 当前 source 表历史数据。
- 当前运行日期。
- 配置项：`source_fallback.max_staleness_days`。

## 输出
- `daily-discover` 可在单个 source 失败时继续运行。
- `merge_three_sources()` 支持按 source 使用不同 snapshot date。
- stats 中包含 source status 和 effective source dates。

## 具体要求

### 1. 配置项

建议新增：

```yaml
source_fallback:
  enabled: true
  max_staleness_days: 30
  sources:
    flixpatrol: true
    tmdb: true
    trakt: true
```

用户已确认决策：

- `max_staleness_days` 默认值为 30 天。
- 兜底范围只覆盖 FP / TMDb / Trakt 三个热源，不覆盖 OMDb / TMDb detail / external_ids 富化。
- 使用 fallback 数据时允许进入供应商主列表，但必须在报告和导出中显著标记。
- TMDb / Trakt 每接口默认 20 条，保留 config / CLI 覆盖能力。

### 2. source-level 兜底

每个 source 独立判断：

- 当日 fresh 数据存在且抓取成功：使用目标日期。
- 当日抓取失败或 rows=0：查找最近可用 snapshot。
- 找到可用 snapshot：记录 `fallback` 并使用该日期。
- 找不到：记录 `failed_no_fallback`，该 source 不参与 merge。

### 3. merge 改造

`merge_three_sources()` 建议签名扩展：

```python
def merge_three_sources(conn, snapshot_date: str, source_dates: dict[str, str] | None = None)
```

要求：

- 默认行为保持兼容。
- `_read_fp` / `_read_tmdb` / `_read_trakt` 分别使用自己的 effective date。
- 输出 candidate 的 `snapshot_date` 仍为目标日期。
- candidate 或后续 dict 中可携带 `source_snapshot_dates`，供 P1.10-E 报告使用。

### 4. daily-discover / run_discovery 行为

要求：

- 不因单个 source 失败直接终止。
- 所有 source 都失败且无 fallback 时，返回明确错误 stats。
- dry-run 也应走 fallback 判断，但不写 `content_updates`。
- stats 中包含：
  - `source_status`
  - `source_effective_dates`
  - `source_fallback_used`

## 验收标准
1. ✅ TMDb 当日失败时，可使用上一可用 TMDb snapshot。
2. ✅ Trakt 当日失败时，可使用上一可用 Trakt snapshot。
3. ✅ FP 当日失败时，可使用上一可用 FP snapshot。
4. ✅ 一个 source 失败不会阻断其他 fresh source。
5. ✅ 超过 max staleness 不使用兜底。
6. ✅ 不复制历史 source rows 到当前日期。
7. ✅ stats 明确展示哪些 source 使用了 fallback。

## 测试要求
- `merge_three_sources()` 支持三个 source 不同日期。
- TMDb fallback + Trakt fresh + FP missing 仍能 merge。
- 超过 `max_staleness_days` 时 source 被标记 failed_no_fallback。
- dry-run 不写 `content_updates`，但返回 fallback stats。
- 全 source failed_no_fallback 时返回清晰错误。

## 风险点
- 使用旧 snapshot 会让“今日推荐”带有滞后性，所以必须由 P1.10-E 报告清楚。
- 默认允许最多 30 天内的 source snapshot fallback，可能引入较旧数据；必须依赖 P1.10-E 的显著标记降低误读风险。
- FP 当前存在订阅不可用阻塞，fallback 只能缓解短期失败，不能替代订阅恢复。
