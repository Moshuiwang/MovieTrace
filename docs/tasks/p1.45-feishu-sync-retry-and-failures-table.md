## 具体要求

### 1. Schema 变更（migration 018）

`feishu_sync_failures` 表：
```sql
create table if not exists feishu_sync_failures (
    id integer primary key autoincrement,
    synced_at text not null default current_timestamp,
    table_id text not null,
    record_id text,                   -- batch_update 场景有，batch_create 失败时为 null
    operation text not null,          -- 'create' | 'update'
    payload_json text not null,       -- 失败 record 的 fields，便于重试
    error_code text,
    error_message text,
    retry_count integer not null default 0,
    resolved_at text                  -- 重做成功后写入，便于历史审计
);
create index if not exists ix_feishu_sync_failures_unresolved
on feishu_sync_failures(table_id, resolved_at);
```

执行前 **必须** 备份：`cp data/movietrace.db data/movietrace_backup_20260522_<HHMM>_p1.45.db`

`schema.py` 的 `SCHEMA_SQL` 同步追加 CREATE 语句；`SCHEMA_VERSION` 改为 18。

### 2. 重试 helper

```python
def _batch_with_retry(fn, *args, max_retries=3, base_delay=1.0, **kwargs):
    """指数退避重试。fn 抛异常时按 base_delay * 2^attempt 等待。
    超过 max_retries 仍失败 → 重新抛出最后一次异常（由调用方持久化）。"""
```

- 重试条件：所有异常都重试（飞书 SDK 不区分细类型），但记录 `retry_count`
- 退避序列：1s, 2s, 4s（三次共等待 7s 上限）
- 不静默：每次重试 logger.warning 输出 attempt / 异常摘要

### 3. 失败持久化

`batch_create` 整批失败 → 拆分单条重试一次 → 仍失败的写入 `feishu_sync_failures(operation='create', record_id=null, payload_json=...)`
`batch_update` 同上，`record_id` 填实际 ID。

### 4. 次日重做入口

`sync_table` 开始时（在主同步逻辑之前）：
```python
_replay_unresolved_failures(conn, token, app_token, table_id)
```
- 查 `feishu_sync_failures` 中 `table_id == 当前表` 且 `resolved_at is null` 的记录
- 按 `operation` 分别调用 batch_create / batch_update（同样走 `_batch_with_retry`）
- 成功 → 更新 `resolved_at = current_timestamp`；失败 → `retry_count += 1`
- 若单条 `retry_count >= 10` → 走飞书 IM 告警（运营介入）

### 5. IM 告警

全部 batch 重试失败时调用 `notify.send_alert(title="飞书同步失败", body=f"table={table_id} created={N_failed} updated={N_failed}")`。复用现有 P1.30 接口。

## 风险点

- **指数退避在 cron 窗口内的累积等待**：3 次重试 × N 张表 × 多次 batch 可能挤压后续步骤时间预算；最坏情况估算 < 30s，可接受
- **`payload_json` 隐私**：飞书 record fields 不含 token / 密钥（只是业务字段），存库安全；不要把 access_token 写进 payload
- **migration 018 回滚**：DROP table 不可逆。如果上线后想撤销 → 走新 migration 019 DROP（在 ADR 记录）。**禁止**直接改 018 文件
- **重试 + replay 双重轨道**：当前批失败先就地重试 3 次 → 仍失败入 failures 表 → 次日 replay 再试。两个轨道不重叠，但需测试覆盖"replay 中再次失败"路径
- **`notify.send_alert` 是否会阻塞**：如果告警通道本身挂了，告警调用不能拖垮主同步——告警调用必须 try/except 包裹，失败仅 logger.error，不抛
