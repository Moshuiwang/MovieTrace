## 具体要求

### B1 — stat 命名一致化（finding #5）
1. `run_discovery` 当前把 `cd_stats["skipped_source_date_fallback"]` 透传成 `stats["observations_skipped_source_date_fallback"]`，该值在该调用路径下永远为 0。可选两种处理：
   - 删除该 stat（更彻底，调用方在 P1.57e 期间无依赖），或
   - 把它改名为 `_write_current_discovery_batch_internal_safety_skip` 并在 docstring 写明"仅当本函数被绕过 gate 直接调用时才非零"。
2. **必须二选一**，并写一行注释解释选择。
3. `_write_current_discovery_batch` 仍保留内部 `should_write_observation` 防御调用（不删），但内部 stat 命名要和外层 stat 区分。

### B2 — `_should_skip_enrichment` 从 stable_metadata_json 反查（finding #4）
1. 在 `_should_skip_enrichment` 命中 `current_discovery_items` 后，若 `candidate.tmdb_data` 缺 `original_language` 或缺日期字段，**继续读取**对应行的 `stable_metadata_json`，从中读 `original_language` / `last_air_date` / `last_episode_air_date` 作为后备。
2. 后备数据满足条件时也允许 skip；后备数据仍不满足则 return False。
3. `current_discovery.py` 可以新增一个只读 helper（如 `get_stable_metadata(conn, discovery_key)`）；helper 不允许做任何写操作；不引入新 schema 字段。
4. 测试覆盖："candidate tmdb_data 缺 last_air_date，但 stable_metadata_json 有 → True"、"两者都缺 → False"。
5. `stats["repeat_hit_enrichment_skipped"]` 在 TV repeat hit + stable_metadata 充分时必须 >= 1。

### C1 — row_factory finally（finding #6）
1. `_load_current_discovery_rows` 中的 `prev_factory` 还原必须放进 try/finally，保证 SELECT 异常也能复位。

### C2 — discovery_keys 去重（finding #9）
1. `src/movietrace/feishu/sync.py` 构造 `discovery_keys` 后传入 `_list_discovery_records_by_keys` 前必须 dedupe（用 `list(dict.fromkeys(...))` 保留顺序）。
2. dedupe 不允许丢失 records 中后续的 routing 信息，只去查询用的 list。

### C3 — `build_discovery_key` 一致使用（finding #10）
1. `src/movietrace/pipeline/discovery.py` 的 `_should_skip_enrichment` 当前 f-string 拼 key，改为调用 `build_discovery_key(content_type, tmdb_id)`，配合 try/except ValueError → return False。
2. grep `f"discovery:{...}:{...}"` 类构造，全部改用 helper（仅限本任务允许修改范围）。

### D1 — format_markdown 暴露新字段（finding #8）
1. `format_markdown` 的"其他更新"或新增的"📋 热点发现"分组里渲染 discovery 行时，必须展示 `首次发现` / `最近发现` / `发现次数` 三列。
2. new_season 行不展示这三列（旧版式不变）。
3. 不引入 emoji、不重画整段板式；只在表头和数据行加列。

### D2 — 测试使用 P1.57i 稳定键（finding #12）
1. `tests/test_feishu_sync.py` 的 `test_sync_table_update_excludes_ops_note` 当前 mock 的 key 是 `discovery:tv:1396:2026-05-17`（带日期后缀），改为 `discovery:tv:1396`（P1.57i 实际生产格式）。
2. 同步更新 record 的 `content_update_id` 到 stable key。

### E1 — 顺序假设文档化（finding #11）
1. `upsert_current_discovery_item` docstring 增加一段："本函数假设 observed_date 单调非降；如果 caller 以乱序调用，discovery_count 可能少计。生产 daily-discover 与 backfill 已保证按时间升序"。
2. 不修改逻辑（避免在 daily-discover 路径上多排序开销）。

## 风险点
- B2 stable_metadata_json 反查只对**已经被写过一次 detail 的 item** 有效；如果 P1.57e 之前的旧数据没有写 stable_metadata（NULL），skip 仍然不触发——这是预期降级，不视为 bug；需在 docstring 写明。
- C2 dedupe 不影响 routing；但实现要谨慎区分"查询用 list（去重）"和"写入用 records（保持原序）"。
- D1 新增列可能影响下游解析 markdown 的脚本；当前无此类脚本，但需 grep 一次 `latest.md`、`reports/`下消费者；如果有，先在 P1.57n 内补适配，不擅自扩范围。
- B1 删除 stat 是 breaking change：先 grep 所有读取该 stat key 的位置（CLI、notify-feishu、监控），无消费者再删；有消费者就走"改名 + docstring"路径。
