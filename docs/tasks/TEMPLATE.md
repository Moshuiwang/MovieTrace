## 创建 / 修改任务包后的同步要求

只要新增任务包、修改任务包边界、记录任务执行顺序，或改变任务之间的依赖关系，必须同步更新 [`STATE.md`](../../STATE.md)：

- 在 `STATE.md` 的当前阶段中列出新增或更新的任务包链接。
- 记录新的执行顺序、阻塞项、待用户决策或重大产品决策。
- 如果只是草拟任务包但尚未执行，标注当前状态（如”草案”或”待执行”），不要写成已完成。
- 完成后用文本检查确认任务包和 `STATE.md` 可互相追溯。

推荐验证命令：

```bash
rg -n "<task_id>|执行顺序|待用户" STATE.md docs/tasks/<task_id>.md
```

## 原子任务判断

一个任务可执行，必须**同时**满足：

1. 目标只有一个
2. 能追溯到设计文档或任务列表中的一个任务
3. 修改范围清楚
4. 验收标准可判断
5. 验证方式可执行
6. 完成后能独立提交

不满足时**先拆分**。拆分建议必须包含：子任务目标、范围、依赖、验收标准、验证方式。

## 写任务包前的失败信号

如出现下列情形，**先停下来澄清，再写任务包**：

- 需求还在猜测中
- 单次任务横跨多个目标
- 修改范围说不清楚
- 验证命令不存在或无法运行
- 代码改动无法解释为什么需要
- 架构、环境或验收用例还没有确认
- 测试当前在失败状态，但想继续开发新功能
