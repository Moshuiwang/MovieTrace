## 发现的问题

### 1. 日志 spam（已修复）

**现象：** 生产 DB 未迁移时，`log_api_call()` 每次写入失败都打印完整 traceback，导致数百行 stderr 输出。

**修复：** `logger.warning(..., exc_info=True)` → `logger.debug(..., exc_info=True)`。生产 DB 也已升级到 schema version 10。

### 2. FP 同时触发两次抓取

**现象：** FP 调用了 48 次而非 24 次。原因是 CLI `cmd_daily_discover` 和 `run_discovery` 各自独立调用了 `_ensure_fp_data`。

CLI Step 1 调用一次（独立 `connect_database`），Step 4 `run_discovery` 内部又调用一次。第一次写入 FP 数据后，第二次会检测到已有数据跳过。但由于第一次 FP 全部 402 返回 0 条，两次都尝试了 API 调用。

**建议：** 合并两次调用，或让 CLI 复用 `run_discovery` 的 FP 步骤。

### 3. OMDb 401 未被标记为 quota_error

**现象：** OMDb 的 HTTP 401 被 `http.py::_log_error` 记录为通用 `http_error`，`quota_error=0`。只有 OMDb 返回 200 + `{"Error": "Request limit reached!"}` 时才会被 `omdb.py::_log_omdb_quota_error` 标记为 `quota_error=1`。

**建议：** 在 `omdb.py::_log_ctx` 或 `http.py::_log_error` 中，对 OMDb 服务的 401 自动标记 `quota_error=1`。
