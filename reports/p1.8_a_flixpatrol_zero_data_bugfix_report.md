## 任务理解

修复 Phase 1.7 验收时 FlixPatrol 在 `2026-05-13` 为 0 的问题。修复范围限定在 FP 当日补数判断、FlixPatrol API 日期窗口和回归验证，不做 hot_score 权重调优，不改 TMDb / Trakt / OMDb 采集逻辑，不改数据库 schema。

---

## 根因

定位到两个直接问题：

1. `src/movietrace/pipeline/discovery.py::_ensure_fp_data()` 用 `snapshot_date >= ?` 判断是否已有 FP 数据。
   - 如果库里已有未来日期数据，会误判目标日期已补齐。
   - 目标日期实际可能仍为 0。

2. `src/movietrace/sources/flixpatrol_api.py::fetch_top10()` 只传 `date[from][gte]`。
   - FlixPatrol API 会返回目标日期之后的数据。
   - 这会污染指定日期补数语义，也会放大上面的问题。

---

## 完成内容

### 代码修复

- `src/movietrace/pipeline/discovery.py`
  - `_ensure_fp_data()` 改为按 `snapshot_date = ?` 判断目标日期是否已有数据。
  - 插入前跳过 `snapshot_date != date_from` 的 API 返回行。
  - 当 API 调用完成但目标日期仍插入 0 行时，记录 warning。

- `src/movietrace/sources/flixpatrol_api.py`
  - 当调用方只传 `date_from` 时，自动设置 `date_to = date_from`。
  - 请求参数变为同日闭区间：`date[from][gte] = date` + `date[from][lte] = date`。

### 回归测试

- `tests/test_discovery.py`
  - 新增测试：已有未来日期数据时，目标日期仍必须触发补数。

- `tests/test_flixpatrol_api.py`
  - 新增测试：单日查询必须传同日 `gte` / `lte` 参数，不能拉入后续日期。
