# 验证与完成判定

## 验证命令的硬要求

- 任务包提供验证命令 → **必须运行并读取输出**，不能只展示命令
- 任务包未提供 → 应要求补充
- 纯文档任务可用结构检查 + 链接检查 + 人工阅读替代（参考 STATE / SCOPE 优化的验证方式）
- 核心功能必须有测试**或**明确的人工验证步骤
- Bug 修复必须说明原因，并补充或更新回归测试

## 测试失败时的处置

1. **不掩盖** — 失败时禁止删测试或删逻辑"修复"
2. **不跳过** — 测试失败状态下，不开新功能（CLAUDE.md 规则 10）
3. **先解释** — 报告**现象 → 已排除原因 → 下一步定位计划**
4. **后修复** — 修复仅限当前任务范围内，不顺手扩大

## 声明完成前的 6 项检查

- [ ] 运行了任务包列出的所有验证命令吗？
- [ ] 读取了验证输出（不只是"命令退出 0"）吗？
- [ ] 验收标准每一条都可判断、都通过吗？
- [ ] 修改文件 100% 在允许范围内吗？
- [ ] 风险点段已诚实更新（含已知 TODO 与未覆盖边界）吗？
- [ ] 汇报按 [`docs/workflow/report-format.md`](../../docs/workflow/report-format.md) 格式吗？

任意一项答"否"或"不确定"——**不要声明完成**。

## 验证失败但原因不明

报告**现象 → 已排除原因 → 下一步定位计划**三段式，禁止靠猜测推进。

详细排障流程见 [`docs/workflow/troubleshooting.md`](../../docs/workflow/troubleshooting.md)。
