# API 探索铁律（防盲目猜测）

## 开工前必答（任一不清楚 → 停下澄清）

- 目标 API 具体是哪个端点？（不接受笼统描述如"wiki"）
- 官方文档 URL 已读确认？
- 认证方式？（Bearer / tenant_access_token / API Key）
- 必选参数清单？

## 流程：文档 → curl 手工测试 → 代码（缺一不可）

1. 读文档，记录 endpoint / HTTP 方法 / 必选参数 / 期望状态码
2. 用 curl 跑一次真实请求，拿到 200 再写代码
3. 代码写完立即跑验证命令，不积累再测

## 错误诊断

| 状态码 | 处置 |
|---|---|
| 401 | token 过期或 header 传错 |
| 403 | scope 权限不足，查飞书后台 |
| 400 / 422 | 必选参数缺失或类型错，查参数表 |
| 404 | **停止猜路径，查官方 endpoint 列表** |
| 5xx | 服务方问题，等待或查状态页 |

报错时必须记录：状态码 + 响应体（脱敏）+ 下一步定位方向。

## 5 个绝对禁止

1. **404 不猜第二个路径** — 查文档找唯一正确 endpoint
2. **不改 UA / Token 绕过 400** — 400 意味着参数错，查参数表
3. **不静默重试** — 重试必须有日志 + 上限（指数退避，3 次为限）
4. **不跳过文档直接写 SDK** — 手工测试成功后才抽象
5. **不一次改多个变量** — 改一个、测一个、定位原因后再改下一个
