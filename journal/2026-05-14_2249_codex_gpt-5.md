## 3. 关键决策记录

1. **飞书运营同步定位为旁路/运营同步层。**  
   MovieTrace 继续以 SQLite + MD/JSON 为事实源；飞书多维表格是运营消费层。

2. **第一阶段采用 `sync-feishu-table --notify-summary` 形态。**  
   表格同步和简要总结通知绑定，避免 daily_run.sh 漏掉成功通知。

3. **优先使用官方 `lark-cli`，不直接写 Python Feishu SDK。**  
   这样减少 API 客户端维护成本，但需要把 `lark-cli` 作为部署前置条件和 cron 环境风险管理。
