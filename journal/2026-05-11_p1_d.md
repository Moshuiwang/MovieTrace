## 匹配率低的原因分析

high+medium 仅 5/89 (5.6%)，原因：

1. **数据源不同** — FlixPatrol Top 10 反映的是 **今日美国流媒体实时热度**，baseline_items（855 条）是从飞书导入的 **精编内容库**（手工维护，非实时）
2. **时间窗口错位** — FlixPatrol 抓取的是 2026-05-11 当天数据，baseline 是历史积累
3. **baseline 标题格式差异** — 许多 baseline 标题含季号（S01）、年份（2021）、中文标注，影响 SequenceMatcher 得分

这是 V1 MVP 的预期行为 — 发现管道（P1-C）产出的是"今日热门"，基线匹配标记哪些是"我们库里有"的。

## 公共接口（给 P1-E 使用）

### `run_baseline_matching(db_path) -> dict`

```python
from movietrace.pipeline.baseline_matching import run_baseline_matching

result = run_baseline_matching("data/movietrace.db")
# result = {"total": 89, "high": 1, "medium": 4, "low": 22, "no_match": 62}
```

### candidate_matches 表

| 字段 | 说明 |
|------|------|
| `candidate_id` | FK → candidates.id |
| `is_in_baseline` | 1 if confidence ∈ {high, medium} |
| `baseline_item_id` | FK → baseline_items.id |
| `match_confidence` | high/medium/low/no_match |
| `match_method` | exact_title_year/exact_title/fuzzy_title/edit_distance/no_match |
| `match_score_detail` | 0.0-1.0 相似度 |
| `requires_human_review` | 1 if confidence = low |

### P1-E 使用方式

```sql
-- 按匹配状态 + priority 分组统计
select cm.match_confidence, c.priority, count(*)
from candidate_matches cm
join candidates c on c.id = cm.candidate_id
group by cm.match_confidence, c.priority;
```

## 剩余风险

1. **基线标题格式不一致** — baseline 标题含 `S01`/`S04` 等季号后缀、`2021` 等年份、中文翻译，导致 SequenceMatcher 得分偏低。建议后续标准化基线标题（去季号/年份/括号）。

2. **低置信度误匹配** — 22 条 low 中有 ~60% 是噪声。`requires_human_review` 标记已生效，P1-E 日报将分组展示供人工确认。

3. **baseline content_type 大量为 NULL** — 855 条基线中 content_type 多为空，类型过滤条件失效，可能导致 movie↔tv_show 间错误匹配。但不影响当前结果（low 中大多数本身就是标题不相似）。

4. **匹配未使用 tmdb_id** — 当前匹配仅用 title+year，未使用 candidates.tmdb_id 做精确链接。candidates 中 98% 有 tmdb_id，如果能与 baseline_items.canonical_item_id → external_ids 链路打通，可大幅提升匹配精度。建议 P1-I 改进。
